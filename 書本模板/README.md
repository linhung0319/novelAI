# 書本模板

這是「一本新書」的資料夾模板。**複製整個 `書本模板/` 到 repo 根目錄，改名為你的書名**（例如 `奇幻冒險/`），就能開始一本新書。

> 書資料夾與系統套件一起在本 repo 版控（刻意設計，見系統 README「開新書怎麼做」第 7 點）——開新書後記得 commit。

## 用法

1. 複製本資料夾 → 改成書名（放在 `novelAI/` 根目錄下）。
2. 把你的瑣碎筆記、靈感、半成品（任意格式）丟進 `raw/`。
3. **看 raw/ 收斂到哪，選一句說**（新書多半是前者）：
   - raw 還很亂、主線／題旨／結局未定 → **「幫我把 raw/ 想清楚」** → 觸發 `develop`，陪你把大方向談清楚，產出 `00-摘要.md` 與 `參照/世界觀.md`。**這是新書的正常起點**——`develop` 刻意不產大綱與幕綱。
   - 大方向已大致就緒 → **「幫我把 raw/ 整理成符合結構的大綱」** → 觸發 `organize`，產出 `01-大綱.md` 並回報缺口診斷。
   - 只有某一段想清楚、全書結局還沒定 → 「先幫我把『這一段』整理成大綱」→ `organize` 局部模式，產出 `大綱/arcNN.md`，不必替未定的結局搶先拍板。
4. 大綱／摘要大致定稿後可用 `outline-test` 審；再反覆用 `beat-sheet` 共創幕綱、用 `beat-test` 審稿，直到幕綱通過測試。
5. 幕綱就緒後，對 Claude 說「幫我把這幾幕寫成正文」→ 觸發 `write`（子專案 B）：AI 先提議分章方案供你拍板，再依幕綱展開成 `chapters/` 下的散文草稿。寫完用 `write-test` 審稿、`revise` 修訂。
6. 回頭改了上游之後，說「哪些下游過期了」→ 觸發 `sync` 列出待重驗清單。
7. 小說內容的版本歷史：本資料夾隨系統 repo 一起版控，開新書、寫到段落時記得 commit。

## 結構

```
你的書/
├── raw/        # 你丟進來的任意格式素材（靈感、筆記、半成品）
├── story/      # 整理後符合結構的產出
│   ├── 00-摘要.md
│   ├── 01-大綱.md
│   ├── 幕綱/
│   │   ├── _index.md
│   │   └── arc01.md
│   └── 參照/
│       ├── 角色需求.md
│       └── 結構.md
└── chapters/   # 依幕綱寫的章節草稿（子專案 B，由 write 產生）
    ├── ch01.md
    └── _index.md
```

`story/` 與 `chapters/` 下各檔的權威格式定義見系統套件的 `結構定義/*.schema.md`（含 `章節.schema.md`）。

> ⚠️ **上面這些檔在模板裡都是空骨架，不代表你已經有大綱或幕綱。** 它們只是先把位置佔好、讓你看得出這本書最後長什麼樣；每個檔的檔頭都標了「尚未產出」，由 `develop` / `organize` / `beat-sheet` / `write` 依 schema 逐步填寫。看到 `01-大綱.md` 存在就以為可以直接跳去拆幕，是這個模板最容易誤導人的地方。
