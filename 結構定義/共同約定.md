# 結構定義：共同約定（所有 skill 共用）

> **這是什麼**：所有寫作 skill（develop／organize／outline-test／beat-sheet／beat-test／write／write-test／revise／sync／worldbuild／character／expand）共用的基本約定。各 SKILL.md 以一行引用本檔，不再各自複述——避免十來份逐字重複的條文各自漂移。
> **修改本檔＝修改所有 skill 的行為**，變更前先確認影響面。

---

## 一、資源定位（不得寫死絕對路徑）

- **技巧知識庫／結構定義**：讀「本寫作系統 package 根目錄」下的 `技巧知識庫/` 與 `結構定義/`（與 `.claude/` 同層）。不確定根目錄時，從當前工作目錄向上尋找「同時含這兩個資料夾的目錄」。
- **書內容**：`raw/`、`story/`、`chapters/` 一律相對於「作者當前所在的那本書資料夾」。作者未指明是哪本書時，先確認。

## 二、四種所有權（本系統的骨架）

每個檔案都歸入一種所有權，決定「誰能寫它、有沒有 schema」。**動任何檔前先確認它的所有權**。

| 所有權 | 誰寫 | schema | 例 |
|--------|------|--------|-----|
| **人管·源** | 作者自由寫，**無格式要求**（重可讀可改） | 無 | `raw/`；設定層 `<實體>.md`（角色／世界觀／風格）；`00-摘要.md`；章節正文 `chapters/chNNNN.md` |
| **AI管·衍生** | AI 依「源」重生，**作者不手改** | 嚴格（`.ai.md`） | 設定層 `<實體>.ai.md`、`_index.ai.md`、`_總覽.ai.md`；`00-摘要.ai.md`；章節 `chNNNN.ai.md`；`chapters/_index.ai.md`；`就緒儀表`／`結構`／狀態事件流**投影**（綜合檔，宣告式·不帶 `.ai.md`·不走 hash） |
| **AI管·append log** | AI 章末追加，只增不重生 | 嚴格 | `狀態事件流.md`（狀態的唯一真實來源） |
| **作者利器·直改** | 作者直接改，含可機讀連結語法 | 中（連結語法） | `大綱`、`幕綱` |

### 源 → 衍生（本次重設計的核心）

- **源是唯一真實來源**；`.ai.md` 是「重生自源的建構產物」，如同 `.c`→`.o`。衍生檔**只會「新鮮／過期」，不會與源漂移**——因此拆兩檔不違反單一真實來源。
- **作者只碰源檔。** 衍生檔即使被誤改，下次重生即覆蓋；整個衍生層可隨時由源整批 rebuild。
- 衍生檔**照樣 commit 進 git**（本 repo 刻意版控書內容；衍生檔小、且 skill 讀它當 context）。

### 衍生檔 `.ai.md` 格式（混合：MD 殼＋YAML front-matter）

```markdown
---
generated-from: 3f9a1c…      # 對應源檔（正規化後）內容的 hash，見第三節
generated-at: 2026-07-19
角色: 凱                       # 以下為「機器事實」：對映／旗標／禁用詞表／維度狀態…
所屬arc: [arc01, arc03]        #   供 tools/ 快而準地讀（YAML，可程式解析）
伏筆: { 埋: [怕水], 收: [] }
影響力: 高
---
## 需求四象限                  # 以下為「prose 分析」：供 LLM skill 讀
- 期盼：…
- 想要（誤導）：…
## 限制與代價 / 升格哨兵
…
## 待裁決回饋                  # 下游 test 發現的可疑點 append 於此（見第四節）
```

- **機器事實放 front-matter**（YAML），**prose 分析放 MD 本體**。哪些欄位是機器事實由各產物 schema 規定。
- `_index.ai.md`／`_總覽.ai.md` 是 rollup 衍生檔：其「源」＝同層所有源 `*.md` 的集合（見第三節 digest 規則）。

## 三、變動偵測與同步

- **機械偵測（`tools/derived_sync`）**：對每個 `<實體>.ai.md`，重算其源檔正規化 hash，與 front-matter 的 `generated-from` 比對；不符即 **stale**。這給「像 git 那樣偵測人動了設定」但零外部 DB——hash 隨檔走。
  - 正規化＝統一換行、去每行尾空白、去檔尾多餘空行（見 `tools/derived_sync`），使無語意差異不誤判。
  - rollup（`_*.ai.md`）的源 digest＝同層各源 `<檔名>:<內容hash>` 排序後再 hash——**新增／刪除／改動任一同層源檔都會令 rollup stale**。
  - skill 重生某 `.ai.md` 後，呼叫 `derived-sync stamp` 把新 hash 蓋回 front-matter（別手算）。
- **stale 的處置**：偵測器只報「哪些衍生檔過期」；**重生內容是 skill（LLM）的工作**（worldbuild／character／write…），不是偵測器能做的。整批重生＝把偵測器列出的 stale 清單逐一交對應 skill。
- **深層語意過期**（結局→伏筆回收、題旨→立場鋪陳幕、世界觀規則→幕可行性）**仍走 `sync` 的語意比對**；機械偵測只管「人動了某源檔」這一類，兩者互補。

## 四、下游回饋（動尾改頭的落地）

- 下游發現問題根源在上游的「設定／摘要」時：**不改源、不改衍生的分析區**，而是在對應 `.ai.md` 的「## 待裁決回饋」區依 `結構定義/回饋記錄.schema.md` **追加一列**。
- 由 `worldbuild`／`character`／`develop` 回頭消化：提請作者修改**源檔**，再重生 `.ai.md`（回饋列隨重生清掉或轉「已採納」註記）。
- **幕綱沒有回饋區塊**——它是「作者可直接修改的利器」；發現幕綱問題只在對話中指出，建議作者直接改或重跑 `beat-test`，不代寫。
- 更新 `就緒儀表` 時，順手發現某 `.ai.md` 有「待裁決」列，於對應列備註欄加註「⚠️N 筆待裁決回饋」（被動偵測，不另開掃描流程）。

## 五、AI 角色 ＝ 共創 ＋ 審稿（非門檻）

- **所有創作決定權在作者。** AI 提草稿、抓重點、攤開選項與後果，但承重的創作抉擇（主角、結局、題旨、排幕、分章、任何一段文字）必須由作者拍板，絕不代決。
- **審稿員不是門檻。** 所有測試／診斷輸出「可疑點報告」交作者判斷，不做自動 pass/fail、不設數值門檻；嚴重度標示僅供排序參考。
- AI 產出的一切（骨架或散文）都是**可編輯、非受保護、非定案**的草稿。

## 六、揭示層級標記（🧊 水下）

有些設定與角色動機是**後段才向讀者揭示**的（母愛護盾的機制、反派的真動機）。這類條目寫進設定層 `.ai.md` 時，用下列標記註明它何時揭示。

### 語法

```markdown
（🧊 水下｜揭示於 收[[伏筆:母愛護盾]]）      ← 指向幕綱裡的回收點
（🧊 水下｜跨集留白，本書不揭）              ← 本書內不揭示
```

**每個 🧊 標記必須二選一**：指向一個伏筆名，或明標跨集留白。伏筆名須與幕綱裡的 `收[[伏筆:x]]` 完全一致（沿用 `幕綱.schema.md` 的連結規範）。

### 為什麼是指標，不是描述

**揭示時機的唯一真實來源是幕綱的 `收[[伏筆:x]]`。** 這裡只放一個指向它的連結，不用散文複述「此條於結局引爆」——那等於在幕綱之外另立第二本揭示帳，正是 `幕綱.schema.md`「伏筆不另立登記簿」刻意避開的壞味道。要知道某條 🧊 何時揭示，去幕綱查它的 `收` 在哪一幕。

### 解析規則

- **判先後不能比幕號大小。** 幕號是穩定 ID、不代表全書順序（見 `幕綱.schema.md`「幕號配號規則」）。要判斷某個揭示點排在某章之前或之後：先查該幕屬於哪個 arc，再對照 `story/幕綱/_index.md`「全書順序」比較兩個 arc 的先後；同一 arc 內才以幕號序為準。
- **揭示點還不存在是合法狀態。** 🧊 指向的伏筆名若在所有幕綱裡都還找不到 `收`（例如該 arc 尚未拆幕），標「揭示點待落幕」列為**資訊、非可疑點**——比照「本 arc 伏筆狀態」表對刻意留白的處理。

### 分工

- **寫標記**：`worldbuild`（世界觀）、`character`（角色）、`develop`／`organize`（冷啟動時發現該留水下的條目）——皆作者確認後，寫進衍生 `.ai.md`。
- **驗標記**：`beat-test` 機械核對每個 🧊 是否解析得到 `收` 點或明標跨集留白。
- 標記是**給下游看的資訊**，不是門檻——它讓下游知道「這條在某章是不是還在水下」，不阻擋任何撰寫決定。
