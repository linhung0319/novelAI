# 結構定義：大綱（`story/01-大綱.md`）

> **這是什麼**：定義「符合結構的大綱」長怎樣，作為 `organize` 產出與 `beat-sheet` 拆幕的輸入。
> **方法依據**：`技巧知識庫/創作流程.md`（大綱＝完整故事、兩次發想兩次收斂）、`技巧知識庫/結構公式.md`、`技巧知識庫/主題題旨.md`。

---

## 用途

大綱是**把摘要擴寫成結構完整、真正的故事**：連續敘述、不講究文筆修辭，直觀地把情節一一寫下。這是創作的最大難關（創造故事的「結構」）。**結局與主題在此階段必須已定。**

## 檔案格式

```markdown
# 大綱

## 選用結構公式
（開頭標註：三幕劇／起承轉合／七問題公式／編劇九階段之一，見 結構公式.md。）

## 故事全文（連續敘述）
（從頭到尾把故事講一遍。情節依因果排列，結局與主題明確。
 給自己看的草稿，不必雕文筆；建議別超過三千字。）

## 結局與題旨
- 結局：……
- 題旨：因為（X），導致（Y）。
（結局是回頭修正前文的地方——改結局就要回頭檢視前文邏輯。）
```

## 規則

- **結局與主題已定**：沒有結局，大綱不算定稿（`organize` 缺口診斷應標「還沒有結局 → 大綱無法定稿」）。
- **開頭標註選用的結構公式**，供後續 `beat-sheet` 對映幕、`beat-test` 做結構完整性測試。
- **連續敘述**：這是一個精簡的故事，不是幕的列表（拆幕是幕綱的事）。
- 兩種大綱版本（見 `創作流程.md`）：本檔是**創作用大綱**（無長度限制、可含草綱→細綱的演進）；對外宣傳大綱不放這裡。
- **結局是回頭修正前文的樞紐**：改結局 → 回頭檢視前文因果與伏筆（呼應「動尾就要改頭」）。

## 檢查點（供 outline-test 參照）

- 結局是否明確？題旨是否可寫成「因為 X 導致 Y」？
- 故事的立場是否自始至終一致（無中途詭變）？
- 選用的結構公式是否標明，且大綱大致依該骨架推進？
