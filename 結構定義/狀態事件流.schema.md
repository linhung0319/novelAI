# 結構定義：狀態事件流（`story/參照/狀態事件流.md`）

> **這是什麼**：定義「逐章實體狀態變化」怎麼記。狀態的**單一真實來源**——append-only、一行一事件；輕量「章末狀態快照」降級為它之上、可重算可丟棄的投影 cache（見 `章節.schema.md`）。這是子專案 C（逐章實體狀態定位）的核心 schema。
> **方法依據**：`結構定義/共同約定.md`（第六節「解析規則」定序——判先後走 `story/幕綱/_index.md`「全書順序」，非幕號大小）、`結構定義/章節.schema.md`（章末快照區塊，降級為本檔投影）、`docs/superpowers/specs/2026-07-17-subproject-c-state-layer-build-spec.md`（落地建置設計書）。
> **所有權**：**AI管·append log**（見 `共同約定.md` 二）——由 `write` 章末**只增不重生**地追加事件；作者可審可改，投影程式（`tools/state_projection`）唯讀。它是「狀態變化」這條軸的**單一真實來源**，非源→衍生兩檔、不走 hash 偵測（append log 天生只增不改，無「過期」問題）。

---

## 用途

當 AI 在第 XX 章協助寫作時，需要精準取得某角色（及其他設定）在該章**當下**的狀態（知識前沿／關係／持有…）。事件流把每個狀態變化只寫一次（在它發生的那一幕），查任何章的狀態＝從同一份權威事件流算出的 **as-of 投影**，不必回頭重讀前文、也不必在每章快照裡重抄同一事實。

## 檔案配置

- **每本書一檔**：`story/參照/狀態事件流.md`，append-only，一行一事件。
- 只被 `write` 追加（寫完某章順手產該章的狀態變化事件）、被投影程式（`tools/state_projection`，fold）讀取。
- 投影程式**唯讀**，永不寫此檔或任何書內檔；事件由 `write` 產、**作者可審可改**。

## 邊界（不另立第二本帳）

事件流是唯一新增的軸（逐章「狀態變化」delta）。它**不吸收也不重抄**其他軸——引用它們一律**只用名字**：

| 軸 | 歸屬檔 | 記什麼 |
|----|--------|--------|
| 逐章狀態變化（delta） | **本檔** `狀態事件流.md` | 某幕某實體某維度的變化 |
| 靜態角色弧線 | `story/設定/角色/<名>.ai.md` | 每角一份、終局弧線（非逐章時間線） |
| 伏筆埋／收機制 | `story/幕綱/`（本 arc 伏筆狀態表） | `埋/收[[伏筆:x]]` 在哪一幕 |
| 已承諾事實 | `sync`（反向盤點） | 下沉段已釘死、不可翻案的事實 |

引用伏筆／實體用名字（如「713 包裹」「隱形斗篷」），**不重抄幕綱的埋／收狀態**——呼應 `章節.schema.md`「章末狀態快照」節的「避免副本漂移」與「伏筆不另立登記簿」。

## 固定 4 欄信封格式

每個事件是一個 Markdown 清單項，固定四欄：

```markdown
- 幕NNN（arcAA）· <實體 或 實體A↔實體B> · <變化維度>：<自由內容>
```

四欄以 `·`（U+00B7 MIDDLE DOT）分隔前三欄，維度與內容以全形冒號 `：` 分隔。

### 逐欄定義

- **位置** ＝ `幕號＋所屬 arc`（如 `幕009（arcF）`）。**不用章號**——章號可亂序、非定序原語（見 `章節.schema.md`「章號不預配號段」）。判某事件在目標之前／之後：走 `story/幕綱/_index.md`「全書順序」spine，跨 arc 比 arc 先後、同一 arc 內才以幕號序為準（見 `共同約定.md` 第四節解析規則）。
- **實體** ＝ 角色／物件名（如 `哈利`、`隱形斗篷`）。**關係類**用 `實體A↔實體B`（如 `哈利↔榮恩`），視為一個雙向 slot。
- **變化維度** ＝ **封閉枚舉（固定 6 軸，跨小說通用）**：

  | 維度 | 記什麼 |
  |------|--------|
  | **知識前沿** | 此刻 POV／實體知道什麼、仍不知什麼（**最承重**，服務 Deep POV 資訊控制與伏筆回收） |
  | **關係** | 兩實體間關係的演變刻度（用 `A↔B`） |
  | **持有** | 得到／失去某物、道具易手 |
  | **位置** | 所在地的移動 |
  | **能力** | 學會／喪失某能力、力量狀態改變 |
  | **所屬** | 陣營／組織／隊伍歸屬的變動 |

  **維度不在此 6 枚舉內 → 投影程式報錯**（不靜默丟）。維度是 schema 裡的固定枚舉，**不是 runtime 登記簿**——需求驅動要新增軸，先過方法論閘門、改本 schema，不在執行期動態擴充。
- **內容** ＝ 自由文字，承載小說特有細節。可含 `·`（如轉譯名 `尼樂·勒梅`）——投影程式從左端定位、從右端取維度，名字內的 `·` 不影響解析。

## 固定 vs 通用如何同時成立

- **固定**：所有小說共用「4 欄信封 ＋ 6 維度枚舉」，無 per-novel schema。
- **通用**：靠「內容自由 ＋ 名字引用」——維度是普世的敘事狀態軸，內容是每本書自己的東西。

## 定序規則（as-of 投影）

1. 每個位置的 arc 對照 `story/幕綱/_index.md`「全書順序」解析成**全書序位** ＝ `(arc 在全書順序的排名, 幕號)`。
2. 跨 arc 比排名；同一 arc 內比幕號。
3. **as-of 第 N 幕**＝篩全書序位 ≤ 目標的事件，按 `(實體, 維度)` 分組，組內取序最新一筆當該 slot 的當下值。
4. 同一輸入永遠得同一輸出（純確定性、零 LLM、可覆算）。

> **as-of 語意**：投影含所有序位 ≤ 目標幕的事件。因 `write` 只在**寫完某章後**才 append 該章事件，事件流自然只含目標章之前的事件，故查「進場某章章首狀態」直接傳該章對應幕號即得。

## 投影程式契約與錯誤處理

- 投影程式：`tools/state_projection`（Python／uv，fold）。`uv run state-project --book <書資料夾> --as-of 幕NNN（arcAA） --entities <實體…>`。
- **lint 報錯、不靜默丟**：格式壞行（清單項以「幕」起頭卻缺 `·` 或 `：`）、未知維度、arc／幕無法在 spine 定位 → 報錯。
- **漏事件**（正文有狀態變化但 `write` 沒 append）＝ LLM 產出端的天生缺口；v1 靠**作者審 ＋ `write-test`** 兜底。程式端一致性校驗（比對事件流 vs 正文／幕綱的矛盾漏缺）**延到日後**，不進 v1。

## 範例事件（中性泛例，不引任何實書內容）

```markdown
- 幕012（arc01）· 少年 · 持有：獲得師傳的舊劍（此後隨身）
- 幕034（arc02）· 少年 · 持有：舊劍在渡口遺失（此後無劍，直到 幕051 尋回）
- 幕008（arc01）· 少年↔同伴 · 關係：萍水相逢 → 結為同盟
- 幕040（arc02）· 少年↔同伴 · 關係：結為同盟 → 因誤會決裂
- 幕019（arc01）· 少年 · 知識前沿：得知信物存在，尚不知其真正用途
```

（範例用中性泛例，不引某本書的實際內容——schema 範例一律不污染以該書做盲測的驗證。）
