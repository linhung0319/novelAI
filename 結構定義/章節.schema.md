# 結構定義：章節（正文源 `chapters/chNNNN.md` ＋衍生 `chapters/chNNNN.ai.md`）

> **這是什麼**：定義「幕綱寫成散文後」長怎樣。這是子專案 B（撰寫）的核心 schema——把幕綱的結構化欄位（角色／時空／行動／衝突／結果）展開成有場景、有對白、有感官細節的正文段落，同時保留可回溯到幕綱的錨點，讓因果與伏筆的機械可追溯性不因進入散文層而中斷。
> **本次重設計的核心變更**：章節改成**正文（源）／metadata（衍生）分離**——正文 `chNNNN.md` 是純散文人管源，機讀 metadata（對應幕／POV／基調・風格參照／狀態）移到衍生 `chNNNN.ai.md` 的 front-matter。**幕界錨點 `<!-- 幕NNN -->` 是唯一例外**：它內嵌回正文源檔（HTML 註解不可見、不礙人讀改），因為錨點必須貼著散文位置才有意義。所有權見 `共同約定.md` 二。
> **方法依據**：`技巧知識庫/場景與情節推進.md`（Scene/Sequel 展開）、`展示而非講述與五感描寫.md`（句子層級執行）、`視角與敘事人稱.md`（POV／知識邊界）、`敘事距離與DeepPOV.md`（Deep POV／心理距離調控）、`對白.md`（對白句型與壞習慣）、`動作場面寫作.md`（動作場面專項，選用）、`畫面感與電影化.md`（運鏡／示現／插敘／轉場）、`修辭與字句鍛鍊.md`（譬喻／擬詩／映襯／字句，依取向定位選用）、`心理描寫.md`（內在描寫五技法＋疼痛描寫）、`字詞句法與抗生硬.md`（抗生硬地板）、`網文與文學取捨.md`（取向定位決定節奏、爽感密度與文字功底門檻）。

## 用途

章節＝依作者拍板的分章方案，把連續幾幕的幕綱內容展開成正文。**章節是幕綱的散文呈現，不是取代品**——幕綱仍是唯一的結構真實來源（因果鏈、伏筆狀態）；章節正文只是它的「渲染結果」，可隨時重寫，不影響幕綱本身的正確性。

## 檔案配置（正文源／metadata 衍生 兩檔）

```
chapters/
  chNNNN.md        # 人管·源：純散文正文＋內嵌幕錨點 <!-- 幕NNN -->（無 front-matter）
  chNNNN.ai.md     # AI管·衍生：對應幕/所屬arc/POV/基調・風格參照/狀態（front-matter，重生自同名 .md）
  _index.ai.md     # AI管·rollup：權威章序表；源＝本資料夾所有 chNNNN.md 的集合
```

- **每章兩檔**：正文源 `chNNNN.md` 由作者／`write` 自由編輯（散文修改權在作者，見規則）；`chNNNN.ai.md` 由 `write` 生成、`derived-sync stamp` 封章，作者**不手改**機讀事實。
- **章號補零 4 位**：`ch0001.md`、`ch0002.md`……**全書連續編號，不因跨 arc 而重置**。補零使檔名字典序＝章序，避免 `ch10` 排在 `ch2` 前。
- **`_index.ai.md` 是權威章序**（見下），不是 `_index.md`——章序表由 `write` 依實際章檔重生。

### chNNNN.ai.md 的「源」是 chNNNN.md（重生語意）

`chNNNN.ai.md` 經 `tools/derived_sync` 以同 stem 對映到 `chNNNN.md`——**改動正文即令對應 `.ai.md` stale**，提示「正文變了，metadata 該重新校對」。重生時：

- **對應幕**＝直接掃描正文裡的 `<!-- 幕NNN -->` 錨點抽出（機械可導，見下「幕錨點規範」）。
- **POV／基調・風格參照／狀態**＝`write` 落筆時的決定，重生時**沿用並複核**（正文改動未動到涵蓋的幕時，這些欄不變；若正文改到跨進別的幕，`write` 據新錨點更新並提醒作者）。
- 校對後 `derived-sync stamp chNNNN.ai.md` 封回新 hash。

### 章號不預配號段（與幕號的不對稱是刻意的）

`幕綱.schema.md` 替每個 arc 預配 100 個幕號，**是因為幕號是連結標的**——`[[幕NNN]]` 前因、`埋/收[[伏筆:x]]` 所在的幕、正文錨點 `<!-- 幕NNN -->` 全都指著它，重編號得追著散落各處的引用改，代價極高、明令禁止。

**章號不是連結標的**：沒有任何產物用 `[[chNNNN]]` 指向章節，章號只活在檔名與 `chapters/_index.ai.md`。所以章號**不預配號段**，`write` 也**不替尚未寫成的上游 arc 猜測預留章號**——猜測本身才是風險（猜少了會與已寫的章相撞，猜多了留下一段永久空洞）。

亂序下沉、先寫後段的 arc 時：從當下最小的可用章號往後編，`_index.ai.md` 以「（未寫）」列標出前面 arc 的幕範圍。日後補寫排在前面的 arc，若作者要讓章號回到閱讀順序，改**檔名（正文源＋衍生對一起）、正文檔頭標題、`_index.ai.md`** 即可，沒有引用要追。挪不挪由作者決定——`_index.ai.md` 的「對應幕」欄才是章與幕的權威對映，章號只是檔案的名字。

## 正文源 `chNNNN.md` 的格式（純散文＋內嵌幕錨點）

```markdown
# ch0001 · 禁忌的叮嚀與焚夜

<!-- 幕001 -->
（幕001「禁忌的叮嚀」的 Scene 正文……）

<!-- 幕001→幕002 -->
（銜接幕001到幕002的 Sequel／過渡正文，若份量足以成獨立段落才另起錨點……）

<!-- 幕002 -->
（幕002「溜進畫廊」的 Scene 正文……）

……
```

- **正文源只含**：章名標題（`# chNNNN · 章名`，供人閱讀）＋純散文＋內嵌幕錨點。**無 front-matter、無 metadata bullet 列**——那些一律在 `chNNNN.ai.md`。
- **章名標題留在正文源**是刻意的：作者讀改的是這份檔，標題貼著內容才好認；它是人性化標籤、不是機讀事實（機讀章序在 `_index.ai.md`）。

### 幕錨點規範（唯一內嵌回正文源的機讀標記）

- **`<!-- 幕NNN -->`**：標記緊接其後段落主要在渲染該幕的 Scene（Goal/Conflict/Disaster）正文。同一幕若跨多段，只需在第一段前標一次，直到下一個錨點出現前都視為同一幕的延伸。
- **`<!-- 幕NNN→幕MMM -->`**：標記兩幕之間的 Sequel／過渡正文（銜接前一幕結果到下一幕動機的反應/兩難/決定）。若過渡文字很短（一兩句話），可省略獨立錨點、併入前一幕錨點段落尾端。
- **為何內嵌正文（唯一例外）**：錨點必須**貼著散文位置**才有意義（它標的是「這一段在渲染哪一幕」）；抽到 `.ai.md` 就失去定位。HTML 註解對讀者不可見、不礙閱讀與編輯，是承載它的自然容器。`chNNNN.ai.md` 的 `對應幕` 欄即由掃描這些錨點導出。
- **作者可用人性化標記**：作者手寫過渡時可用 `--- 幕002 ---` 這類好認的分隔，`write`／工具正規化回 `<!-- 幕002 -->`。
- **不新增第二套 ID 系統**：一律沿用幕綱既有的 `幕NNN` 三位數編號——正文層的追溯基準仍是幕綱編號，呼應幕綱「單一真實來源」的精神。
- **一幕不可跨章節切開**：章節邊界只能落在某個 `幕NNN` 錨點之前或某個過渡錨點之前，不可把同一幕的 Scene 正文切成兩半分屬兩章。

| 欄位 | 說明 |
|------|------|
| 對應幕 | `[起, 訖]` 連續幕區間，由掃描正文 `<!-- 幕NNN -->` 錨點導出；一章對應一段**連續**幕範圍，不交錯覆蓋其他章已涵蓋的幕 |

## 章密度規範（分章單位＝一個節奏單位）

- **章＝一個節奏單位**，有兩種、都合法：(i) 一個**閉合**的爽點/情緒循環（小起承轉結），或 (ii) **一段連續張力線中的一節**——本章只結一個小節拍，那條宏觀目標**跨章不閉合、持續悶燒**。分章單位是「節奏單位」，不是「篇幅感」。**不是每章都得閉合**——整段（數章）都各自閉合小勝利、章末各自打包，正文就會碎、無記憶點（見下「反模式②」與 `技巧知識庫/場景與情節推進.md` §六）。
- **依 `00-摘要.md`「取向定位」定基準**：**中文網文向預設一章一幕的粒度**（一幕＝一個 Scene）、**約 3000 字**（起點體例）＋章末鉤子；文學向才容許章節自然收束、跨幕邊界較鬆。**「一章一幕」只定粒度、不等於「每章都要閉合一個完整爽點循環」**——該幕的 Scene 可只推進一條跨章張力線而不了結它。
- **兩個反模式（皆避免）**：①把兩個各自完整的爽點循環壓進一章 → 蒙太奇→承重事件被塞進子句→下一章引用即「跳躍、要回頭查」（字數遠超 3000 是信號）；②每章各自閉合小勝利、**章末總結式收尾**、宏觀張力從不跨章 → 碎片化、無記憶點。
- **章末三種切法（`write` 撰寫章邊界時選一種；細則見 `技巧知識庫/戲劇張力與懸念.md` §二）**：**硬切斷章**（下一章緊接同一張力→停在未解畫面/動作、不回顧不預告）、**前開式軟鉤**（用具體新擾動向前拋）、**收束斷章**（段/arc 邊界才閉合＋喘息、鉤子仍是新擾動非回顧）。**反模式「總結式收尾」明令避免**：章末用「回顧本章處境＋預告未來」的抽象總結段收尾（例：「他站在一個死人的屋子裡…欠著死人的債」／「三日之後，真正的那個人要親自來了」）＝講述，是「硬要總結感」的來源；續接章根本不該收攏、硬切即可。
- **字數依張力起伏重分配、非等長勻平**（3000 字為基準非死線，依段落自然邊界浮動）：一段的**高潮幕鋪成 set-piece**（厚）、**過場幕壓薄**（快），並安排喘息/日常節拍與峰後餘韻讓記憶點長出來（見 `場景與情節推進.md` §六）。某幕遠不足基準且與鄰幕合成一節奏單位→可併章（作者拍板）；某幕遠超基準含多循環→訊號是幕綱該拆幕（回 `beat-sheet`），受「一幕不可跨章切開」限制、`write` 不硬拆。
- 方法依據見 `技巧知識庫/網文與文學取捨.md`（節奏速度·連載鉤子強度）、`場景與情節推進.md` §五（單章快慢）§六（跨章張力線·快慢交替·餘韻）、`戲劇張力與懸念.md` §二（章末三種切法）、`Data/跨語言技巧/結構與節奏.md`（連載單位）。

## 章末狀態快照（`chapters/_index.ai.md` 選用區塊）

> **⚠️ 定位（子專案 C v1）**：本區塊是**衍生產物**——狀態的單一真實來源是 `story/參照/狀態事件流.md`（見 `結構定義/狀態事件流.schema.md`），章末快照是「fold 投影程式（`tools/state_projection`）從事件流算出的 as-of 投影」，**非權威、非手抄、可重算可丟棄**。保留本區塊格式定義，供投影輸出與人眼閱讀對照。

逐章記錄 POV／關鍵角色在「**該章結束時**」的狀態。直接服務三件事：**Deep POV 資訊控制**（此刻 POV/讀者知道什麼、仍不知什麼）、**伏筆連貫**（哪章誰碰過/知道某物）、**關係演變刻度**。

- **這是一條「別處沒有」的獨立軸，不是第二本帳**：`story/設定/角色/<名>.ai.md` 記的是**靜態弧線**（每角一份、非逐章時間線）、幕綱「本 arc 伏筆狀態」表記的是**埋/收機制**、`sync` 反向記的是**已承諾事實**——都不追蹤「章末此刻誰知道什麼/持有什麼」。本區塊只補這個角色視角切片；**引用伏筆用名字即可（如「713 包裹」），不重抄幕綱伏筆表的埋/收狀態**（避免副本漂移，呼應「單一真實來源」）。
- **只記已寫章**；未寫章不預填。最承重的是「知識前沿」一格，其餘按需填。
- 格式（每個已寫章一則，接在索引表後）：
  ```markdown
  ## 章末狀態快照
  ### ch0002（章末）
  - 知識前沿：小女巫已知自己是被老婦人收養、村裡大人刻意迴避「收養」這件事；**尚不知**自己的身世真相、也不知老婦人隱瞞的原因
  - 關係刻度：與老婦人（表面順從、暗生疏離）；同齡玩伴尚未登場
  - 持有／狀態：帶著養母給的舊布包；**尚未**察覺自己的能力
  ```
  （範例用中性泛例，非任何書內容——schema 範例一律不引某本書的實際章節，以免污染以該書做盲測的驗證。）

## 規則

- **正文修改權在作者**：`write` 產出的 `chNNNN.md` 是可直接編輯的草稿，不是受保護的定稿。正文源改動後，對應 `chNNNN.ai.md` 由 `derived-sync check` 報 stale，`write`／`revise` 據新正文重新校對 metadata 並封章。
- **機讀事實不手改**：`chNNNN.ai.md` front-matter 由 `write` 生成、工具封章；作者要改 POV／風格指定，改的是「請 write 重寫這章」的決定，不手動改衍生檔（改了下次重生即覆蓋）。
- **待裁決回饋走衍生檔**：撰寫時若發現問題根源在**幕綱**，走「建議作者直接編輯幕綱或重跑 beat-test」（幕綱是作者利器·直改，無回饋區）；若溯源到**世界觀／00-摘要／角色／風格**，在對應那份 `.ai.md` 的「## 待裁決回饋」區追加列（格式見 `回饋記錄.schema.md`），交 `worldbuild`／`character`／`develop` 提請作者改源後重生。write-test 若要對「章節本身」記回饋（罕見），append 於 `chNNNN.ai.md` 的「## 待裁決回饋」區。
- **不做自動一致性追蹤**：幕綱若在章節寫成後又被修改，本 schema 不要求自動比對／自動標記過時（`derived-sync` 只偵測「正文源動了」這一類；幕綱→章節的語意過期走 `sync` 語意比對）。

## 檢查點（供 write-test 參照）

- **幕涵蓋完整性**：本章「對應幕」範圍內每一幕，是否都能在正文中找到至少一個 `<!-- 幕NNN -->` 錨點？`chNNNN.ai.md` 的 `對應幕` 是否與正文實際錨點一致（不一致＝該重生封章）？

- **動作場面專項**：若本章有幕屬於戰鬥/動作類型，是否套用動作場面寫作的專項準則；是否誤將「詞彙鋪張」「鋪墊框架」套用到非玄幻/仙俠類型的一般動作幕？
- **基調・風格一致性**：本章筆調是否與 `00-摘要.md` 基調一致，無不自覺轉調？用詞是否命中對應 `風格.ai.md` 的 `禁用詞表`、人物互稱是否越出 `稱謂系統`、段落性質是否套了對應 `可用句式` 檔位（見 `風格.schema.md` 檢查點）？
- **節奏與定位一致**：本章實際節奏／揭露速度／資訊密度／鉤子強度，是否與 `00-摘要.md`「取向定位」欄宣告的網文↔文學傾向相符（見 `網文與文學取捨.md`）；文字是否達「流暢四層次」地板（無錯別字、直白、順口、無時代錯亂詞彙破壞氛圍——第②③層的**操作化句法細則**見 `技巧知識庫/字詞句法與抗生硬.md`，時代錯亂詞彙的機讀清單見 `風格.ai.md` 禁用詞表，並由下一條「句法生硬／翻譯腔」把關）？
