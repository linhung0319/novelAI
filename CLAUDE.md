# novelAI — AI 輔助小說寫作系統

這個 repo 是「寫作系統套件」本身，不是某一本書；書資料夾（如 `gothic_witch/`）放在根目錄下，與系統套件一起版控。完整說明見 `README.md`。

## 三個組成部分

- `結構定義/` — 各產物（摘要／世界觀／角色／風格／大綱／幕綱／章節／參照檔）的權威格式（schema），含所有 skill 共用的 `共同約定.md`。設定層（世界觀／角色／風格）＋摘要＋章節採**源→衍生兩檔**（人寫 `<實體>.md`＝唯一真實來源，AI 重生 `<實體>.ai.md`；變動偵測 `tools/derived_sync`）。
- `技巧知識庫/` — 整合後的寫作方法（創作流程／因果法則／結構公式／需求四象限／角色塑造／伏筆與意外／戲劇張力與懸念／主題題旨／結局類型／世界觀與背景／氛圍與筆調／選題與收斂／視角與敘事人稱／場景與情節推進／展示而非講述與五感描寫／對白／動作場面寫作／網文與文學取捨）；扁平並列，檢索入口見 `技巧知識庫/_index.md`。網文↔文學的技巧取捨（何時用哪套）見 `技巧知識庫/網文與文學取捨.md`；「看情況二選一」的 OR 抉擇對照表見 `技巧知識庫/_index.md`。
- `.claude/skills/` — 操作：skill 承載「操作」、知識庫承載「方法」。

## Skill 地圖（作者在哪個階段，用哪個）

| 情境 | skill |
|------|-------|
| raw 很亂、主線/題旨/結局未定（粗階段） | `develop` |
| 大方向就緒，整理成大綱（全書或局部 arc） | `organize` → 審稿 `outline-test` |
| 某個 arc/段落只有一個橋段、劇情單薄，要橫向長厚（偵測缺口→提案衝突/子情節/新角色） | `expand`（作者拍板後才寫入）|
| 粗階段後任何時候，世界觀累積待確認矛盾／某維度要深做補厚／力量體系要跑限制代價檢查／要折入外部 research | `worldbuild`（作者拍板後才修訂正文）|
| 粗階段後任何時候，要獨立設計單一角色（新反派／配角，或替既有角色補需求弧線）——非因劇情缺口、非冷啟動打包 | `character`（作者拍板後才寫源 `設定/角色/<名>.md` 並重生 `.ai.md`）|
| 大綱拆幕綱 | `beat-sheet` → 審稿 `beat-test` |
| 依幕綱寫正文 | `write` → 審稿 `write-test` → 修改 `revise` |
| 上游改動後找過期下游／回頭收斂前盤點已承諾事實 | `sync` |

大多數 skill 是**垂直**推進（摘要→大綱→幕綱→正文）；目前有兩個對稱的**橫向**能力——不往下一層走，而是把同一層某個產物就地補厚＋自洽：`expand` 做**劇情層**（把單薄的 arc 就地長厚）、`worldbuild` 做**設定層**（把 `設定/世界觀/<主題>.md` 就地補厚並消矛盾，並回頭消化下游追加到 `<主題>.ai.md`「待裁決回饋」的回饋列）。設定愈厚，`expand` 的世界觀自洽檢查愈準——兩者互相餵養。

## 沒有 skill 觸發、或需求太小不值得跑整套 skill 時

系統有三個「檔位」：**答問**（無產物）、**輕量輔助模式**（小型生成／定點編輯）、**完整 skill 流程**（大動作）。skill 是「大動作」的策展式操作，不是每個小需求都要動用它——小需求走輕量模式，避免大砲打小鳥。

- **答問**：回答任何寫作技巧問題（對白、角色動機、結構、伏筆、POV……）之前，先查 `技巧知識庫/`（入口 `_index.md`「何時查它」欄）有無對應方法，依方法作答並註明出處檔名；知識庫沒有的，明說是 LLM 自身判斷。
- **輕量輔助模式（小型生成／定點編輯）**：當作者要的是「只動單一產物的局部、不產生新結構承諾、不需拍承重抉擇」的小需求（想一則幕尾懸念／改好這一句對白／補一個感官細節／擬一小段文字）時，走輕量模式，**不跑整套 skill**：
  1. 用 `技巧知識庫/_index.md`「何時查它」欄定位對應技法檔，讀其檔頭標**核心**的小節；
  2. 依 `結構定義/共同約定.md` 資源定位，拉相關權威資料當 context（`00-摘要.ai.md` 取向定位／源 `00-摘要.md` 基調、`設定/世界觀/*.ai.md` 核心規則、`設定/角色/<名>.ai.md` 對應角色、`設定/風格/風格.ai.md` 腔調禁用詞——只讀需要的）；
  3. 產出**單點提案**，並註明取自哪個技法檔；
  4. **不寫入書內檔，或只提議一處編輯待作者拍板**，不套用整個 skill 的流程與儀式。
- **降檔閥（反僵化）**：即使某重 skill 的 description 命中，但作者實際要的只是上述單點需求，可用一行說明改在輕量模式處理，並主動問「要不要跑完整 `<skill>` 做系統版」——升不升級由作者決定。
- **升級閥**：小需求若牽出結構問題（要新增／移動幕、動到既定因果或伏筆、改承重抉擇），停在輕量模式、指回對應 skill（`beat-sheet`／`write`／`organize`…），不硬在小檔位裡解。
- **修改書內檔前**：先讀 `結構定義/` 對應 schema 與 `共同約定.md`，遵守格式與「AI＝共創＋審稿、非門檻」原則（輕量模式若只提議不寫檔，此條在作者採納、要落檔時才適用）。
