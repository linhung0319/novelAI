## 一、Context / 背景與定位

子專案 A（`docs/superpowers/specs/2026-07-10-novel-structure-pipeline-design.md`）終點＝幕綱通過測試；子專案 B（`docs/superpowers/specs/2026-07-11-subproject-b-write-skill-design.md`）＝依幕綱寫正文。兩者都把「跨 arc／跨集的規模化連貫性基建」明確**釘死歸給尚未實作的子專案 C**（見 `結構定義/章節.schema.md:88,104`、`.claude/skills/write/SKILL.md:84`、`.claude/skills/write-test/SKILL.md`、`.claude/skills/sync/SKILL.md`）。本文件就是那個 spec。

觸發問題（作者原話）：**「AI 在第 XX 章協助寫作時，如何精準找到該角色（及其他設定）在第 XX 章當下的狀態？」**

### 現況診斷（三點，附行號）

1. **機制已在、但只寫不讀。** 系統已有逐章狀態機制——`chapters/_index.md` 的「章末狀態快照」，由 `write` 寫完每章的收尾步驟維護（`.claude/skills/write/SKILL.md:65`；schema 定義 `結構定義/章節.schema.md:83-98`）。但它是 **write-only**：`write` step 1 讀 `chapters/_index.md`（`.claude/skills/write/SKILL.md:47`）**只為「避免重複撰寫已完成的章節、避免重新編號」**，沒有任何 skill 把某章快照當「進場狀態」讀回去。今天「第 XX 章當下狀態」的真實答案是——**AI 靠讀幕綱該幕欄位＋POV 知識邊界規則＋前文即時重建**，不是查一個現成的狀態索引。

2. **這不是現在的 bug，而是 C 的設計核心。** 在 ch05（只寫成一章）的規模下，即時重建完全夠用、未複現任何失敗（`情境測試/哈利波特實測/G7-write.md` 的 713 Deep POV 知識邊界守住即為證）。read-loop 只在**規模化**時才咬人：第 30 章要用第 8 章確立的狀態，靠重讀 30 章正文既貴又不可靠。這正是 GP3 被判為「三探針中最痛的真缺口」、且「子專案 C 去向正確」的原因（`情境測試/哈利波特實測/GP1-3-缺口探針.md`、`情境測試/哈利波特實測/_index.md` 缺口 #45）。

3. **現有輕量快照有一個規模化隱患。** 它是「每章存一份完整狀態」（full-snapshot-per-chapter）。同一事實（例：「哈利有嘿美」）會在其後每一章的快照裡重複出現——**把作者已在對抗的「副本漂移」放大成 C 級災難**。schema 自己已在打補丁（`結構定義/章節.schema.md:87`：「引用伏筆用名字即可，不重抄幕綱伏筆表的埋／收狀態——避免副本漂移，呼應單一真實來源」）。C 若沿用「完整快照」而只是加密度，等於把這個補丁反過來作廢。

### 外部研究對照（2026-07）

一次外部研究（AI 寫作工具、長上下文 LLM 記憶、軟工時間點查詢、定序）高度一致地指向同一個設計，而它**恰好就是本系統既有哲學換到狀態層**：

- **軟體工程「時間點狀態查詢」**：Event Sourcing 把真實來源定為「不可變、只追加的事件序列」，任一時刻狀態＝回放事件；**snapshot 被明確定位為純效能 cache、非真實來源、可過時可丟棄可重算**（AWS/Azure Event Sourcing & Materialized View patterns）。Bitemporal（valid time／transaction time）專門處理「系統會往過去補寫、寫入順序凌亂」時如何維持「演進但一致」的視圖（XTDB／JUXT）。
- **敘事一致性研究**：SCORE（arXiv 2503.23512, 2025）為 AI 敘事維護角色屬性／物件狀態／關係的內部表徵，用檢索＋時間追蹤強制 point-in-time consistency，正是防「既定事實無故改變／物品無故得失」；Narrative Continuity Test（NCT）形式化量測 entity tracking 與歷時連續性。
- **記憶系統**：Mem0 對記憶做 add/update 的 **delta 操作而非堆疊**；Zep 用**時間知識圖**（邊帶時間、可查某時刻實體狀態）。
- **AI 寫作工具的反面教材**：NovelAI Lorebook／AI Dungeon Story Cards 是「一實體＝一份可變快照」，改一次就毀掉早期狀態、**無 as-of 概念**；Novelcrafter Codex 較進步（自動回鏈偵測、宣稱可追跨 timeline 演變）但仍是商業黑箱。共通教訓：**單快照可變 = 副本漂移**。
- **定序**：業界用「與檔名 ID 解耦的顯式順序 spine」（topological／stable-topological order；git 把穩定身份 commit-hash 與歷史順序 parent-DAG 分離）。

> 術語（供本文件其餘章節引用）：*event sourcing、append-only event log、materialized view/projection、snapshot-as-cache、bitemporal（valid time / transaction time）、as-of / point-in-time query、entity state tracking、temporal knowledge graph、narrative continuity、topological / stable-topological order、narrative spine*。

## 二、範圍與非目標

**範圍**：規模化的實體狀態定位——「給定第 XX 章（或某幕），投影出某實體此刻的狀態」，服務三命脈：Deep POV 資訊控制、伏筆連貫、關係演變刻度（GP3 三命脈）。

**非目標（守邊界，勿僭越）**：

- **不取代輕量快照的近端用途**：單集／連續章內的人眼可讀切片仍有價值；C 是把它的「真實來源」角色抽走，改由事件流承擔（見六）。
- **不做創作判斷**：C 是狀態的記錄與投影機制，不替作者決定劇情、不自動改寫正文。
- **不改上游 schema 語意**：`角色需求.md`＝靜態弧線、`世界觀.md`＝靜態設定＋🧊 揭示指標、幕綱＝伏筆埋／收＋因果——這些分軸不變，C 不吸收也不重抄它們（`結構定義/章節.schema.md:87` 的四軸分界維持）。
- **不自動鏈式觸發 skill**：skill 間不自動觸發是刻意的檔位設計（`CLAUDE.md`「避免大砲打小鳥」；`情境測試/哈利波特實測/_index.md` #35 已判自動觸發為反模式）。C 的維護與查詢由既有 skill 在既有時機順手做，不新增自動掃描流程。

## 三、核心資料模型抉擇（本 spec 的重心）

**抉擇**：C 的真實來源應是**「帶章序的狀態變化事件流」（append-only），而非「每章一份完整快照」**。輕量快照降級為事件流之上、可重算的投影（cache）。

| 面向 | 現況（多份完整快照） | C 建議（狀態變化事件流 ＋ 快照當 cache） |
|------|----------------------|------------------------------------------|
| 真實來源 | 每章一份完整狀態 | append-only「狀態變化事件」流，每條綁**章序位置** |
| 同一事實 | 在其後 N 章快照各存一份 | **只寫一次**（在它發生的那一章） |
| 漂移風險 | 高（改一處要同步 N 處，漏一處即漂移） | 近零（單一真實來源；快照可重算故永不與真相衝突） |
| 查第 XX 章 | 直讀該章快照（O(1)） | 沿 spine 回放至該章 → 投影（as-of query） |
| 往前補寫 | 需回頭改多章快照 | 只新增一條事件、標其 valid-time 位置 |

**為什麼事件流能消除「多快照重複而漂移」（作者的關鍵問句）**：事件流裡每個事實只被寫一次（single point of truth）。「哈利得到嘿美」只存在於發生的那一章那條事件，不會在其後每章快照裡各存一份副本。要問任何章的狀態，是**從同一份權威事件流計算出的投影**，不是另建一本帳。這與本系統「伏筆不另立登記簿」（`結構定義/幕綱.schema.md`）、「🧊 是指標不是描述、揭示時機唯一真實來源是幕綱的收」（`結構定義/共同約定.md:38-40`）、「不另立登記簿」（`結構定義/回饋記錄.schema.md`，見 `情境測試/哈利波特實測/_index.md` #42/#16 兩度援引）是**同一條原則**，只是套到狀態層。快照被定位為 event log 之上、可丟棄可重算的 cache——這正是抵抗「第二本帳」的理論護城河。

**事件條目的最小 Markdown 形狀（純文字、無 runtime、待落地時定稿於 schema）**：

```markdown
- 幕NNN（arcAA）· 哈利 · 持有：獲得貓頭鷹「嘿美」（海格生日禮）
- 幕NNN（arcAA）· 哈利↔榮恩 · 關係：初識 → 同行結伴
- 幕NNN（arcAA）· 哈利 · 知識前沿：得知 713 包裹存在，尚不知其為何物
```

- 每條綁**發生位置**（幕號＋所屬 arc，非章號——章號非連結標的、可亂序，見四）。
- 引用伏筆／實體**用名字**（「713 包裹」），不重抄幕綱埋／收狀態（沿用 `結構定義/章節.schema.md:87` 精神）。
- **變化類型**（知識前沿／關係／持有／能力／位置／所屬…）需求驅動擴充（見五），不預先窮舉。
- 「知識前沿」仍是最承重的一類（服務 Deep POV 資訊控制與伏筆回收），與現況一致。

## 四、定序：as-of query 沿既有 spine

**好消息：定序原語已經存在，C 不必新建。**

- 章號／幕號是**穩定 ID、不代表全書順序**（`結構定義/章節.schema.md:18-24`、`結構定義/幕綱.schema.md` 幕號配號規則）。
- 「全書順序」spine 活在 `story/幕綱/_index.md`；`結構定義/共同約定.md:44` 已明訂：「**判先後不能比幕號大小**……先查該幕屬於哪個 arc，再對照 `story/幕綱/_index.md`「全書順序」比較兩個 arc 的先後；同一 arc 內才以幕號序為準。」

C 的狀態查詢＝**沿這條既有 spine 做 as-of 回放**：把所有 valid-time 位置 ≤ 目標章（對映到幕、再對映到 spine 位置）的相關事件折疊（fold）成投影。這對映到 bitemporal 的 valid-time 查詢；「往前補寫」（先寫 arc05、後補 arc03）正是 valid-time ≠ 寫作順序、需要顯式 spine 的正當理由（stable-topological 精神：能顯式排就顯式排，未定序處用編號兜底）。

**含義**：C 不引入第二套定序，只是把「查揭示點在某章之前或之後」（`結構定義/共同約定.md:44` 已在做的事）通用化為「查任何實體狀態在某章的投影」。

## 五、通用化到非角色實體（需求驅動）

事件流模型天然通用：**「實體有一條 delta 時間線」** 一體涵蓋角色之外的組織關係、地理、力量體系狀態、道具。今日這些在 `世界觀.md` 都是**靜態**條目、無逐章演變機制（刻意留白）；C 的事件流可承接它們的「隨劇情變化的版本」。

**但嚴格需求驅動、不預先鋪。** 依本系統累積的鐵律（`情境測試/哈利波特實測/_index.md`「加欄前先 grep 有沒有機械消費者」「觀察到摩擦 ≠ 存在失敗」），每一種實體型別要進 C，須各自先過三關（見七）。設計上留出「變化類型」欄的擴充位即可，不在 schema 預先窮舉組織／地理／道具等型別——那會重蹈「未經失敗支撐的契約擴張」（`情境測試/哈利波特實測/_index.md` 第 4 群結論）。

## 六、從輕量快照平滑併入 C 的遷移路徑

呼應 `結構定義/章節.schema.md:88`「本區塊是它近乎零成本的前身，未來可平滑併入 C」：

1. **現況**：`chapters/_index.md` 章末狀態快照＝人手寫的「投影」，同時（不當地）兼任真實來源。
2. **C 落地**：新增事件流檔（暫名 `story/參照/狀態事件.md` 或每 arc 一檔，落地時定），成為真實來源；把章末狀態快照重新定義為「事件流的投影（cache）」，標明可重算、非權威。
3. **相容**：`write` 收尾時，改為「新增本章發生的狀態變化事件」（delta），投影仍可如今日般人手或（未來）機械產生；`write`／`revise`／`write-test` 在協助第 XX 章時，改為**讀該章的投影**當進場狀態（關閉 read-loop）。
4. **可逆**：因投影可重算，遷移不破壞既有 ch05 快照——它成為第一筆可被事件流回算驗證的樣本。

## 七、方法論閘門（落地前提，非本 spec 執行）

本系統五輪實測累積出一條鐵律：**真缺口＝契約缺一條、有機械消費者、且剝乾淨冷跑就複現——不是靠想像**（`情境測試/哈利波特實測/_index.md` 五條方法論教訓）。動系統前的固定三步：

1. **誰在承載、寫成文了嗎？**（翻 schema／SKILL）
2. **有機械消費者嗎、它會誤讀嗎？**（grep）
3. **剝乾淨 fixture 冷跑，失敗複現嗎？**（RED）

**本 spec 只做設計，不主張缺口已成立。** read-loop 只寫不讀在現規模（ch05）**未複現失敗**；C 的落地時機，應由作者在資料量成長到即時重建不可靠時，跑一次「剝乾淨冷跑」RED——例如：構造一本已寫 N 章的 fixture，剝掉 AI 直接重讀全文的路徑，觀察它在缺事件流時是否對「第 XX 章某角色知道什麼／持有什麼」判錯——複現失敗，才落地；且只補最小契約，不建帳本／自動觸發／登記簿。

## 八、Q2：引進程式（template／script）的判斷與延後觸發條件

現況：全系統 100% Markdown、無 runtime；所有「機械可驗證」靠 `[[幕NNN]]`、`埋／收[[伏筆:x]]`、`<!-- 幕NNN -->` 等文字慣例，由 LLM 執行。

- **Template（不做）**：`書本模板/` 已是複製式模板（開新書複製改名）。在 skill 內再綁 template 會與之重複，並違反「不寫死絕對路徑、可攜、零基建」（`結構定義/共同約定.md:8-12`、`README.md` 設計原則）。無收益。

- **Script（同意延後，但界線要畫清）**：唯一真正值得程式之處＝**跨檔的確定性 link／token／定序完整性檢查與事件回放 fold**。這裡有一條來自本系統自身的鐵證——第 2 群「假閉合」事件（`情境測試/哈利波特實測/_index.md:202`）：備註裡字面的 `埋[[伏筆:史奈普誤導線]]` 被機械 set-diff 誤配成「已閉合」，誤導線其實一次都沒埋過；根因是「**所謂機械抽取、確定性檢查，其實是 LLM 在讀上下文——跨 arc 情境下那個宣稱是空的**」。這正是 C 級規模「LLM 假扮 parser」會撞的可靠性牆：長列表 set-diff、token 是視圖還是埋設點的區分、spine 定序、事件 fold——都是 LLM 不可靠而真 parser 完美的確定性操作。

  **建議**：延後至進入 C 門檻、且冷跑證實 LLM-grep 在該資料規模不可靠時再引入；範圍**嚴格限縮**在確定性的 link／token／spine 檢查與事件回放 fold（即 `beat-test`／`write-test`／`sync` 的機械子檢查、以及 C 的投影計算），**絕不碰創作判斷或審稿的語意判斷**（那永遠是 LLM／作者的事）。在此門檻之前，LLM-grep 是正確且夠用的。

## 九、對既有原則的呼應

| 既有原則 | C 的對映 |
|----------|----------|
| 伏筆不另立登記簿；不另立第二本帳 | 事件流是唯一真實來源，快照＝可重算投影，非第二本帳 |
| 🧊 是指標不是描述（真實來源在幕綱的收） | 事件引用實體／伏筆用名字，不重抄埋／收狀態 |
| 判先後走 `_index` 全書順序 spine（`共同約定.md:44`） | as-of query 直接沿用這條 spine，不新建定序 |
| 避免副本漂移（`章節.schema.md:87`） | 每個事實只寫一次，結構性消除漂移 |
| AI＝共創＋審稿、審稿員非門檻 | C 只記錄與投影狀態，不做創作判斷、不設自動 pass/fail |
| 加欄前先 grep 消費者；觀察到摩擦 ≠ 失敗 | 通用化與落地一律需求驅動、過三關閘門才動系統 |
