# 依幕綱撰寫正文 — 設計書（子專案 B）

- 日期：2026-07-11
- 狀態：設計定案，待審閱
- 範圍：子專案 B 的第一個（也是本階段唯一一個）skill：`write`

---

## 一、Context / 背景與定位

子專案 A 的設計書（`docs/superpowers/specs/2026-07-10-novel-structure-pipeline-design.md`）明言：「依幕綱寫每章的動筆 skill → 子專案 B，另開 spec」。本文件就是那個 spec。

子專案 A 的終點＝幕綱通過測試；子專案 B 的起點＝從「通過測試（或至少已共創完成）的幕綱」出發，寫出真正的散文。`技巧知識庫/創作流程.md` 引洛克教學指出：「靈感→摘要→大綱→幕綱是『創造故事』難的一半，撰寫→修稿是『製造小說』容易的一半」——子專案 B 處理的是「容易的一半」的前半段（撰寫），修稿留待之後。

**定位聲明**（延續全系統「AI 是共創＋審稿，作者掌所有創作決定權」的哲學，但需說明撰寫層的差異）：上游 skill（develop/organize/beat-sheet）產出的是「骨架草稿」供作者拍板；`write` 產出的是「可直接閱讀的散文草稿」——完成度更高，但地位不變：可全盤修改、非受保護、非定案。分章方案與 POV 選擇仍是「AI 提議、作者拍板」。

`技巧知識庫/外部延伸/` 底下四份方法論文件（`場景與情節推進.md`、`展示而非講述與五感描寫.md`、`視角與敘事人稱.md`、`動作場面寫作.md`）已在先前的知識庫擴充階段寫好，且逐一明確聲明是為了本階段準備的執行層依據。本 spec 是把這些已就位的方法論，接上一個實際可執行的 skill 與 schema。

## 二、範圍

管線圖：

```
raw ──（子專案 A：develop → organize → beat-sheet → beat-test）── 幕綱通過測試
                                                                        │
                                                                        ▼ （作者說「開始寫這幾幕/這一章」）
                                                              chapters/chNN.md 散文草稿
                                                                        ┆
                                                                        ┆（write-test 審稿／修稿潤色 = 未來另開 spec）
```

**明確終點**：某範圍的幕（一個 arc 或作者指定的幕範圍）都已寫成含 POV／基調一致、附幕錨點可回溯的散文草稿，並登記於 `chapters/_index.md`。

## 四、資料模型 / Artifact 格式

摘要指向 `結構定義/章節.schema.md`（完整定義見該檔）。三項關鍵設計決定與理由：

1. **不擴充就緒儀表 schema**，改用 `chapters/_index.md` 專職追蹤撰寫覆蓋度。理由：`就緒儀表.schema.md` 的語意是「粗→大綱→幕綱」的成熟度／可否下沉判斷；正文進度是逐幕覆蓋率的不同性質追蹤，比照 `幕綱/_index.md` 對幕綱的角色，不強行塞入子專案 A 已定案的儀表。
2. **幕錨點沿用既有 `幕NNN` 編號**，兩種標記：`<!-- 幕NNN -->`（該幕 Scene 正文）與 `<!-- 幕NNN→幕MMM -->`（Sequel／過渡正文）。硬規則：**一幕不可跨章節切開**，章節邊界只能落在某個錨點之前。理由：延續幕綱「單一真實來源」哲學，不另發明第二套 ID 系統；沒有過渡錨點的話，未來 write-test 的「幕涵蓋完整性」檢查無法區分「渲染某幕」與「單純銜接兩幕」的文字，會誤判為漏幕或誤判為多餘。
3. **幕綱層問題不開新回饋機制**：`幕綱.schema.md` 沒有下游回饋記錄區塊，且幕綱定位為「作者可直接修改的利器」，故 `write` 發現幕綱本身有問題時，只在對話中提示、建議作者直接修改或重跑 `beat-test`，不新增回饋記錄機制。若問題溯源到有回饋記錄區塊的世界觀／00-摘要／角色需求，則沿用既有 `回饋記錄.schema.md` 機制。

章節編號：**全書連續**（`ch01, ch02, …`），不因跨 arc 重置，呼應 `幕NNN` 本身也是跨 arc 連續編號的既有慣例。

## 五、Skill 設計

`write`（完整見 `.claude/skills/write/SKILL.md`）：

- 資源定位：同既有 boilerplate，另解析 `chapters/`。
- 依據：`結構定義/{章節,幕綱,回饋記錄}.schema.md` ＋ 4 份外部延伸方法論 ＋ 既有知識庫子集（氛圍與筆調／因果法則／伏筆與意外／需求四象限／主題題旨）。
- 流程（10 步）：讀輸入 → 就緒提醒（non-blocking）→ 提議分章 → 決定/沿用 POV → 逐幕 Scene 展開＋Sequel 銜接 → 句子層級執行（Show-Don't-Tell／Deep POV）→ 動作場面特判 → 寫入與索引維護 → 基調自我檢查 → 上游問題處理。
- 邊界：不做審稿／不做已寫章節的修稿／不建規模化連貫性基建／不強制章節邊界／不做同章節內多 POV 交錯。

為何本輪只需 1 個 skill：呼應子專案 A「分工明確才拆 skill」的論證方式——`write-test` 是獨立的審稿定位（比照 beat-test/outline-test 的模式），刻意不與 `write` 合併，但本輪不建置，只在 `章節.schema.md` 留延伸點。

## 六、明確非目標

- **`write-test`（正文審稿 skill）**：留待未來另開 spec；本 spec 僅在 `章節.schema.md` 留一節「檢查點（供未來 write-test 參照）」延伸點。
- **修稿／潤色已寫成章節的正文**：留待未來（可能是 `write-test` 之後的另一個 skill，或併入 `write-test` 的處置建議）；本輪 `write` 只做「從無到有」的首次動筆。
- **規模化連貫性基建**（digest／state／index，跨大量章節的一致性追蹤、自動偵測幕綱是否於成章後又被修改）：維持子專案 A 原判定，留給子專案 C。
- **同一章節內多 POV 交錯**（head-hopping 的合法化版本）：v1 不支援；書級的多線視角（不同章節切換 POV，如 gothic_witch 已定的雙線視角結構）不在此限。
- **不強制章節邊界**：分章永遠是 AI 提議、作者拍板，不自動定案。
- **不擴充 `就緒儀表.schema.md`**：理由見第四節。

## 七、Verification

以 `gothic_witch` 的 `story/幕綱/arc01.md`（幕001–幕007，POV 為小女巫）端到端跑 `write`，驗收清單：

- 幕001–幕007 都能在產出章節找到對應 `<!-- 幕NNN -->` 錨點；過渡文字（如有）使用 `<!-- 幕NNN→幕MMM -->`。
- 分章提案落在自然邊界（例如環繞「起火／生還」轉折切成 ch01/ch02 是合理候選），且不把同一幕拆到兩章。
- 全程 POV 一致，無 head-hopping。
- 幕003「花園入畫」：Deep POV 只呈現小女巫視角所見的「遊戲」，不因用詞洩漏讀者該擔心的伏筆。
- 幕004「宅邸焚夜」：她熟睡未直接目擊起火，正文不可跳到全知視角交代火勢經過，只能呈現她甦醒瞬間能感知的片段，事件全貌留給幕005的「事後發現」揭露。
- 幕006「偽善悼詞」：大人的虛偽只透過孩子能觀察到的行為/對白呈現，不由旁白代為戳破。
- 文字展現 Show-Don't-Tell／五感技巧的具體證據；基調吻合「哥特恐怖、黑暗童趣、黑色幽默」三者並存。
- `chapters/_index.md` 正確記錄章節-幕範圍-POV 對應。
