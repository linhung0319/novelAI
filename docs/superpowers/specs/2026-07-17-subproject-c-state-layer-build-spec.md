## 一、Context / 為什麼建、以及這份 spec 和 07-16 的關係

子專案 C 要解的問題（設計書 07-16 第一節）：AI 在第 XX 章協助寫作時，如何精準取得某角色（及其他設定）在第 XX 章當下的狀態（知識前沿／關係／持有…）。現況病根：系統已有「章末狀態快照」機制，但它 **write-only**——`write` 收尾寫它，卻沒有任何 skill 把它當「進場狀態」讀回去；今天「第 XX 章當下狀態」靠 AI 即時重讀前文重建。

**兩輪 RED 的結論**與其對本 spec 的意義：

- 兩輪剝乾淨冷跑（忠於正史／反正史）在現規模（≤12 章、全文可讀）都 **0 複現**。反正史輪額外證明：只要 AI 讀得到全文，它就以**單一真實來源（正文）為準、不會背正史蓋掉正文**。
- 因此 C **不是正確性缺陷的修補**（那條路徑已被乾淨關掉），而是**規模化下的效率／可靠性基建**：當書大到「要用第 30 章的狀態、卻得回頭重讀前 29 章」變得又貴又易漏（context 塞不下、略讀漏掉早章狀態變化）時才咬人。
- RED 這個「正確性閘門」結構上摸不到該規模（冷跑要讀得完 fixture，失敗要 fixture 大到讀不完，互斥）。**本 spec 的驗證改用「限制 context 大小」的模擬**（第五節）來抵達該失敗 regime——這是作者拍板後對驗證方法的關鍵修正。

作者已拍板 C 必須做；本 spec 依四項鎖定決策把 07-16 的提案收斂成可建置的設計。

## 二、四項鎖定決策（brainstorming 收斂）

1. **機械邊界＝程式只做投影，事件由寫作時產**：程式只負責「事件流 → as-of 狀態」的確定性摺疊（零 LLM、100% 可覆算）；每章的狀態變化事件由 `write` 在寫該章時順手產出（它本來就在推理該章狀態）、作者可審。程式不從自由散文機械抽事件（那是脆弱的資訊抽取）。
2. **固定結構＋固定維度枚舉＋自由內容**：事件永遠是 4 欄信封，變化維度是一組封閉、跨小說通用的敘事狀態軸，內容自由文字承載小說特有細節。
3. **schema 固定、又通用不同小說**：靠「固定 4 欄 + 固定維度枚舉」達成「固定」，靠「內容自由 + 實體/伏筆用名字」達成「通用」；一份 schema 適用所有小說，無 per-novel schema。
4. **驗證用限制 context 模擬**：抽掉決定狀態的早章正文、沿用反正史 fixture，強迫進入「讀不完前文」regime，兩臂對照看 C 是否兌現。

## 三、架構總覽與元件邊界

四個各有單一職責、以檔案介面溝通的元件（LLM 產、程式算、作者審三者分離）：

```
寫作當下                          事件真實來源                    查詢當下（章首）
┌─────────────┐   append   ┌──────────────────┐   讀    ┌──────────────────┐
│ write skill │──────────▶│ 狀態事件流（單檔） │◀───────│ 投影程式(fold)    │
│ 寫 chNN 時   │  一行一事件 │ append-only Markdown│        │ 純確定性、零 LLM  │
│ 順手產事件   │            │ 單一真實來源        │        │ 算 as-of 第 N 章  │
└─────────────┘            └──────────────────┘        └────────┬─────────┘
      ▲                            ▲ spine 定序                   │ 投影(實體×維度的≤N最新值)
      │ 作者可審可改        ┌─ story/幕綱/_index.md ─┐            ▼
      └──── 作者 ───────────┘（全書順序，判 ≤N 用）    ┌──────────────────┐
                                                      │ write 章首讀投影   │← 閉合 read-loop
                                                      │ 當進場知識邊界     │
                                                      └──────────────────┘
```

1. **狀態事件流（single file，資料）**：每本書一檔 `story/參照/狀態事件流.md`，append-only，一行一事件。狀態的**單一真實來源**；輕量「章末快照」降級為它之上可重算的投影 cache（不再是第二本帳）。
2. **投影程式（fold，純程式）**：讀事件流 + `story/幕綱/_index.md`（取全書順序 spine），對「實體 × 維度 as-of 第 N 章」做確定性摺疊。零 LLM、可覆算——即「機械式、程式取得」。
3. **write 產出端（LLM 寫、作者審）**：寫 chNN 時把該章狀態變化 append 成事件；只記已寫章；事件純文字、作者可讀可改。
4. **write 消費端（閉合 read-loop）**：章首先呼叫投影程式、拿「本章相關實體 as-of 狀態」當進場知識邊界。**這是 C 兌現價值處**——設計書病根就是「快照只寫不讀」，C 必須補上讀回。

**邊界**：投影程式唯讀、不寫任何檔；事件流只被 write 追加、被程式讀；每元件可獨立測（程式用固定事件流跑純單元測試、不需 LLM）。

## 四、資料模型 / schema

固定 4 欄信封：

```
- 幕NNN（arcAA）· <實體 或 實體A↔實體B> · <變化維度>：<自由內容>
```

- **位置** ＝ `幕號＋所屬 arc`（**不用章號**——章號可亂序、非定序原語；判 ≤N 走 `story/幕綱/_index.md` 全書順序 spine，既有原語，C 不新建）。
- **實體** ＝ 角色/物件名；關係類用 `A↔B`。
- **變化維度** ＝ **封閉枚舉（固定 6 軸，跨小說通用）**：`知識前沿｜關係｜持有｜位置｜能力｜所屬`。「知識前沿」最承重（服務 Deep POV 資訊控制與伏筆回收）。維度不在枚舉內 → 程式報錯。
- **內容** ＝ 自由文字，承載小說特有細節；引用伏筆/實體用名字，**不重抄幕綱埋/收**（避免副本漂移，呼應單一真實來源）。

範例（反正史版偏離，可當 fixture 事件流種子）：
```
- 幕009（arcF）· 哈利 · 持有：隱形斗篷遭麥教授沒收（此後無斗篷，直到 幕012 歸還）
- 幕009（arcF）· 哈利↔榮恩 · 關係：摯友 → 鬧翻冷戰
- 幕011（arcF）· 哈利↔榮恩 · 關係：鬧翻冷戰 → 和好
- 幕005（arcF）· 哈利 · 知識前沿：認定胡奇夫人＝偷石嫌疑人
```

**固定 vs 通用如何同時成立**：schema 固定在「4 欄 + 6 維度枚舉」；通用靠「內容自由 + 名字引用」。維度是普世軸、內容是每本自己的東西。

**schema 落檔位置**：新增 `結構定義/狀態事件流.schema.md`（定義 4 欄、6 維度枚舉、定序規則、與其他軸的分界）；並在 `結構定義/章節.schema.md` 的「章末狀態快照」區塊註明它降級為衍生 cache。

## 五、資料流、錯誤處理、測試

**寫作當下（產出端）**：`write` 寫完 chNN 草稿後，把該章狀態變化 append 成事件；只有已寫章產事件；位置＝該章對應幕號；作者當場審改。

**查詢當下（章首、消費端）**：動筆 chNN 前，呼叫投影程式帶入「目標幕號 + 本章相關實體（POV 主角＋關鍵配角）」，拿回 as-of 投影當進場知識邊界。

**fold 演算法（純確定性）**：
1. parse 事件流每行 → `(位置, 實體, 維度, 內容)`；維度不在枚舉 → 報錯。
2. 每個位置的 arc 對照 `story/幕綱/_index.md` spine 解析成全書序位；無法定位 → 報錯（不靜默丟）。
3. 篩序位 ≤ 目標；按 `(實體, 維度)` 分組；組內取 spine 序最新一筆 → 該 slot 當下值。
4. 輸出各 slot 現值。同輸入永遠同輸出。

**章末快照的處置**：由「手維護、只寫不讀」改為「投影程式從事件流自動生成的衍生產物」（非權威、非手抄）；事件流是唯一真實來源。拔掉「同一事實在每章快照重複」的漂移風險。

**錯誤處理／已知限制**：
- 格式壞行、未知維度、無法定位的幕號 → 程式 lint 報錯，不靜默丟。
- **漏事件**（正文有狀態變化但 write 沒 append）＝ LLM 產出端天生缺口；v1 靠**作者審 + `write-test`** 兜底。程式端一致性校驗（比對事件流 vs 正文/幕綱矛盾漏缺）**延到日後**，不進 v1。

**測試**：
- **fold 程式純單元測試**：固定事件流輸入 → 期望投影輸出（純函數、無 LLM、快）。涵蓋 ≤N 邊界、跨 arc 定序、實體×維度分組、得而復失（同 slot 多事件取最新）、關係雙向。
- **整合／效果驗證**：第六節 context-cap 模擬。

## 六、驗證策略：context-cap 模擬（C 的 GREEN／RED）

補上兩輪 RED 摸不到的規模失敗，用「限制 context」抵達：

1. 沿用 `情境測試/哈利波特實測/GP3-fixture-反正史`；先為它手搭事件流（偏離事件）。
2. **人為抽掉決定狀態的早章正文**（如 ch05 誤疑胡奇夫人、ch09 斗篷沒收＋榮恩鬧翻），只餵「最近 K 章 + 靜態參照」，讓早章落在視窗外，模擬大書「讀不完前文」。
3. 晚章（如 幕011 章首）問同批 as-of，跑 N 次冷跑：

| 臂 | 額外給什麼 | 預測 | 判讀 |
|---|---|---|---|
| **控制臂（無 C）** | 只有截斷後的前文 | 讀不到早章 → 猜錯／回退背正史（反正史 fixture 讓「背正史＝錯」乾淨可判） | 失敗複現 → 規模失敗成立 |
| **處理臂（有 C）** | 額外給程式算出的投影（承載偏離事件） | 沒早章正文也照投影答對 | 答對 → C 兌現價值 |

- **判讀落點**：控制臂**失敗** ＋ 處理臂**成功** → C 被驗證、值得接進正文管線；控制臂在 cap 下**照樣答對** → 這關還不需要 C（cap 沒咬到，調小 K/加大 fixture 再試）。
- 反正史 fixture 是神隊友：context 一抽掉，模型唯一能回退的就是正史、而正史＝錯，失敗訊號乾淨。
- 紀律：K 與 cap 要調到「決定性早章真的出視窗」、白紙寫下 cap 值，否則量到假失敗。

## 七、YAGNI 與系統哲學守則

**v1 只做四樣**：① 事件 schema（`結構定義/狀態事件流.schema.md`）；② 投影 fold 程式（＋純單元測試）；③ `write` 的產事件＋章首讀投影整合；④ context-cap 驗證。**其餘一律不做。**

**守住哲學、明確不做**：
- **不另立第二本帳**：事件流是唯一新增軸（狀態變化）；快照變衍生、伏筆埋/收仍歸幕綱、靜態弧線仍歸角色需求、已承諾事實仍歸 `sync`；事件流引用它們只用名字、不重抄。
- **不自動鏈式觸發**：write 產事件是寫作的一部分、程式在章首被明確呼叫；無常駐程序、無自動全域一致性掃描。
- **不建登記簿**：維度是 schema 裡的固定枚舉，不是 runtime registry。
- **作者掌決定權**：事件可審可改；程式永不改正文或事件。
- **延到日後（有複現需求才做）**：程式端一致性校驗、持久 cache 層、跨 arc／跨集狀態機。

## 八、建置順序（交 writing-plans 展開）

1. 寫 `結構定義/狀態事件流.schema.md`（4 欄、6 維度枚舉、定序規則、與其他軸分界）；`章節.schema.md` 快照區塊註明降級為衍生。
2. 建投影 fold 程式 ＋ 純單元測試（TDD：先寫失敗測試，固定事件流 → 期望投影）。此步零 LLM、可完全離線驗。
3. 用手搭事件流跑 context-cap 驗證（第六節）：確認「控制臂失敗、處理臂成功」。
4. **只有第 3 步驗證通過，才把產事件＋章首讀投影接進 `write` skill**（複現失敗才落地，系統一貫紀律）。

**驗收準則**：fold 程式單元測試全綠 ＋ context-cap 驗證顯示控制臂失敗/處理臂成功 ＋ 系統套件既有三資料夾除了「新增 schema／write 整合」外無破壞性改動 ＋ 章末快照不再手維護（改為衍生或退場）。
