# 以幕綱為核心的前期創作管線 — 設計書（子專案 A）

- 日期：2026-07-10
- 狀態：設計定案，待審閱
- 範圍：整個 AI 輔助小說系統的**第一個子專案**

---

## 一、背景與定位

### 1.1 為什麼重新聚焦

原始計畫書（`novel-writing-system-plan.md`）把幾乎所有工程投資放在**創作流程的後半段**——管理已寫出的章節、對成品做事後一致性檢查（state 快照、digest 摘要、consistency-checker、三層提醒機制）。

但洛克的教學方法論明白指出：**靈感→摘要→大綱→幕綱是「創造故事」的難的一半，撰寫→修稿是「製造小說」的容易一半**，而且「真正的修稿時機就是在安排幕綱的時候」「在幕綱刪掉一句話總比最後刪到五千字好」。
> 【勘誤 2026-07-12】「難的一半／容易一半」的說法不見於原文（見 `技巧知識庫/創作流程.md` 忠實性註記：原文說幕綱是「最辛苦的階段」），正確表述是「創作故事／製造小說」二分；「各占成敗一半」的原文所指是「原創概念」與「幕綱編排」。保留原文供歷史比對。

用 Elon Musk 的問法（先確認問題是否值得解決）：原計畫解決的是一個真問題（規模化下的連貫性失控），但**不是最痛的那個問題（動筆前的結構是否健全）**。最痛的第 1、第 2 名痛點——「結構/大綱牆」與「幕的因果健全」——在原計畫裡只是一個扁平的 `outline.md`，沒有結構化支援。

### 1.2 核心設計決定

1. **系統以幕綱/結構為核心重新設計**，把「測試」從成品散文 shift-left 到動筆前的幕綱層。
2. **嚴格分離「不變的系統/技巧」與「每本書都不同的內容」**（見第三節）。
3. **幕綱升為一等公民**，以結構化 Markdown 表示，讓結構測試可跑。

### 1.3 「小說測試」的定位（誠實聲明）

程式測試精準、可自動 pass/fail；小說測試是模糊的、需要判斷的。本系統的測試是**LLM 判斷型的「審稿員」，把可疑點攤開讓作者判斷，不做自動綠燈/紅燈的門檻**。全自動精準判定不現實；「一個永遠盯著全書結構的資深編輯」現實且高價值。

方法論交給我們 5 條半可檢查的不變量，構成測試套件（見第七節）。最有價值的執行時機是**幕綱層**（等同在設計評審抓 bug，而非 production）。

## 二、範圍

### 2.1 本 spec 的三階段管線（每本書都走一遍）

```
raw data（任意格式的瑣碎筆記／靈感片段／半成品大綱、幕綱、角色設定）
   │  ← 作者說「幫我把 raw/ 整理成符合結構的大綱和幕綱」
   ▼
符合結構的產出（摘要／大綱／角色需求／幕綱）   ← 「抓重點」那一步
   │  ← 作者補充 or Claude 生成幕綱，反覆修改
   ▼
【幕綱通過測試】  ← 本 spec 的終點
   ┆
   ┆（依幕綱寫每一章 = 子專案 B，另開 spec）
```

### 2.2 明確終點

本 spec 做到：`raw` → 整理 → 結構化 `story/` → 幕綱測試通過。

### 2.3 非目標（不在本 spec）

- **依幕綱寫每章的動筆 skill** → 子專案 B。
- **規模化連貫性基建**（digest / state / index）→ 子專案 C，等累積足夠章節再做。
- **完整技巧整理**（對白、戲劇手法、結局、創作技巧懶人包等）→ 子專案 D；本 spec 只提煉 A 需要的技巧子集。
- 原計畫的 symlink `state/current`、三層提醒機制 → 已判定為過度設計，不採用。

## 三、架構：系統套件 vs 每本書內容

核心洞察：系統與寫作技巧**每次寫作都需要且不變**；小說內容**每本都不同**。兩者嚴格分離，正好對應 Claude Code 的個人層 / 專案層分層。

### 3.1 第一部分：寫作系統套件（不變、跨所有書、version-controlled）

**這個 `novelAI` repo 本身就是「系統套件 repo」**——它裝著方法論與 `Data/` 技巧素材，天生是系統層，不是某本書。

```
寫作系統 repo（= 現在的 novelAI）
├── 結構定義/                # 「符合專案結構」到底長怎樣（schema）
│   ├── 摘要.schema.md
│   ├── 大綱.schema.md
│   ├── 幕綱.schema.md
│   ├── 角色需求.schema.md
│   └── 結構.schema.md
├── 技巧知識庫/              # 從 Data/ 提煉的 craft knowledge（承載「方法」）
│   ├── 創作流程.md          # 六步驟、哪一層壞了的診斷
│   ├── 因果法則.md          # 前果=後因、幕的五要素、孤兒
│   ├── 結構公式.md          # 三幕劇／九階段／七問題
│   ├── 需求四象限.md        # 期盼／想要／落差／需要
│   ├── 伏筆與意外.md
│   └── 主題題旨.md          # 「因為 X，導致 Y」
├── skills/                  # 個人層 skill 的原始檔（見第六節）
│   ├── organize/
│   ├── beat-sheet/
│   └── beat-test/
├── Data/                    # 洛克教學原始素材（技巧知識庫的來源）
└── docs/superpowers/specs/  # 本設計書所在
```

**部署：** 系統 repo 維護原始檔；`skills/` 安裝到 `~/.claude/skills/`（個人層），不管作者 `cd` 到哪一本書的資料夾都能用。安裝方式（複製 / symlink / plugin）屬實作細節，設計上要求 skill 不依賴特定書資料夾的絕對路徑。

### 3.2 第二部分：每本書的資料夾（每本都不同）

```
某本書/
├── raw/        # 作者丟進來的任意格式瑣碎筆記、靈感、半成品
├── story/      # 「整理」後符合結構的產出
│   ├── 00-摘要.md
│   ├── 01-大綱.md
│   ├── 幕綱/
│   │   ├── _index.md
│   │   ├── arc01.md
│   │   └── arc02.md
│   └── 參照/
│       ├── 角色需求.md
│       └── 結構.md
└── chapters/   # 依幕綱寫的實際章節（子專案 B）
```

書資料夾**不套用**原計畫「bible 受保護」的規則——在子專案 A，`story/` 全是正在積極創作、頻繁修改的工作文件，AI 可直接與作者共同修改。檔案保護模型留給後期（動筆後設定凍結）。

## 四、資料模型 / Artifact 格式

所有格式的權威定義放在系統套件的 `結構定義/*.schema.md`；以下是設計摘要。

### 4.1 `00-摘要.md`（主線的壓縮測試）
洛克的「三長度壓縮」：寫不出一句話 = 還沒搞懂自己的主線。

```markdown
# 摘要

## 一句話
## 50 字
## 100 字
## 200 字

## 題旨
因為（X：行為／觀念），導致（Y：結果）。   ← 可檢查的最小形式

## 高概念（Look / Hook / Book）
```

### 4.2 `01-大綱.md`（完整故事）
連續敘述、**結局與主題已定**、不講究文筆。開頭標註選用的結構公式。結局是回頭修正前文的地方（改結局 → 必須回頭檢視前文邏輯）。

### 4.3 `幕綱/arcNN.md`（脊椎）
每幕一個 section，固定欄位；`[[連結]]` 讓因果與伏筆可機械抽取。

```markdown
## 幕023 · 冰原奪髓
- 角色：林小凡、血衣老者
- 時空：北域冰原／第三夜
- 行動：闖禁地取寒髓
- 衝突：老者守護、出手阻攔
- 結果：奪得寒髓，左臂遭封脈
- 前因：[[幕019]]（得知寒髓解毒）
- 伏筆：埋[[伏筆:封脈之術]]
- 結構階段：磨練成長

（下方可續寫自由靈感註備，捕捉當初的畫面…）
```

**連結規範（機械可抽取）：**
- `[[幕NNN]]`：指向作為前因的幕。
- `埋[[伏筆:名稱]]` / `收[[伏筆:名稱]]`：伏筆的埋設與回收點。
- **單一真實來源：** 伏筆不另立登記簿。閉合狀態直接從幕綱裡的 `埋`/`收` 標記算出（要索引就由 AI 生成視圖，作者只維護幕綱一份）。刻意避開「三層防護才記得同步」的 smell。

### 4.4 `參照/角色需求.md`
每個主角一個 section：**需求衝突四象限**（期盼＝動力／想要＝誤導／落差＝阻礙／需要＝圓滿）+ 預期弧線。作為「角色需求弧線測試」的基準。

### 4.5 `參照/結構.md`
選用哪條公式（三幕劇 / 九階段 / 七問題），以及每個必要階段對應到哪幾幕。作為「結構完整性測試」的基準。

### 4.6 `幕綱/_index.md`
幕綱的索引：各 arc 的範圍、標題、選用公式概覽。作為 AI 檢索入口。

## 五、三階段管線與「整理」操作

「整理」是作者描述的核心操作，也是本系統的冷啟動主力。

**運作：** 作者在某本書資料夾說「幫我把 `raw/` 整理成符合結構的大綱和幕綱」→ Claude 載入系統套件的**結構定義**（知道目標長怎樣）+**技巧知識庫**（知道怎麼抓重點、怎麼判斷因果）→ 讀 `raw/` 雜亂素材 → 產出/更新 `story/` 下符合 schema 的檔案。

**診斷缺口（關鍵行為）：** 依洛克「哪一層壞了」的觀念，整理後回報各階段狀態——哪些有了、哪些缺、哪些弱（如「還沒有結局 → 大綱無法定稿」「題旨待定」「幕023 結果沒有催發下一幕」），指引作者下一步補哪裡。

## 六、Skills 設計

分工：**技巧知識庫承載「方法」，skill 承載「操作」。** 各階段的創作方法（摘要三長度、大綱結局→主題、幕的五要素）寫在知識庫，skill 引用它。因此核心只需 3 個 skill。

命名用英文 kebab-case（slash command 較穩），內文與知識庫全繁中；名稱可調整。

### 6.1 `organize`（整理）— 冷啟動主力
- 觸發：自然語言「把 raw/ 整理成符合結構的…」
- 載入 `結構定義` + `技巧知識庫`；讀 `raw/`；產出/更新 `story/`（摘要／大綱／角色需求／幕綱）
- 完成後輸出**缺口診斷報告**
- 摘要、大綱的逐步精修不另開 skill——方法已在知識庫，`organize` + 對話即可。（未來若需固定觸發，再加 `/summary`、`/outline` 作為低成本擴充點。）

### 6.2 `beat-sheet`（幕綱共創）— 針對「最大難關」
- 從 `大綱` 生成候選幕綱（結構化 Markdown、五要素、`[[連結]]`），供作者重排／增刪／修改
- 維護 `參照/結構.md`（幕 ↔ 結構階段對應）
- **AI 角色：共創 + 審稿。** AI 提草稿降低白紙恐懼，但所有創作決定權在作者。

### 6.3 `beat-test`（幕綱測試）— 核心價值，把測試 shift-left
- 對幕綱（可指定 arc 範圍）跑第七節的 5 項測試
- 輸出結構化報告；定位為「審稿員」而非 pass/fail 門檻
- MVP 在主 session 執行；若日後全書測試的 context 過大，再考慮改為獨立 subagent（非本 spec 範圍）

## 七、測試層：5 項結構不變量

| 測試 | 檢查什麼 | 性質 | 基準來源 |
|------|----------|------|----------|
| **因果連續性** | 每幕結果是否催發下一幕動機；有因無果／有果無因的孤兒；未連到主線的幕 | `[[前因]]` 連結可機械抽取 + LLM 判語意 | 因果法則.md |
| **伏筆閉合** | 每個 `埋[[x]]` 是否有後續 `收[[x]]`；結尾有無懸空線 | 標記可確定性檢查 + LLM 判回收品質 | 伏筆與意外.md |
| **結構完整性** | 對照選用公式，缺哪個必要階段／重複／錯序 | LLM 對照 `結構.md` | 結構公式.md |
| **角色需求弧線** | 需要是否拖到最後才滿足；轉變是否掙來（非機器神） | LLM 對照 `角色需求.md` | 需求四象限.md |
| **主題與 POV** | 立場有無中途詭變（對照題旨「因為 X 導致 Y」）；第一人稱知識洩漏 | LLM | 主題題旨.md |

**輸出格式（每個問題）：** 幕 id ｜ 測試類型 ｜ 嚴重度 ｜ 說明 ｜ 建議修正。作者決定如何處理。

## 八、技巧知識庫子集（本 spec 需從 Data/ 提煉）

只提煉子專案 A 用得到的包（對白／戲劇手法／結局／創作技巧等留給子專案 D）：

- `創作流程.md` — 六步驟、摘要 vs 大綱 vs 幕綱之別、「哪一層壞了」診斷（來源：【小說】【幕綱】）
- `因果法則.md` — 前果=後因、幕的五要素（角色／時空／行動／衝突／結果）、有因無果／有果無因、貴精不貴多（來源：【幕綱】）
- `結構公式.md` — 三幕劇、起承轉合、七問題公式（目標→阻礙→努力→結果→意外→轉彎→結局）、編劇九階段（來源：【編劇結構】【故事】）
- `需求四象限.md` — 期盼／想要／落差／需要、兩難設計、馬斯洛需求、反派正當需求（來源：【獨門密技】【人物】）
- `伏筆與意外.md` — 伏筆閉合、意外須源於主角努力（非機器神）、謎三角雙重誤導（來源：【編劇結構】【獨門密技】【寫作技巧懶人包】）
- `主題題旨.md` — 題旨公式「因為 X 導致 Y」、主題漂移／立場詭變、主題三角（來源：【主題】）

## 九、建置工作清單

1. 建立系統 repo 結構：`結構定義/`、`技巧知識庫/`、`skills/`
2. 從 `Data/` 提煉第八節的技巧知識庫子集
3. 撰寫 `結構定義/*.schema.md`（摘要／大綱／幕綱／角色需求／結構）
4. 撰寫 3 個 skill：`organize`、`beat-sheet`、`beat-test`
5. 定義每本書資料夾模板（`raw/`、`story/`、`chapters/`）
6. 定義 skill 安裝到 `~/.claude/skills/` 的方式（不依賴書資料夾絕對路徑）
7. 以一份小樣本（一段弧線）端到端驗證：raw → organize → beat-sheet → beat-test

## 附錄：關鍵術語

- **幕**：角色在某時某地做了某個行為；五要素 = 角色／時空／行動／衝突／結果
- **幕的因果**：前一幕的結果 = 下一幕的動機
- **摘要 vs 大綱 vs 幕綱**：摘要＝主線幾句話（壓縮測試）；大綱＝完整故事、結局主題已定（連續敘述）；幕綱＝故事拆成有序離散的幕（可執行藍圖）
- **需求衝突四象限**：期盼（動力）／想要（誤導）／落差（阻礙）／需要（圓滿）
- **題旨**：帶結論的主題，最小形式「因為 X，導致 Y」
