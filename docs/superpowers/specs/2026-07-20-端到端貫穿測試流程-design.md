# 端到端貫穿測試流程 — 設計 spec

> 日期：2026-07-20　狀態：設計已與作者確認，待寫成方法論 doc
> 定位：一套**可重用**的測試流程——用一本現成好書當素材，由自動化的「模擬作者」驅動系統，跨多個 session 從 `raw/` 一路把整卷小說內文建構出來，藉此測「AI 小說寫作輔助系統」在**真實互動 + 大量內容**下的效果與問題。首跑標的＝`examples/一世之尊/卷一`。
>
> 上游關聯：`skill覆蓋情境測試.md`（路由回歸）、`情境測試/哈利波特實測/`（逐 skill 品質）、`情境測試/芯片巫師/`（管線貫穿·靜態 fixture）。本流程＝**芯片巫師式管線貫穿測試 + 動態神諭作者 + 多 session 續跑**。

---

## 1. 目的與定位

系統經過大修正，要重新測功能。既有測試套件各測一個面向：

| 既有套件 | 測什麼 |
|----------|--------|
| `skill覆蓋情境測試.md` | 路由回歸：作者求助→有沒有對的 skill／檔位承接、不誤觸 |
| `哈利波特實測/` | 逐 skill 產物品質（HP canon 當 ground truth） |
| `芯片巫師/` | 管線貫穿（原創書＋**靜態** fixture），重壓正文層 |

**本流程補的面向＝端到端互動整合 + elicitation 易用性 + 規模/長程效應**：一個素材**分層且不完整**的作者，能不能透過系統把整條管線（raw → 設定 → 大綱 → 幕綱 → 正文）走完；系統一路上引導/追問/路由/承接/產出得好不好；而且是在**多 session 累積大量內容**的壓力下。

相對於 `芯片巫師/`，唯一但關鍵的升級：**作者從靜態 fixture 變成「被問時能即時翻查原著」的動態神諭**——這也是為什麼要用一本現成好書（一世之尊）而非原創草稿，見 §4。

## 2. 兩個決定（已與作者確認，是全案地基）

1. **量的是「系統輔助力」，不是「重建保真度」。** 一世之尊 canon 只當「真實作者素材來源 + 作者臨場改編的一致性參照」，**不是**拿來比對的正解。系統是創作輔助、作者有自由，產物本就不必貼合原著；若去獎勵「貼合 canon」＝獎勵背誦，正是選這本書要避開的失敗。
2. **編排＝檔案共享 + 冷啟單步 + 作者當神諭。** 連續性靠**磁碟上的檔案**，不靠 agent 記憶；契合現行 HP 方法與「每個 skill 是一次策展式操作」的系統設計。

## 3. 三個角色與防火牆

| 角色 | 是什麼 | 能讀 | **絕不能讀** |
|------|--------|------|--------------|
| **導演（中介）** | 主 agent。排步驟、**逐字**轉述問答、留痕。本身**不注入** canon 知識，只當訊息匯流排 | 全部 | —（但對系統↔作者間只逐字轉述，不改寫、不順手補） |
| **系統 agent（受測體）** | 每個管線步驟開一個**冷** subagent，跑對應 skill | `<書>/` 書資料夾＋系統套件（結構定義／技巧知識庫／.claude/skills） | `examples/`、`情境測試/`（**硬防火牆**） |
| **作者 agent（神諭）** | 每次被問時開一個冷 subagent，扮作者答話 | `<書>/raw/`＋`情境測試/<書>/`＋（fallback）`examples/<書>/` | — |

**硬防火牆是測試有效性的地基**：系統 agent 完全不讀原著，否則無法區分「系統輔助產出」與「AI 背誦原著」。作者拍板：**可以，硬隔離**。

## 4. 資料佈局與知識邊界

```
<書>/                        ← 書資料夾（版控、累積共享狀態）
  raw/                       ← 系統『讀得到』的素材（凌亂、不完整）
  story/ chapters/           ← 系統產出（下游逐步累積）
情境測試/<書>/                ← 作者『腦裡』：系統看不到，只能靠『問對問題』撈出（測 elicitation 的核心）
                             ← 同目錄放測試 meta：session 台帳、fallback log、每 session 記錄
examples/<書>/…              ← 原著：作者臨場改編的神諭＋『人工通讀評估』的合理性參照（系統絕不可讀）
```

**raw vs 情境測試 的切分原則**（切法本身是方法論一步，由讀過原著的 subagent 在 P session 產生）：

- `raw/` = 真作者會**寫下**的凌亂靈感：幾個場景、基調、主角種子、也許結局念頭——**片段、無序、不完整**，逼系統去**結構化／追問／擴充**而非抄寫。
- `情境測試/<書>/` = 作者**腦裡有、但不會主動寫**的連接組織：次要角色細節、世界規則、特定 reveal——只在系統**問到**時才吐。
- 切分要刻意製造「系統得做真工」的缺口。答案不能都塞進 `raw/`，否則 elicitation 測不到。

**為什麼作者能翻 canon 是升級而非作弊**：原創草稿裡作者的「腦」是唯一來源，臨場發明只能和當下瞎編一樣好。有 canon 撐腰，作者能像「腦中早有豐富自洽世界的能幹作者」那樣，隨問給出好且不矛盾的細節——這正是 end-to-end 易用性測試需要的忠實作者替身。代價（過度翻查→重建偏誤）由 §7 的 fallback log 盯住。

## 5. 執行迴圈（每個管線步驟）

1. 導演選下一階段＋範圍。
2. 冷啟**系統 agent**（帶防火牆）＋用「作者口吻」下該階段指令。
3. 系統做事；**需要作者輸入時，回傳問題**（不靠假設硬闖）。
4. 導演**逐字**轉問 → 冷啟**作者 agent** → 得答案（＋fallback log）→ **逐字**轉回。
5. 系統完成，寫入 `<書>/` 書資料夾；系統會照常 inline 跑自己的審稿 skill（outline-test／beat-test／write-test）。
6. 導演 **commit/快照**書資料夾（版控＝天然快照，可診斷可回滾），並更新 session 台帳＋每 session 記錄。→ 迴圈。

## 6. 多 session 切分與續跑協定 ⭐（本方法論核心）

目標是**跨多 session 把整卷內文建構出來**，故「怎麼切 session、新 session 怎麼續」必須事先規劃好——這是本流程與單發的 `芯片巫師/` 的關鍵差異。

**Session 切分原則**：一個 session = 在有界範圍上跑**完整的幾個管線階段**，**絕不切斷一次 elicitation 迴圈**（半途留懸空線）。大小以「塞得下雙 agent 轉述開銷後仍有餘裕」為準（比單 agent 跑得少）。粗略序列：

| Session 型 | 做什麼 | 產出 |
|-----------|--------|------|
| **P（前置，每本一次）** | 讀原著 → 按切分原則產 `raw/`＋`情境測試/<書>/`；init 書資料夾（複製 `書本模板/`） | 素材＋神諭腦＋空書架 |
| **S1** | develop（raw→摘要＋設定冷啟動） | `story/00-摘要`、`story/設定/` |
| **S2** | organize→outline-test（→大綱） | `story/01-大綱` |
| **S3…** | 逐 arc：beat-sheet→beat-test | `story/幕綱/arcNN` |
| **…** | 逐 arc：write→write-test→revise | `chapters/` |

**續跑協定**（每 session 頭尾固定動作）：

- **開場三讀**：① 本方法論 doc　② **session 台帳**（上個 session 做到哪、下一步、有無待決 elicitation）　③ 書資料夾的 `就緒儀表`（系統自維的 in-world 進度）。
- **收場二寫**：① 更新 session 台帳　② commit 書資料夾。
- **兩本台帳分離**：`就緒儀表` ＝系統維護的 in-world 產物（真的，在書資料夾內）；**session 台帳＋fallback log ＋每 session 記錄 ＝測試 meta，放 `情境測試/<書>/`，不進書資料夾**——否則系統 agent 會讀到測試提示（HP 學到的污染教訓）。
- **狀態永不 reset、累積前進**：這正是規模測試的本體（見 §7）。

## 7. 評估模型

**自動化只負責『忠實產出 + 留痕』，不自評打分。真正的評估＝作者本人在全部內文產出後的人工通讀，憑讀感找問題。** 卷一之長是刻意的**規模/長程效應**壓力測試。

自動化迴圈只做三件事：

1. **Inline 跑系統自己的審稿 skill**（outline-test／beat-test／write-test）——因為真作者用這系統本來就會跑，是管線的一部分，非另加的評分層。
2. **honesty guards**：硬防火牆、fallback log、purity side-lane（見下）。
3. **可追溯留痕**：每 session 記錄「做了什麼決定、哪些作者答案來自腦 vs canon-fallback、每階段 git 快照」。用途——作者日後讀到第 N 章讀感不對時，能**回溯**：壞正文 → 哪個 write session → 哪個幕綱 → 哪個設定決定 → 哪次 elicitation。

**規模/長程效應（本流程特有的觀察對象）**：狀態累積不 reset，刻意盯只在大量內容下才破的東西——跨 arc 伏筆償還、多章後的設定自洽、角色狀態漂移、基調一致性。**多 session 累積建構本身就是規模測試。**

**purity side-lane（測試自身的純度守門）**：系統 agent 就算不讀 examples/，仍可能**認出**一世之尊而憑記憶重建。持續盯「系統吐出沒被告知的原著專屬細節」；命中＝污染，記一筆，**非扣分項而是有效性警訊**。

**post-hoc 診斷 lens（給作者人工通讀時用，非自動門檻）**：需要時可沿五維檢視某段——① 路由　② elicitation 品質（fallback log 當量尺）　③ 上下游承接　④ 產物品質　⑤ 摩擦與缺口（真作者想要、系統接不住 → 未來新功能候選，同 HP gap harvest）。

## 8. 陷阱與有效性威脅（集中列，含作者邀請反駁的點）

- **污染**（HP 教訓）：端到端本就累積上游，屬正常；但**測試 meta 別漏進書資料夾**；日後若做 ablation 對照組須先剝乾淨並連 git 工作區狀態一起剝。
- **導演注入**：導演也認得原著，故系統↔作者間**只逐字轉述**，不改寫不補話。
- **作者過度翻原著→重建偏誤**：最小發明、按問題具體度回答、腦裡真沒有才撈、每次撈都記 log。
- **辨識性洩漏**：選一世之尊（比 HP 難背）已降風險，purity side-lane 續盯。
- **鷹架到底有沒有用（循環性）**：綠燈只證明「管線不礙事且產出連貫」；要把品質**歸功於 skill／schema**，需偶爾 ablation（剝掉某 skill 指引冷跑、看品質是否掉）——列為**選配深挖**，借 HP RED 紀律，非每跑必做。
- **自評循環**：第 4 維品質借系統自己的審稿 skill，屬「自己改自己考卷」；靠**最終人工通讀**當獨立裁判化解，canon 只當合理性參照。

## 9. 可重用化（換任意書）

流程零硬編一世之尊。換書＝跑一次 P session（挑書→按 §4 切分原則產 raw＋情境測試→複製 `書本模板/` init 書資料夾→建 session 台帳）→ 照 §6 的 S1…Sn 序列跨 session 跑。一世之尊只是首跑的 worked example。

## 11. 首跑範圍（一世之尊，可調）

目標＝**卷一全部內文**，明確接受需跨多 session 逐步建成。P→S1→S2→逐 arc 拆幕→逐 arc 寫正文，狀態累積。範圍是流程的參數，非流程本身；換書可自訂。
