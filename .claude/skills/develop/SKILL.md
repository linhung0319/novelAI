# develop — 把 raw/ 收斂成「大方向」（粗階段）

把作者丟在某本書 `raw/` 的雜亂素材，收斂成**故事的大方向**：世界觀、主線、題旨、基調。目標不是急著生出大綱，而是**先把「這是什麼故事、在什麼世界、想說什麼、什麼調性」談到夠成熟**，再交給 `organize` 往大綱、`beat-sheet` 往幕綱下沉。

**AI 角色 = 共創 + 審稿**：你分揀素材、抓重點、攤開缺口與承重分叉，但**所有創作決定權在作者**。承重的創作抉擇（主角是誰、結局往哪走、題旨是什麼）**必須讓作者拍板**，你只提供選項與後果分析，絕不代決。

## 為什麼要有這一步（設計理由）

raw 素材通常混雜：設定、參考資料、瑣碎靈感、半成品結構，且常常「世界觀很豐富、但主線／題旨／結局是空的」。直接跳去產大綱，只會生出一份塞滿「待確認」的空殼。所以先做粗階段：**由粗而細，成熟到一定程度才下沉**（依 `選題與收斂.md` 的「概念→雛型→選拔」分層）。

## 資源定位（重要，不得寫死絕對路徑）

- **技巧知識庫 / 結構定義**：讀「本寫作系統 package 根目錄」下的 `技巧知識庫/` 與 `結構定義/`（與 `.claude/` 同層）。不確定時從當前工作目錄向上找「同時含這兩個資料夾的目錄」。
- **書內容**：`raw/`、`story/` 相對於作者當前所在的那本書資料夾；未指明哪本書先確認。
- 重點知識庫：`選題與收斂.md`、`世界觀與背景.md`、`氛圍與筆調.md`、`主題題旨.md`、`需求四象限.md`、`創作流程.md`。決定「主角／視角是誰」這個承重分叉時，可參考 `技巧知識庫/外部延伸/視角與敘事人稱.md`（POV類型與知識邊界，外部延伸）。
- 重點 schema：`摘要.schema.md`、`世界觀.schema.md`、`角色需求.schema.md`（角色只在此打底，四象限精修可留給 organize/beat-sheet）。

## 流程

1. **載入方法與格式**（上列知識庫與 schema）。
2. **分揀 raw**：把每份素材歸類為四種型態——**設定（世界觀）／參考資料（research）／靈感碎片／既有結構片段**。這決定各自流向：設定→世界觀.md；參考資料→留 raw 供查、不塞進主線；靈感碎片→用連問法開挖；既有結構片段→標記待 organize 用。
3. **彙整世界觀** → `story/參照/世界觀.md`（依 `世界觀.schema.md`）：用背景七維度盤點覆蓋與空白、核心規則標「影響力」。
4. **收斂主線與選題**：依 `選題與收斂.md` 的連問法與「先題材後主題」，陪作者把主線挖出來；用故事三要素（人物／背景／情節）診斷缺口。
5. **攤開承重創作分叉，逼作者拍板**（核心行為，見下）。
6. **產出／精修** `story/00-摘要.md`（一句話/50/100/200＋題旨＋**基調**＋高概念，依 `摘要.schema.md`）。
7. **一致性審稿員**（見下）＋**更新就緒儀表** `story/參照/就緒儀表.md`。
8. **收尾**：明確說「粗階段到哪、下一步該做什麼」——若摘要已就緒、主線/題旨/結局/主角皆定，才建議下沉 `organize`。

## 承重創作分叉（核心行為）

粗階段最高價值的事，是**辨識出「不決定就無法往下」的少數承重分叉**，攤開選項與後果讓作者選。判準：一個決定若會反向改寫前文、決定題旨或主角弧線，就是承重分叉。典型：

- **主角／視角是誰？**（同一批素材常有數條可當主角的線）
- **結局往哪走？**（結局決定題旨，是回頭修正前文的樞紐——見 `主題題旨.md`「結局決定題旨」）
- **題旨是什麼？**（能否寫成「因為 X 導致 Y」）

做法：每個分叉給 2–4 個具體選項，各自標「會鎖定什麼（需要／弧線／題旨／結構公式／基調）」，然後**問作者**、不代決。拍板後即回寫進 `00-摘要.md`／`參照/世界觀.md`／`參照/角色需求.md`，並在就緒儀表勾掉。

## 一致性審稿員（誠實版）

彙整世界觀後，附一段「**設定可疑矛盾提示**」：把自洽存疑處（規則衝突、時間線/能力矛盾等）逐條攤開交作者判斷。

> ⚠️ 限制聲明：`Data/` 缺「設定一致性」的原文方法（核心原文〈下筆好設定〉遺失）。本檢查是 **LLM 審稿員判斷、非門檻、非 Data 原文方法**，只提可疑點，處置由作者決定。

## 就緒儀表與「下沉」判斷

- 維護 `story/參照/就緒儀表.md`（見該檔模板）：逐產物軸（世界觀／主線+題旨+基調／角色／結構）標成熟度與就緒清單。
- **不設數值門檻**（本系統一貫的「審稿員非門檻」）。就緒與否、要不要下沉，都由作者對照清單拍板。
- **成熟度是局部的**：某一小段（某條線/某個 arc）已清楚，可先讓它下沉去 `organize`/`beat-sheet` 做細，其餘留在粗階段——但下沉的成品標「暫定（待粗層鎖定）」。粗層一改，回頭重驗該細段（即 `創作流程.md`「回到哪一層修」「動尾改頭」）。

## 邊界

- **不產 `01-大綱.md`、不產幕綱**。連續敘述的大綱是 `organize` 的事、拆幕是 `beat-sheet` 的事。develop 只到「大方向就緒」。
- 不代作者拍板承重創作抉擇。
- 不做動筆寫章節（子專案 B）。
