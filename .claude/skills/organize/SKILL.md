# organize — 把已就緒的大方向整理成大綱

把**大方向已收斂**的素材，整理成結構完整的 `story/01-大綱.md`（連續敘述、結局與題旨已定）與角色/結構參照，並診斷「還缺什麼、哪裡弱」。

> ⚠️ **先確認粗階段已完成**：organize 假設 `story/00-摘要.md`（含題旨、基調）與 `story/參照/世界觀.md` 已大致成形、主線/結局/主角已拍板。**若 raw 還很亂、主線/題旨/結局未定，先用 `develop` 做粗階段**（分揀素材、彙整世界觀、攤開承重分叉），再回來 organize。發現粗層未就緒時，就把控制權交回 develop，不要在大綱裡硬塞一堆「待確認」。

**AI 角色 = 共創 + 審稿**：你提草稿、抓重點、指出缺口，但**所有創作決定權在作者**。不要替作者拍板創作選擇，而是攤開選項與診斷讓作者決定。

## 資源定位（重要，不得寫死絕對路徑）

- **技巧知識庫 / 結構定義**：讀取「本寫作系統 package 根目錄」下的 `技巧知識庫/` 與 `結構定義/`（與 `.claude/` 同層）。若不確定根目錄，從當前工作目錄向上尋找「同時含 `技巧知識庫/` 與 `結構定義/` 的目錄」。
- **書內容**：`raw/`、`story/`、`chapters/` 一律相對於「作者當前所在的那本書資料夾」。若作者未指明是哪本書，先確認。

## 流程

1. **載入方法與格式**：讀 `結構定義/*.schema.md` 與 `技巧知識庫/*.md`。重點：`創作流程.md`、`因果法則.md`、`主題題旨.md`、`需求四象限.md`、`結構公式.md`。
2. **讀粗階段產物與 raw/**：先讀 `story/00-摘要.md`（含題旨、基調）與 `story/參照/世界觀.md`——這是 develop 收斂出的大方向。再讀 `raw/` 補既有的大綱/幕綱片段、角色與情節細節。**若摘要缺主線/題旨、或世界觀.md 不存在，判定粗階段未就緒 → 停下並建議先跑 `develop`。**
3. **產出／更新 story/**（依 schema）：
   - `story/01-大綱.md`——把摘要擴寫成連續敘述的完整故事；標註選用結構公式；**結局與題旨已定**。
   - `story/參照/角色需求.md`——各主角四象限 + 預期弧線。
   - `story/參照/結構.md`——選用公式與（如已拆幕）幕↔階段對應。
   - 摘要若需微調可順手更新，但主線/結局/主角的**承重改動應退回 develop**，別在此偷偷改掉作者的大方向。
   - 更新既有檔時，保留作者已寫定的內容，只補缺與標示待確認處。
4. **拆幕交 beat-sheet**：若 raw 已有足夠幕素材可產雛形 `arcNN.md`，但完整拆幕、排序、伏筆是 `beat-sheet` 的職責。
5. **輸出缺口診斷報告 + 更新就緒儀表** `story/參照/就緒儀表.md`（見下）。

## 缺口診斷報告（核心行為）

依 `創作流程.md` 的「回到哪一層修」觀念，整理後回報各階段狀態——**哪些有了、哪些缺、哪些弱**，並指引下一步。範例語氣：

- 「還沒有結局 → 大綱無法定稿（結局是回頭修正前文的樞紐）」
- 「題旨待定 → 摘要的『因為 X 導致 Y』填不出，主線可能還沒收斂」
- 「幕023 的結果沒有催發下一幕（有果無因/有因無果），建議修結果或補前因」
- 「主角『需要』不明確，角色弧線測不了」

報告格式建議：分「✅ 已具備／⚠️ 偏弱／❌ 缺」三類，每條附「建議下一步」，最後點出**最該先補的一到兩項**（通常是前三步：主線／題旨／結局）。

## 邊界

- **上游**：粗階段（分揀素材、彙整世界觀、收斂主線/題旨、攤開承重分叉）是 `develop` 的事；organize 不做冷啟動，發現大方向未就緒就退回 develop。
- **下游**：不負責把大綱拆成完整幕綱（那是 `beat-sheet`），但可產出雛形。大綱/摘要大致定稿後，可用 `outline-test` 跑形式化的結構不變量測試（organize 本身的缺口診斷是共創過程中的非正式版本，outline-test 是定稿前的正式審稿）。
- 不做動筆寫章節（子專案 B）。
- 大綱的逐步精修不另開 skill——用本 skill + 對話即可。
