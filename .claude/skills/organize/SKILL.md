# organize — 把已就緒的大方向整理成大綱

把**大方向已收斂**的素材，整理成結構完整的 `story/01-大綱.md`（連續敘述、結局與題旨已定）與角色/結構參照，並診斷「還缺什麼、哪裡弱」。**若只有某一段（arc）已收斂、全書結局/題旨仍未定**，改整理成局部的 `story/大綱/arcNN.md`（見 `結構定義/大綱.schema.md`「局部下沉（scoped）大綱」節）——不因為要拆一小段就被迫替尚未收斂的全書結局搶先拍板。

> ⚠️ **先確認粗階段已完成**：organize 假設 `story/00-摘要.md`（含題旨、基調）與 `story/參照/世界觀.md` 已大致成形、主線/結局/主角已拍板。**若 raw 還很亂、主線/題旨/結局未定，先用 `develop` 做粗階段**（分揀素材、彙整世界觀、攤開承重分叉），再回來 organize。發現粗層未就緒時，就把控制權交回 develop，不要在大綱裡硬塞一堆「待確認」。

## 共同約定（必讀）

資源定位、AI 角色（共創＋審稿、非門檻）、下游回饋記錄規則見 `結構定義/共同約定.md`——你提草稿、指出缺口，但所有創作決定權在作者。

## 流程

1. **載入方法與格式**：讀 `結構定義/*.schema.md`（含 `大綱.schema.md` 的全書版與局部下沉版兩節）與 `技巧知識庫/*.md`。重點：`創作流程.md`、`因果法則.md`、`主題題旨.md`、`需求四象限.md`、`結構公式.md`。
2. **讀粗階段產物與 raw/**：先讀 `story/00-摘要.md`（含題旨、基調）與 `story/參照/世界觀.md`——這是 develop 收斂出的大方向。再讀 `raw/` 補既有的大綱/幕綱片段、角色與情節細節。再讀 `story/參照/就緒儀表.md`（就緒清單、局部下沉紀錄）。**若連局部都摘要缺主線/題旨、或世界觀.md 不存在，判定粗階段未就緒 → 停下並建議先跑 `develop`。**
3. **判斷模式：全書或局部**：
   - **全書模式（預設）**：就緒儀表的「就緒清單」大致勾齊，或作者明確要整理全書 → 走 4a。
   - **局部模式**：全書結局/主線仍多處未定，但作者已在「局部下沉紀錄」指出某 arc 已清楚可下沉，或直接說「先幫我整理這一段」→ 走 4b。**不得**因為要拆一小段而強迫作者替未定的全書結局／配角搶先拍板。
   - 判斷不確定時，直接問作者：「要整理全書大綱，還是只先整理某一段？」
4. **產出／更新 story/**（依 `大綱.schema.md`）：
   - **4a 全書模式**：
     - `story/01-大綱.md`——把摘要擴寫成連續敘述的完整故事；標註選用結構公式；**結局與題旨已定**。
     - 若 `story/大綱/` 已有先前局部下沉的 arc 片段，**讀入作為素材來源**，併入連續敘述；併入後在該片段檔頭加註「已併入 `01-大綱.md`（日期）」，不刪除、轉為歷史存檔；同步更新 `story/大綱/_index.md`。
   - **4b 局部模式**：
     - 產出／更新 `story/大綱/arcNN.md` ＋ `story/大綱/_index.md`，依「局部下沉（scoped）大綱」節格式：本段全文、本段收束與鉤子、對全書結局的關係誠實標「未定」。
     - **不寫**全書結局/題旨；arc 編號需與該段未來對應的 `幕綱/arcNN.md` 共用同一識別碼。
     - 檔頭標「⚠️ 暫定，待粗層鎖定」。
   - **兩模式共通**：`story/參照/角色需求.md`、`story/參照/結構.md`（局部模式下標明「僅涵蓋 arcNN」）；摘要微調可順手更新，但主線/結局/主角的**承重改動應退回 develop**；更新既有檔時保留作者已寫定內容。
5. **拆幕交 beat-sheet**：局部模式下，提醒作者 `beat-sheet` 應讀對應的 `story/大綱/arcNN.md`（同一 arc 編號），而非 `01-大綱.md`。
6. **輸出缺口診斷報告 + 更新就緒儀表** `story/參照/就緒儀表.md`（見下）。局部模式下，「大綱」欄記至 arc 粒度（如「草擬（僅 arc01）」），並在「局部下沉紀錄」補一列（同時記大綱層與幕綱層下沉狀態）。

## 缺口診斷報告（核心行為）

依 `創作流程.md` 的「回到哪一層修」觀念，整理後回報各階段狀態——**哪些有了、哪些缺、哪些弱**，並指引下一步。範例語氣：

- 「還沒有結局 → 大綱無法定稿（結局是回頭修正前文的樞紐）」
- 「題旨待定 → 摘要的『因為 X 導致 Y』填不出，主線可能還沒收斂」
- 「幕023 的結果沒有催發下一幕（有果無因/有因無果），建議修結果或補前因」
- 「主角『需要』不明確，角色弧線測不了」

報告格式建議：分「✅ 已具備／⚠️ 偏弱／❌ 缺」三類，每條附「建議下一步」，最後點出**最該先補的一到兩項**（通常是前三步：主線／題旨／結局）。

## 邊界

- **上游**：粗階段（分揀素材、彙整世界觀、收斂主線/題旨、攤開承重分叉）是 `develop` 的事；organize 不做冷啟動，發現大方向未就緒就退回 develop。
- **下游**：不負責把大綱拆成完整幕綱（那是 `beat-sheet`），但可產出雛形。大綱/摘要大致定稿後，可用 `outline-test` 跑形式化的結構不變量測試（organize 本身的缺口診斷是共創過程中的非正式版本，outline-test 是定稿前的正式審稿）。
- **局部模式不是常態**：一旦全書結局/題旨拍板，之後一律走全書模式產出/更新 `01-大綱.md`（並回頭併入既有局部片段）；不會有「全書已有 `01-大綱.md`，還繼續新增局部 `大綱/arcNN.md`」的狀況。
- 不做動筆寫章節（子專案 B）。
- 大綱的逐步精修不另開 skill——用本 skill + 對話即可。
