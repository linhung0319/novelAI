# organize — 把 raw/ 整理成符合結構的 story/

把作者丟在某本書 `raw/` 的任意格式素材（瑣碎筆記、靈感片段、半成品大綱/幕綱/角色設定），整理成符合結構定義的 `story/` 產出，並診斷「還缺什麼、哪裡弱」。

**AI 角色 = 共創 + 審稿**：你提草稿、抓重點、指出缺口，但**所有創作決定權在作者**。不要替作者拍板創作選擇，而是攤開選項與診斷讓作者決定。

## 資源定位（重要，不得寫死絕對路徑）

- **技巧知識庫 / 結構定義**：讀取「本寫作系統 package 根目錄」下的 `技巧知識庫/` 與 `結構定義/`（與 `.claude/` 同層）。若不確定根目錄，從當前工作目錄向上尋找「同時含 `技巧知識庫/` 與 `結構定義/` 的目錄」。
- **書內容**：`raw/`、`story/`、`chapters/` 一律相對於「作者當前所在的那本書資料夾」。若作者未指明是哪本書，先確認。

## 流程

1. **載入方法與格式**：讀 `結構定義/*.schema.md`（知道目標長怎樣）與 `技巧知識庫/*.md`（知道怎麼抓重點、判因果）。重點：`創作流程.md`、`因果法則.md`、`主題題旨.md`、`需求四象限.md`、`結構公式.md`。
2. **讀 raw/**：讀當前書 `raw/` 下所有素材，辨識其中的主線、題旨線索、角色、既有的大綱/幕綱片段、零散靈感。
3. **產出／更新 story/**（依 schema）：
   - `story/00-摘要.md`——主線的一句話／50／100／200 字 + 題旨「因為 X 導致 Y」+ 高概念。
   - `story/01-大綱.md`——連續敘述的完整故事；標註選用結構公式；結局與題旨。
   - `story/參照/角色需求.md`——各主角四象限 + 預期弧線。
   - （若 raw 已有足夠的幕素材）初步的 `story/幕綱/arcNN.md` 與 `_index.md`；否則把拆幕交給 `beat-sheet`。
   - 更新既有檔時，保留作者已寫定的內容，只補缺與標示待確認處。
4. **輸出缺口診斷報告**（見下）。

## 缺口診斷報告（核心行為）

依 `創作流程.md` 的「回到哪一層修」觀念，整理後回報各階段狀態——**哪些有了、哪些缺、哪些弱**，並指引下一步。範例語氣：

- 「還沒有結局 → 大綱無法定稿（結局是回頭修正前文的樞紐）」
- 「題旨待定 → 摘要的『因為 X 導致 Y』填不出，主線可能還沒收斂」
- 「幕023 的結果沒有催發下一幕（有果無因/有因無果），建議修結果或補前因」
- 「主角『需要』不明確，角色弧線測不了」

報告格式建議：分「✅ 已具備／⚠️ 偏弱／❌ 缺」三類，每條附「建議下一步」，最後點出**最該先補的一到兩項**（通常是前三步：主線／題旨／結局）。

## 邊界

- 本 skill 不負責把大綱拆成完整幕綱（那是 `beat-sheet`），但可產出雛形。
- 不做動筆寫章節（子專案 B）。
- 摘要、大綱的逐步精修不另開 skill——用本 skill + 對話即可。
