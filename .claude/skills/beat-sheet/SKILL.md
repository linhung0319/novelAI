# beat-sheet — 幕綱共創

從大綱生成候選幕綱，把故事拆成有序、離散的「幕」。這是創作的最大難關，也是本系統把測試 shift-left 的著力點。

**AI 角色 = 共創 + 審稿**：你提幕綱草稿以降低白紙恐懼，但**所有創作決定權在作者**——排序、增刪、改劇情都由作者拍板。你負責讓每一幕結構完整、因果扣連、伏筆可追蹤。

## 共同約定（必讀）

資源定位、AI 角色（共創＋審稿、非門檻）、下游回饋記錄規則見 `結構定義/共同約定.md`。

## 依據

> 讀技法檔時先讀各檔檔頭「目錄」標**核心**的小節；標「附錄（按需取用）」的深素材只在實際觸及時再讀對應小節。

- `結構定義/大綱.schema.md`——大綱的全書版與局部下沉版格式；若讀的是 `story/大綱/arcNN.md`，其「本段收束」對應本次拆幕範圍的收尾，不是全書結局。
- `結構定義/幕綱.schema.md`——幕的固定八欄位與 `[[幕NNN]]`、`埋/收[[伏筆:x]]` 連結規範（**務必嚴格遵守**），以及選用的「本 arc 伏筆狀態」表（跨 arc／全書懸念時使用）。
- `技巧知識庫/因果法則.md`——幕的五要素、前果＝後因、具意義的結果、貴精不貴多；**§七 賭注兌現＋角色反應可信度**（升起重大威脅/秘密曝光/底線被踩時，同幕綱裡就要設計它的了結，勿留賭注孤兒；對抗降溫偏誤）。
- `技巧知識庫/結構公式.md`——選用公式與結構階段。
- `技巧知識庫/伏筆與意外.md`——伏筆埋設與由果推因。
- `技巧知識庫/需求四象限.md`——讓幕推進角色弧線。
- `技巧知識庫/主題題旨.md`——題旨公式與主題三角演繹法（對立／矛盾／極端），排幕時預留鋪陳點用。
- `技巧知識庫/戲劇張力與懸念.md`——每幕幕尾懸念結尾（七種停止點）、悶燒與兩難的節奏安排、開場鉤子；**§一 資訊差引擎**（同步／優越·扮豬吃虎／替他捏汗——設計每場張力來自誰知道什麼）、**§六(d) 開頭vs常態兩檔位**（開頭懸主角命運·高密度；常態懸這場誰死/怎麼打臉·靠長線大謎供能，章末鉤降級）。
- `技巧知識庫/世界觀與背景.md` §九「**設定帶出的操作判準**」——排「哪一幕揭什麼」的揭露節奏：先現象後定義、術語延後、一次1–2詞＋功能標籤、硬設定塞進有動機問答、疑問要有回收時點（對治「設定突然蹦出」）。
- `技巧知識庫/結局類型.md`——排收尾幕時對照六大結局類型與好結局六要素（首尾呼應／觀念改變預留點）。
- `技巧知識庫/角色塑造.md`——排幕時安排情節逐步揭露角色三層行為、消除或揭穿內外落差。
- `技巧知識庫/網文與文學取捨.md`——依 `00-摘要.md`「取向定位」欄決定爽點鋪設密度 vs 弧線鋪陳的比重；可把爽點三步／五段式與結構公式並列選用。

## 流程

1. **讀輸入**：
   - **大綱來源判斷**：若 `story/01-大綱.md` 存在，讀取（含選用結構公式、結局、題旨）；
   - 若不存在，改查目標 arc 是否有對應的 `story/大綱/arcNN.md`（同一 arc 編號）——若有，讀取作為本次拆幕的大綱來源；此時給的是「本段收束」而非全書結局，排幕時不得假裝已知全書結局。
   - 若兩者都不存在 → **停下**，向作者說明「找不到這個 arc 的大綱依據」，建議先跑 `organize`（全書模式，或只整理這一段），不要在缺乏大綱依據時自行憑空拆幕。
   - 另讀 `story/參照/角色需求.md`、`story/參照/世界觀.md`、既有 `story/幕綱/`。
2. **選用/確認結構公式**：依大綱標註或與作者確認（三幕劇／起承轉合／七問題／編劇九階段）。
3. **生成候選幕綱** → 寫入 `story/幕綱/arcNN.md`：
   - **配號**：依 `幕綱.schema.md`「幕號配號規則」——每 arc 預配 100 號段（arcNN 從幕((NN-1)×100+1) 起），幕號是穩定 ID、不代表全書順序，arc 可亂序下沉；已存在的幕號一律不重編。
   - 每幕一個 `## 幕NNN · 幕名` section，填齊固定八欄位（角色／時空／行動／衝突／結果／前因／伏筆／結構階段）。
   - **因果**：用 `[[幕NNN]]` 標前因；確保每幕「結果」能催發下一幕動機；避免「有因無果／有果無因」的孤兒與離題幕（貴精不貴多）。**升起重大賭注的幕（威脅／秘密曝光／底線被踩），要同時設計它的兌現**——中和／升級成跨章懸而未決的威脅線／給足主角按兵不動的動機，勿留「賭注孤兒」（軟退縮＋放行＋威脅後文蒸發）；主動對抗把衝突寫軟的降溫偏誤，供 `beat-test` 測試6 事後核對（見 `因果法則.md` §七）。
   - **伏筆**：由（全書模式）結局、或（局部模式）該 arc 的本段收束與鉤子，逆推需要哪些伏筆，用 `埋[[伏筆:x]]`／`收[[伏筆:x]]` 標記；局部模式下，跨出本 arc 範圍才會回收的伏筆允許「埋而未收」，於「本 arc 伏筆狀態」表註明原因（不另立登記簿）。
   - **結構階段**：每幕標明對映公式的哪個階段。
   - **主題回指**：每條主線幕盡量能回指題旨（因為 X 導致 Y）；若摘要或作者已定義主題三角（核心／對立／矛盾／極端），排幕時預留至少一處鋪陳點呼應對立／矛盾／極端情境（供角色犯錯或抉擇），供 `beat-test` 測試5 事後核對。
4. **同步維護**：
   - `story/參照/結構.md`——幕 ↔ 結構階段對應表。
   - `story/幕綱/_index.md`——各 arc 範圍、標題、公式概覽。
5. **世界觀回饋**：排幕時若發現世界觀規則不夠用、或與幕的行動矛盾，於 `story/參照/世界觀.md` 的「下游回饋記錄」（見 `結構定義/回饋記錄.schema.md`）追加一列，**不改世界觀正文**；交付候選幕綱時提示作者「已記錄 N 筆世界觀待裁決回饋」。
6. **交回作者共創**：以「候選草稿」姿態呈現，明列可重排/增刪/改的點，等作者決定；不擅自定案。

## 邊界

- 產出的是幕綱（藍圖），**不是散文章節**（動筆寫章節＝子專案 B）。
- 幕綱定案後，用 `beat-test` 跑結構測試。
