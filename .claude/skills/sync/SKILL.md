# sync — 上下游同步診斷（迭代引擎）

創作是循環的：粗層拍板或改動後，已下沉的大綱／幕綱／章節會過期（**動尾改頭**）；反過來，已寫細的段落也會對粗層收斂形成約束（**以細促粗**）。這兩個方向的原則在 `創作流程.md` 與就緒儀表 schema 裡都有陳述，本 skill 就是它們的執行者。

**定位聲明**：本 skill 是 LLM 判斷型「審稿員」，只診斷、不代改任何正文，不做自動 pass/fail——資源定位與其餘共通規則見 `結構定義/共同約定.md`（必讀）。

**兩層偵測互補**（見 `共同約定.md` 三）：
- **機械層 `tools/derived_sync`**：`cd tools/derived_sync && uv run derived-sync check --book <書>` 比對每個 `.ai.md` 的 `generated-from` 與源檔正規化 hash——精準抓「作者手動改了某個**源檔**（`設定/**/<實體>.md`、`00-摘要.md`、章節正文 `chNNNN.md`），對應衍生檔沒重生」這一類過期。零 LLM、可覆算。這是本 skill 順向清查的**起手式**，取代過去「靠作者口述改了什麼」的第一步。
- **語意層（本 skill 的 LLM 判斷）**：深層過期（結局→伏筆回收、題旨→立場鋪陳幕、世界觀規則→幕可行性、幕綱→章節 Scene）——這類 hash 抓不到（改的可能是作者利器·直改的大綱/幕綱，或跨產物的語意依賴），仍靠當場語意比對。
- 仍**不建** digest cache／持久 state／跨集狀態機（子專案 C）——`derived-sync` 只做無狀態的即時 hash 比對。

## 兩種模式

| 模式 | 觸發情境 | 方向 |
|------|----------|------|
| **順向清查** | 上游改了（結局拍板、題旨修改、世界觀規則變更、幕綱在成章後被改） | 上 → 下：找 stale |
| **反向盤點** | 作者要回頭收斂粗層（結局終幕、配角、中後段）之前 | 下 → 上：列已承諾事實 |

作者沒說清楚時，先問一句：「這次是上游改了要找過期下游，還是要回頭改粗層、先盤點已寫死的東西？」兩種可以同一次都做。

## 依賴鏈（判斷 stale 的依據）

```
設定層（設定/{世界觀,角色,風格}/<實體>.md → <實體>.ai.md）＋ 00-摘要.md → 00-摘要.ai.md（粗層）
   → 01-大綱.md ／ 大綱/arcNN.md（大綱層，作者利器·直改）
      → 幕綱/arcNN.md（幕綱層，作者利器·直改）
         → chapters/chNNNN.md → chNNNN.ai.md（正文層，經 chapters/_index.ai.md 的「對應幕」欄對映）
```

- **源→衍生對（同實體 `.md`→`.ai.md`）的過期由 `derived-sync` 機械偵測**（源動、衍生沒重生）；
- **跨產物層的語意過期**（設定變→大綱/幕綱/正文的可行性）由本 skill 語意比對。

「暫定，待粗層鎖定」標記的下沉品，天生是粗層變更的重點重驗對象。

## 流程

1. **機械層起手（順向必跑）**：`cd tools/derived_sync && uv run derived-sync check --book <書>`，列出所有 `stale`／`unstamped`／`orphan` 的 `.ai.md`——這就是「哪些源檔被手動改過、衍生沒跟上」的確定性清單。把它當順向清查的第一批 stale（建議行動＝「交對應 skill 重生並 stamp」：世界觀→`worldbuild`、角色→`character`、風格/摘要→`develop`/`organize`、章節→`write`/`revise`）。
2. **確認改動範圍**：結合 check 結果與作者口述，判斷這次改了哪一層的哪個檔、改了什麼內容（git 歷史可輔助，不依賴）。反向盤點模式則問明「打算收斂粗層的哪一塊」。
3. **讀輸入**：`story/參照/就緒儀表.md`（局部下沉紀錄）、`chapters/_index.ai.md`（若存在），再依依賴鏈讀受影響範圍的產物。
4. **順向清查·語意層**（上游改了，機械層抓不到的深層依賴）：
   - 逐層語意比對：改動的內容，被哪些下游產物**具體依賴**（結局→伏筆回收設計、題旨→立場鋪陳幕、世界觀規則→幕的行動可行性、幕綱→章節的 Scene 內容）。
   - 列出 stale 清單：不是「下游全部作廢」，而是逐項指出「哪個產物的哪一點，因上游哪個改動而過期」；語意上不受影響的明說「不受影響」。
   - 每項附建議行動：先重生機械層報的 stale `.ai.md`（交對應 skill），再重跑對應 test（outline-test／beat-test／write-test，scoped 到受影響範圍）、用 `organize` 全書模式併入、用 `revise` 修訂章節、或作者直接改幕綱。
5. **反向盤點**（回頭收斂粗層前）：
   - 從已下沉的大綱／幕綱／章節，萃取「**已承諾事實清單**」：已寫死的事件與順序、已埋的伏筆（含刻意留白項）、角色已展現的行為與性格、已建立的世界觀細節與基調證據。
   - 每條註明出處（幕號／章號），標記哪些是「粗層收斂時不能翻案、翻案就得回頭重寫」的硬承諾。
6. **彙報待裁決回饋**：掃各衍生檔（`設定/世界觀/*.ai.md`、`00-摘要.ai.md`、`設定/角色/*.ai.md`、`風格.ai.md`）的「## 待裁決回饋」的「待裁決」列，列出筆數與摘要——回頭改粗層正是消化這些回饋（提請作者改源後重生）的時機。
7. **更新就緒儀表**：在受影響產物軸的備註欄加註「⚠️ 上游已變更（日期），待重驗」或更新局部下沉紀錄；不改動任何正文。

## 輸出格式

- **順向**：`產物（檔案/幕號/章號） ｜ 過期原因（依賴了上游的什麼） ｜ 嚴重度 ｜ 建議行動`，末尾按行動歸組（該重跑哪些 test、該併入/修訂哪些檔）。
- **反向**：已承諾事實清單（`事實 ｜ 出處 ｜ 硬承諾？`），供作者收斂粗層時對照。
- 待裁決回饋筆數與摘要。
- **明確聲明**：以上為同步診斷，所有處置由作者決定；本 skill 未修改任何產物正文。

## 邊界

- 不代改任何上下游正文（含幕綱）；建議行動指向對應 skill 或作者手改。
- 機械層僅用 `derived-sync` 的**無狀態 hash 比對**（源→衍生對）；不建持久 digest cache／state／跨集狀態機（子專案 C 範圍）。源→衍生以外的過期（作者利器·直改的大綱/幕綱、跨產物語意依賴、幕綱於成章後被改）仍靠作者告知＋當場語意比對。
- 不替作者決定「要不要接受過期」——作者可以決定某個 stale 就先擱著（標記在就緒儀表即可）。
