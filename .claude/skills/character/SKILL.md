# character — 按需獨立設計單一角色

系統其他 skill 只在別的動作裡「順帶」產角色：`develop` 在粗階段冷啟動時**打包**配角、`expand` 在偵測到某段劇情缺衝突時**副產**一個對手。本 skill 補上缺的一級操作：**任何時候，作者就是想獨立設計一個角色**（新反派、新配角、或替既有角色補上需求弧線），不必為此跑整套 develop 或硬造一個劇情缺口。

定位是「中間檔位」的輕量操作——單一產物、單點寫入，不套用大 skill 的全套儀式。

**AI 角色 = 共創 + 審稿**：你依方法提角色草稿以降低白紙恐懼，但**所有創作決定權在作者**——採不採某個角色、四象限怎麼填、要不要進正史，一律作者拍板。你只提案，不代決。

## 共同約定（必讀）

資源定位、AI 角色（共創＋審稿、非門檻）、下游回饋記錄規則見 `結構定義/共同約定.md`。

## 依據

- `結構定義/角色需求.schema.md`——本 skill 的輸出格式（每個角色一個 `##` section：期盼／想要／落差／需要四象限＋預期弧線＋馬斯洛層次＋選填兩難；反派可給「正當需求」）。
- `技巧知識庫/需求四象限.md`——設計需求弧線與兩難的核心方法：期盼＝動力／想要＝誤導／落差＝阻礙／需要＝圓滿，選正／平／負弧線型，「需要」須拖到最後才滿足，「反派也有正當需求」。
- `技巧知識庫/角色塑造.md`——把角色設計得立體可信：三面向、三層行為反差、由外而內三階段、塑造六法（供描述角色時避免只給扁平標籤）。
- `技巧知識庫/世界觀與背景.md` ＋ `story/參照/世界觀.md`——判斷角色的出身／能力／動機是否違反世界觀核心規則、是否自洽。
- `技巧知識庫/網文與文學取捨.md`——依 `00-摘要.md`「取向定位」欄拿捏：偏網文則角色可帶更強的爽點功能（金手指持有者、打臉對象…），偏文學則倚重需求弧線與內在矛盾的縱深。

## 流程

### Phase 1 — 提案（唯讀，不改任何檔）

1. **確認書與需求**：作者未指明哪本書、或要設計什麼樣的角色（定位／功能／與誰對立）時先確認（見 `共同約定.md` 一）。
2. **載入方法與格式**：讀上列「依據」的 `技巧知識庫/*.md`（先讀各檔檔頭「目錄」標**核心**的小節；附錄按需再讀）與 `結構定義/角色需求.schema.md`。
3. **讀 context**：
   - `story/00-摘要.md`——主線、題旨、取向定位、基調（角色需服務主線、其弧線需能呼應題旨）。
   - `story/參照/世界觀.md`——核心規則（角色的能力／出身／動機不得違反）。
   - `story/參照/角色需求.md`（若存在）——既有角色與弧線，避免重複、並讓新角色的需求與既定角色（尤其主角）**形成衝突**。
   - **先掃既有角色（含只有一行 stub／bullet 的配角，如 `develop` 打包的配角清單）**：若作者要設計的角色**已存在**（完整 section 或一行 stub）→ 本次是「**替既有角色補上需求弧線**」（見 description）＝**就地深化**該既有條目，**不另立平行的重複條目**；確認檔中查無此角色，才是全新角色。
   - **前置檢查**：若連 `00-摘要.md` 主線／題旨或 `世界觀.md` 都不存在 → 判定粗階段未就緒，**停下**建議先跑 `develop`，不要在真空中設計角色。
4. **提 1–N 個角色草稿**：每個草稿給
   - 一句話定位（在故事裡幹什麼、與誰對立）；
   - 需求四象限（期盼／想要／落差／需要；反派同樣給「正當需求」，與主角需求同樣正當會讓矛盾更強）；
   - **若這個角色的真動機是後段才向讀者揭示的**（真凶、臥底、偽裝成敵人的守護者）：逐象限分【表面】/【🧊 水下】兩層，依 `角色需求.schema.md` 的分層格式與 `共同約定.md` 四的指標語法。四象限填的仍是角色的**真實**（否則 `beat-test` 的弧線檢查失去意義），分層只是標明哪一層讀者現在還不能知道。揭示點取自幕綱既有的 `收[[伏筆:x]]`；查不到就標「揭示點待落幕」，別自己造一個伏筆名。
   - 預期弧線（盲目→挫折→醒悟→滿足，或平／負弧線）＋馬斯洛層次；
   - 與世界觀自洽的檢查結果、與既有角色的衝突關係；
   - 選填：一條足以立體化的「三層行為反差」或塑造六法示例。
   - 承重處（要不要進正史、選哪個版本）明確標「待你拍板」。**到此為止，不寫任何檔。**

### Phase 2 — 拍板後寫入（作者選定後才執行）

5. **寫入 `參照/角色需求.md`**：依作者選定，按 `角色需求.schema.md` 寫一個 `##` section（四象限＋預期弧線＋馬斯洛層次＋選填兩難），標「⚠️ 暫定」。若該檔尚不存在則依 schema 新建（含檔頭與「下游回饋記錄」空區塊）。
   - **對象角色已以一行 stub／bullet 存在時（就地深化，不重複）**：把該 stub 升格為完整 `##` section，並**移除或劃除（`~~…~~`）原 stub 那一行**，使全檔同一角色只留一份——**不得新增 section 又留著平行的 stub bullet**（否則同一角色兩份，`beat-test` 角色弧線檢查會對到殘影）。
   - **「保留既有 section 不動」只適用作者已寫定的*其他*角色 section**，不適用正在深化的目標 stub。
   - **後段 reveal 型角色**：分層直接用 `角色需求.schema.md` 的【表面】/【🧊 水下｜揭示於 收[[伏筆:x]]】格式寫進象限本身，**不另外加自創的洩題警告區塊**——標記就是給下游的訊號，schema 已定義它的意思，額外的散文提醒只會讓每次產出長得不一樣。
   - 若角色設計**引入新的世界觀設定**（新種族／新能力來源／新組織）→ 依 `共同約定.md` 三，在 `世界觀.md` 的「下游回饋記錄」追加一列（不改世界觀正文），交付時提示「已記錄 N 筆世界觀待裁決回饋」，交 `worldbuild` 消化。
6. **更新就緒儀表＋指路**：若 `story/參照/就緒儀表.md` 有角色相關列，更新其成熟度；以「已追加、仍暫定」姿態呈現，提醒後續拆幕（`beat-sheet`）或寫正文（`write`）時此角色即可被引用。

## 邊界

- **與 `develop` 分工**：develop 是粗階段冷啟動、一次打包世界觀＋主線＋一組配角；character 是粗階段之後任何時候、針對**單一**角色的定點設計。粗層未就緒就退回 develop。
- **與 `expand` 分工**：expand 因某段劇情缺衝突而**副產**角色（角色是劇情缺口的解法）；character 是**主動獨立**設計角色（角色本身就是目的）。若作者其實是要補某段劇情，導去 expand。
- **不拆幕**（交 `beat-sheet`）、**不寫正文**（交 `write`）、**不做世界觀補厚／消矛盾**（交 `worldbuild`）。
- **不自動 pass/fail**：草稿是提案，承重抉擇（採哪個角色、進不進正史、四象限定稿）一律作者拍板；產出皆為可編輯、非受保護的草稿。
