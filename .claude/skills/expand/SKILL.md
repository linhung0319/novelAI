# expand — 把單薄的 arc 橫向長成完整小故事

系統其他 skill 都是**垂直**推進（摘要→大綱→幕綱→正文）。expand 是**橫向**能力：當某個 arc（弧線）只有一個橋段、劇情單薄時，不急著往下拆幕，而是先把這一段**就地長厚**——偵測它缺哪個戲劇要素，由世界觀／摘要／角色需求驅動，提案補法，讓它成為一個結構完整、半獨立的小故事（有本段收束、可留鉤子）。

> 用哈利波特類比：一集 = 全書大綱；集內「查誰要偷石頭」「穿越守關」這種可獨立成篇的段落 = 一個 arc；expand 處理的就是「這個 arc 現在只有一句『主角進了古堡』，怎麼把它長成完整一段」。arc 拆下去才是幕綱（交 beat-sheet）。

**AI 角色 = 共創 + 審稿**：你偵測缺口、提候選補法以降低白紙恐懼，但**所有創作決定權在作者**——採不採某個衝突、要不要新增角色，一律作者拍板。你只提案，不代決。

## 共同約定（必讀）

資源定位、AI 角色（共創＋審稿、非門檻）、下游回饋記錄規則見 `結構定義/共同約定.md`——你提草稿、指出缺口，但所有創作決定權在作者。

## 依據

- `結構定義/大綱.schema.md`——**局部下沉（scoped）大綱**節的格式（本段全文／本段收束與鉤子／對全書結局關係誠實標未定）；expand 的寫入一律走這個局部格式。
- `結構定義/角色需求.schema.md`——若提案需新增角色，新 `##` section 的四象限＋弧線＋馬斯洛層次格式。
- `技巧知識庫/結構公式.md`——選用公式與其必要階段（判斷缺哪個階段：三幕缺「衝突」、七問題缺「意外／轉彎」…）。
- `技巧知識庫/因果法則.md`——幕的五要素、前果＝後因、有因無果／有果無因／未連主線。
- `技巧知識庫/場景與情節推進.md`——場景 Scene 的「目標→衝突→災難」與 Yes,but／No,and；判斷「困難與解法」是否具備。
- `技巧知識庫/需求四象限.md`——主角「期盼／想要／落差／需要」是否被觸碰、有無弧線塌陷；「反派也有正當需求」（推演新增角色的依據）。
- `技巧知識庫/伏筆與意外.md`——由果推因，標記可供下游 beat-sheet 埋設的伏筆點。
- `技巧知識庫/世界觀與背景.md`——判斷提案是否違反 `世界觀.md` 核心規則、是否自洽。
- `技巧知識庫/戲劇張力與懸念.md`——提案子情節時參考懸念引擎與兩難設計，避免只補「有事發生」卻無張力。
- `技巧知識庫/網文與文學取捨.md`——依 `00-摘要.md`「取向定位」欄，決定提案偏爽點鋪設或弧線鋪陳。

## 流程

### Phase 1 — 診斷＋提案（唯讀，不改任何檔）

1. **確認書與 arc**：作者未指明哪本書／哪個 arc 時先確認（見 `共同約定.md` 一）。
2. **載入方法與格式**：讀上列「依據」的 `技巧知識庫/*.md` 與 `結構定義/*.schema.md`。
3. **讀輸入**：
   - 目標段落：`story/大綱/arcNN.md`（若存在）；若作者只是貼了一個橋段、還沒有 arc 檔，就以貼的內容為輸入，並在報告開頭註明「本 arc 尚無大綱檔」。
   - 上游參照：`story/00-摘要.md`（主線、題旨、取向定位、基調）、`story/參照/世界觀.md`（核心規則）、`story/參照/角色需求.md`（既定角色與弧線）、`story/參照/結構.md`（選用公式與已對應階段，若有）。
   - **前置檢查**：若連 `00-摘要.md` 的主線／題旨都缺，或 `世界觀.md` 不存在 → 判定粗階段未就緒，**停下**建議先跑 `develop`，不要在空中生劇情。
4. **跑缺口偵測引擎**（下節五類）。
5. **輸出報告**（格式見下）：每個缺口附 2–3 個候選提案；提案標動力／阻礙、對映結構階段、與世界觀是否相容、以及是否需新增角色（附四象限草稿）。承重處明確標「待你拍板」。**到此為止，不寫任何檔。**

### Phase 2 — 拍板後寫入（作者選定後才執行）

6. **併入 arcNN.md**：依作者選定，把內容寫進 `story/大綱/arcNN.md` 的連續敘述，沿用 `大綱.schema.md` 局部下沉格式——保留檔頭 `⚠️ 暫定，待粗層鎖定`、維持「本段收束與鉤子」誠實聲明、對全書結局關係標「未定」。arc 編號與未來 `幕綱/arcNN.md` 共用同一識別碼。更新既有內容時保留作者已寫定的部分。若原本無 arc 檔，依格式新建並更新 `story/大綱/_index.md`。
7. **新增角色 → 同步 `參照/角色需求.md`**：若採用需新增角色的提案，依 `角色需求.schema.md` 追加一個 `##` section（期盼／想要／落差／需要四象限＋預期弧線＋馬斯洛層次的草稿），標「⚠️ 暫定」。
8. **新設定 → 走回饋機制**：若提案引入新的世界觀設定，依 `共同約定.md` 三，在 `世界觀.md` 的「下游回饋記錄」追加一列（見 `回饋記錄.schema.md`），**不改世界觀正文**；交付時提示作者「已記錄 N 筆世界觀待裁決回饋」。
9. **更新就緒儀表**：於 `story/參照/就緒儀表.md` 更新該 arc 的成熟度（局部下沉紀錄），順手偵測上游待裁決列並備註。
10. **交回作者＋指路下游**：以「候選已併入、仍暫定」姿態呈現；提醒後續可對這段跑 `outline-test`（scoped 模式）審稿，再交 `beat-sheet` 拆幕。

## 缺口偵測引擎（核心行為）

對照「故事單元完整性」的五類清單，逐項判**有／弱／缺**，每個缺口都要能對映一個具體依據：

| # | 檢查 | 判準 | 依據 |
|---|------|------|------|
| 1 | 結構階段齊全 | 對映選用公式（見 `結構.md`／摘要），缺／重／錯序哪個必要階段（三幕缺「衝突」或「解決」、七問題缺「意外／轉彎」、九階段缺「低潮」…） | `結構公式.md` |
| 2 | 因果連續 | 前果＝後因是否成立；有因無果（結果催發不了下文）／有果無因（憑空出現）／未連主線 | `因果法則.md` |
| 3 | 場景完整 | 有無「目標→衝突→災難」；「困難」與「解法」是否具備（災難是 Yes,but／No,and 而非順風順水） | `場景與情節推進.md` |
| 4 | 角色弧線 | 主角的「期盼／想要／落差／需要」在這段是否被觸碰；有無弧線塌陷（需要太早滿足）；轉變是否掙來、非機器神 | `需求四象限.md` |
| 5 | 世界觀自洽 | 段落與提案是否違反 `世界觀.md` 核心規則；新設定是否與既定衝突 | `世界觀與背景.md` + `世界觀.md` |
| 附 | 伏筆機會 | 標記可供下游 beat-sheet 埋設的伏筆點（由果推因），不強制 | `伏筆與意外.md` |

### 提案生成規則

- **缺衝突** → 套 `衝突 = 動力 + 阻礙`；若阻礙需要一個人格化來源，就推演「新增一個帶著正當需求 X 的角色／對手」（`需求四象限.md`「反派也有正當需求」）。這正是「缺衝突 → 建議衝突 → 推演出需要新增某性格角色」的落地。
- 每個提案都要能**對映一個結構階段**、且與 `00-摘要.md` 主線與 `角色需求.md` 既定弧線相容，並通過第 5 類世界觀自洽檢查。
- 提案一律以「候選 A／B／C」並陳，講清各自的取捨與後果，**不替作者選**。
- 依取向定位調比重：偏網文則提案偏爽點節奏，偏文學則偏弧線與主題鋪陳（`網文與文學取捨.md`）。

## 報告格式

Phase 1 報告直接輸出於對話（不新增 schema、不寫檔）：

```
## 缺口診斷（arcNN）
（每項標 ✅ 已具備／⚠️ 偏弱／❌ 缺，寫明：要素、現況、為何是缺口、對映第幾類檢查）
- ❌ 衝突（#1/#3）：目前只有「主角抵達古堡」，無阻力、無災難
- ⚠️ 角色弧線（#4）：主角的「需要」整段未被觸碰
- ❌ 因果（#2）：橋段無結果，催發不了下一段

## 提案（每項待你拍板）
缺口：衝突
- 候選 A1：管家刻意隱瞞古堡真相（動力＝查真相／阻礙＝管家；對映「觸發→衝突」；不違反世界觀）
- 候選 A2：另一名訪客也在找同一件東西
    ↳ 若採 A2，建議新增角色「對手訪客」——附四象限草稿（期盼／想要／落差／需要）

## 伏筆機會（供 beat-sheet）
- 可在此埋「管家的鑰匙」，於本段收束回收 / 或留作跨 arc 鉤子
```

## 邊界

- **上游**：世界觀冷啟動、主線／題旨／結局收斂、承重分叉拍板是 `develop` 的事；發現粗層未就緒就退回 develop，不憑空生劇情。
- **與 organize 的分工**：`organize` = 把**已收斂**的素材整理成結構；`expand` = 把一個**單薄**的 arc 生成式長厚。兩者都在大綱/arc 層。若作者其實是要整理一堆已想清楚的素材，導去 organize。
- **下游**：不拆幕（交 `beat-sheet`）、不寫正文（交 `write`）。擴充後的 arc 可用 `outline-test`（scoped 模式）審稿，再交 beat-sheet。
- **不自動 pass/fail**：缺口診斷是「可疑點報告」，嚴重度僅供排序；承重抉擇（新增角色、採哪個衝突）一律作者拍板。
