# beat-test — 幕綱測試（審稿員）

對幕綱跑 5 項結構不變量測試，把可疑點攤開讓作者判斷。這是本系統的核心價值：在**幕綱層**抓問題，等同在設計評審抓 bug，而非等成品散文才發現。

**定位聲明**：小說測試是模糊、需判斷的。本 skill 是 **LLM 判斷型的「審稿員」，不做自動綠燈/紅燈門檻**。輸出可疑點報告，**所有處置由作者決定**。

## 資源定位（重要，不得寫死絕對路徑）

- **技巧知識庫 / 結構定義**：讀「本寫作系統 package 根目錄」下的 `技巧知識庫/` 與 `結構定義/`（與 `.claude/` 同層）；不確定時從當前工作目錄向上找「同時含這兩個資料夾的目錄」。
- **書內容**：`story/幕綱/`、`story/參照/` 相對於作者當前所在的那本書資料夾。可由作者指定只測某個 arc 範圍。

## 5 項測試

每項標明「機械可抽取部分」（確定性檢查連結標記）與「LLM 判語意部分」，及基準知識庫。

| # | 測試 | 檢查什麼 | 機械 vs LLM | 基準 |
|---|------|----------|-------------|------|
| 1 | **因果連續性** | 每幕「結果」是否催發下一幕動機；有因無果／有果無因的孤兒；未連主線的幕 | 抽取 `[[幕NNN]]` 前因鏈（機械）＋ 判結果是否「有意義地」催發下一幕（LLM） | `因果法則.md` |
| 2 | **伏筆閉合** | 每個 `埋[[伏筆:x]]` 是否有後續 `收[[伏筆:x]]`；結尾有無懸空線 | 掃描並配對 `埋`/`收` 標記（機械，確定性）＋ 判回收品質（LLM） | `伏筆與意外.md` |
| 3 | **結構完整性** | 對照選用公式，缺哪個必要階段／重複／錯序 | LLM 對照 `story/參照/結構.md` 與各幕「結構階段」欄位 | `結構公式.md` |
| 4 | **角色需求弧線** | 「需要」是否拖到最後才滿足；轉變是否掙來（非機器神） | LLM 對照 `story/參照/角色需求.md` | `需求四象限.md` |
| 5 | **主題與 POV** | 立場有無中途詭變（對照題旨「因為 X 導致 Y」）；第一人稱知識洩漏 | LLM 對照 `story/00-摘要.md` 題旨 | `主題題旨.md` |

## 執行步驟

1. 讀幕綱（指定 arc 或全書）與 `story/參照/`、`story/00-摘要.md`。
2. **先做機械抽取**：列出所有 `[[幕NNN]]` 前因鏈、所有 `埋/收[[伏筆:x]]` 配對狀態——這部分是確定性的，先報硬事實（如「伏筆:封脈之術 有埋無收」）。
3. **再做 LLM 語意判斷**：對照各基準知識庫，逐項找可疑點。
4. 輸出報告（見下）。

## 輸出格式

每個可疑點一列：

```
幕 id ｜ 測試類型 ｜ 嚴重度 ｜ 說明 ｜ 建議修正
```

- **嚴重度**：高／中／低（僅供作者排序參考，非門檻）。
- 範例：
  - `幕023 ｜ 伏筆閉合 ｜ 高 ｜ 埋[[伏筆:封脈之術]] 全書未見收 ｜ 於結局前補一幕回收，或刪除此埋設`
  - `幕017 ｜ 因果連續性 ｜ 中 ｜ 結果「主角睡了一覺」未催發下一幕，疑似孤兒 ｜ 改為有意義的結果，或併入鄰幕`
- 結尾給一段**總覽**（各測試各幾項可疑點），並**明確聲明：以上為審稿建議，處置由作者決定，本測試不下 pass/fail**。

## 邊界

- 不自動修改幕綱（除非作者要求後另行進行）。
- MVP 在主 session 執行；若日後全書測試 context 過大，再考慮改為獨立 subagent（非本階段範圍）。
