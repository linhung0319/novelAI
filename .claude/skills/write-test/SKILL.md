# write-test — 正文測試（審稿員）

對已寫成的章節正文跑 7 項檢查，把可疑點攤開讓作者判斷。這是 `write` 的審稿搭檔：`write` 負責「從無到有」把幕綱展開成散文，`write-test` 負責回頭核對這份散文有沒有漏幕、有沒有 POV 跳脫、有沒有該展示卻用了講述、基調有沒有跑掉、節奏與取向定位合不合、分章邊界合不合理——比照 `beat-test`（審幕綱）／`outline-test`（審大綱），補齊子專案 B 的檢測層。

**定位聲明**：正文測試是模糊、需判斷的。本 skill 是 LLM 判斷型「審稿員」，不做自動 pass/fail，處置由作者決定——資源定位與其餘共通規則見 `結構定義/共同約定.md`（必讀）。

## 測試對象

`chapters/`、`story/幕綱/`、`story/參照/`。作者可指定測試範圍：
  - **指定 arc**：反查 `chapters/_index.md`「所屬 arc」欄，展開為該 arc 已寫成的 chNN 範圍。
  - **直接指定 chNN 範圍**：如「測 ch01–ch02」或單一章「測 ch03」。
  - 若指定範圍在 `chapters/_index.md` 找不到對應已寫章節 → 回報「這個範圍還沒有正文」，建議先跑 `write`；不得對不存在的章節杜撰檢查結果。

## 7 項測試

每項標明「機械可抽取部分」（確定性檢查）與「LLM 判語意部分」，及基準知識庫。對應 `結構定義/章節.schema.md`「檢查點（供未來 write-test 參照）」一節列出的項目。

| # | 測試 | 檢查什麼 | 機械 vs LLM | 基準 |
|---|------|----------|-------------|------|
| 1 | **幕涵蓋完整性** | 章節「對應幕」宣告範圍內每一幕，正文是否都能找到至少一個 `<!-- 幕NNN -->` 錨點；有無錨點宣告了範圍外的幕號 | 掃描正文所有 `<!-- 幕NNN -->` 錨點，與該章「對應幕」欄宣告的幕號集合做集合差（機械，確定性）＋判命中的錨點段落是否真的在渲染該幕的 Scene，而非掛名湊數（LLM） | `章節.schema.md`「幕錨點規範」 |
| 2 | **POV 一致性** | 全程是否維持單一 POV 角色、無 head-hopping；POV 角色是否敘述了自己不該知道的事（含以「像是」「彷彿」等自由間接體語氣，代讀者/敘事者詮釋他人動機或內心，而非單純描述可觀察的外顯行為——這是比直接寫他人心裡想什麼更隱蔽的越界形式）；Deep POV 結構性技巧是否落實（感官/動作由 POV 角色接收起頭、內心獨白直接/間接交替得宜、少用角色自呼其名、用詞是否貼合角色背景） | 檢查每章 header「POV」欄非空（機械）＋通讀全文判知識邊界洩漏與 Deep POV 結構性技巧（LLM） | `視角與敘事人稱.md` 一～五節 |
| 3 | **展示而非講述＋對白** | 關鍵時刻（衝突高點、結果轉折、情緒反應）是否以展示而非講述呈現，過濾詞是否已清除；有無情緒標籤化、一次性資訊傾倒等講述陷阱；對白有無七大壞習慣（重複已知、談論常識、動作過多/背景空白、形容氾濫、話者類同——遮名可辨），每句台詞是否至少幹一件活 | 掃描過濾詞候選（「他/她」+「看見／看到／聽見／聽到／感覺到／感受到／注意到／發現／意識到／發覺／聞到」等組合），列成候選清單（機械，非直接判定）＋判候選是否發生在關鍵時刻（真問題）或可接受的過場講述（非問題），並找出規則沒掃到的情緒標籤/資訊傾倒/副詞化對白標籤，及對白壞習慣（LLM） | `展示而非講述與五感描寫.md`、`對白.md` |
| 4 | **動作場面專項** | 若範圍內有幕屬戰鬥/動作類型，是否套用動作場面寫作的五面向（細節/視角/議論/意境/人物）；是否誤將「詞彙鋪張」「鋪墊框架」這兩條玄幻/仙俠專屬路徑套到非該類型的一般動作幕 | 無機械抽取；先由 LLM 判斷範圍內各幕是否屬戰鬥/動作類型（依幕綱「行動」「衝突」欄與正文內容），若無則本項報告「不適用（本範圍無動作場面幕）」；若有則逐幕核對五面向與專屬路徑誤用（LLM） | `動作場面寫作.md` |
| 5 | **基調一致性** | 本章筆調是否與 `00-摘要.md` 基調一致，有無不自覺轉調 | 檢查每章 header「基調參照」欄非空（機械）＋讀 `00-摘要.md`「基調」欄，對照本章實際筆調判斷有無落差（LLM） | `氛圍與筆調.md`、`00-摘要.md`「基調」 |
| 6 | **分章合理性** | 本章起訖點是否落在自然的場景/時間/結構階段邊界，而非任意切斷一個未完成的 Scene | 無機械抽取；對照 `story/參照/結構.md` 各幕「結構階段」欄與 `場景與情節推進.md` 的 Scene/Sequel 完整度判斷邊界是否自然（LLM） | `章節.schema.md`、`場景與情節推進.md`、`story/參照/結構.md` |
| 7 | **節奏與定位一致** | 本章實際的節奏速度／揭露反轉速度／資訊密度／連載鉤子強度，是否與 `00-摘要.md`「取向定位」欄宣告的網文↔文學傾向相符（如宣告「文學向為底」卻通篇快爽套路、或宣告「爽文」卻節奏拖沓、爽點久久不來） | 檢查每章 header/摘要「取向定位」欄非空（機械）＋讀「取向定位」欄，對照本章實際節奏與爽點鋪設判斷有無落差（LLM）；未宣告定位時本項報告「不適用（摘要未標取向定位）」 | `網文與文學取捨.md`、`00-摘要.md`「取向定位」 |

## 執行步驟

1. **讀輸入**：
   - 依作者指定的 arc（反查 `chapters/_index.md`「所屬 arc」欄展開為 chNN 範圍）或直接指定的 chNN 範圍，讀對應 `chapters/chNN.md`。
   - 讀取範圍涵蓋之幕所在的 `story/幕綱/arcNN.md`（若章節「對應幕」有 `[[幕NNN]]` 引用到範圍外/跨 arc 的編號，一併讀取解析，比照 `beat-test` 的跨 arc 讀取規則）。
   - 讀 `story/00-摘要.md`（基調／題旨）、`story/參照/結構.md`（分章合理性判準）、`story/參照/角色需求.md`（若需要判斷角色內心戲/動機呈現是否越界）。
   - 讀 `chapters/_index.md` 確認指定範圍確有對應已寫章節；找不到 → **停下**，說明「這個範圍還沒有正文」，建議先跑 `write`。
2. **先做機械抽取**：
   - 逐章列出「對應幕」宣告範圍 vs 正文 `<!-- 幕NNN -->` 錨點集合的差集——**宣告涵蓋卻無錨點**是最高優先級的機械事實（漏幕，不需 LLM 判斷），**有錨點卻不在宣告範圍**次之（越界或宣告錯誤）。
   - 逐章掃描過濾詞候選句，列成清單（附上下文，供步驟3複核，非直接判定為問題）。
   - 檢查每章 header「POV」「基調參照」欄是否非空。
3. **再做 LLM 語意判斷**：對照各基準知識庫，逐項找可疑點，包含複核步驟2的機械抽取候選（過濾詞候選是否真的發生在關鍵時刻、幕涵蓋的錨點段落是否掛名湊數）。
4. **輸出報告**（見下）。**不更新 `chapters/_index.md`**——章節測試結果的時效性太短（作者一改動內容，先前的測試結論就過期），比照系統其餘測試 skill「先寫報告」的作法，但刻意不寫回任何追蹤欄位；是否要有自動化的測試狀態追蹤機制，留待系統穩定後再議。

## 輸出格式

每個可疑點一列：

```
章 id ｜ 測試類型 ｜ 嚴重度 ｜ 說明 ｜ 建議修正 ｜ 建議回寫位置
```

- **嚴重度**：高／中／低（僅供作者排序參考，非門檻）。
- **建議回寫位置**：通常填「—」（問題就在章節正文本身，作者直接改文字即可）；若判定問題根源其實在幕綱本身（如分章邊界追溯到幕綱結構階段誤置、或幕涵蓋缺口其實是幕綱本身漏了一幕）→ 仍填「—」，但「建議修正」註明「建議作者直接修改幕綱或重跑 beat-test」（幕綱無下游回饋記錄區塊，不可代寫，見 `章節.schema.md`／`write` skill 步驟10邏輯）；若追溯到有下游回饋機制的 `世界觀.md`／`00-摘要.md`／`角色需求.md` → 填該檔案名，並依 `結構定義/回饋記錄.schema.md` 格式在該檔案末尾追加一列「下游回饋記錄」。
- 範例：
  - `ch02 ｜ 展示而非講述 ｜ 中 ｜ 幕006「她看見老婦人的視線第一個移開」用了過濾詞「她看見」，發生在情緒轉折段落（大人集體迴避收養的關鍵時刻），削弱 Deep POV 貼近感 ｜ 刪去「她看見」，直接寫「老婦人的視線第一個移開」 ｜ —`
  - `ch02 ｜ POV 一致性 ｜ 中 ｜ 幕006「像是在確認有沒有人在看她表演得夠不夠好」以自由間接體代小女巫（POV角色）詮釋老婦人的表演動機——這超出一個孩子單憑外顯行為能做的判斷，接近敘事者代為戳破偽善，而非 POV 角色自己觀察到的具體證據 ｜ 改為只寫小女巫實際觀察到的外顯行為（如老婦人視線的移動軌跡本身），把「是不是在表演」的詮釋空間留給讀者 ｜ —`
  - `ch01 ｜ 幕涵蓋完整性 ｜ 高 ｜ 對應幕宣告涵蓋幕001–幕004，正文中找不到 <!-- 幕003 --> 錨點 ｜ 補上幕003的 Scene 正文與錨點，或確認是否併入了幕002/幕004段落，若是則修正錨點標記 ｜ —`
- 結尾給一段**總覽**（各測試各幾項可疑點；測試4若無動作幕則註明「不適用（本範圍無動作場面幕）」，測試7若摘要未標取向定位則註明「不適用（摘要未標取向定位）」），並**明確聲明：以上為審稿建議，處置由作者決定，本測試不下 pass/fail，且未寫入 `chapters/_index.md`**。

## 邊界

- 不自動修改章節正文（除非作者要求後另行進行）。
- **不寫入 `chapters/_index.md`**（狀態欄、備註欄皆不動）——刻意不同於 `beat-test`/`outline-test` 會更新就緒儀表的慣例：章節測試結果時效性短，作者稍一修改內容就會過期，與其維護一個容易失真的狀態欄，不如等系統整體穩定後再考慮程式化追蹤。
- 不做已寫章節的修稿／潤色——本 skill 只出報告；作者要把可疑點修掉時走 `revise`（定點修訂，只改被點名段落）。
- 不建規模化連貫性基建（跨大量章節的一致性追蹤、自動偵測幕綱是否於成章後又被修改）——子專案 C 範圍。
- MVP 在主 session 執行；若日後全書測試 context 過大，再考慮改為獨立 subagent（非本階段範圍）。
