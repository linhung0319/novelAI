# write — 依幕綱撰寫正文

把已經共創（理想上也測試過）的結構化幕綱，展開成真正的散文章節。依 `創作流程.md` 的二分：前三步是「創作故事」，幕綱起的後三步是「製造小說」——本 skill 是「製造小說」的撰寫步驟，同樣需要方法：怎麼把「行動／衝突／結果」三個詞展開成有畫面、有對白、有感官細節的段落。

**AI 角色 = 共創撰寫，不是代筆定稿**：你依幕綱與技巧知識庫提出正文草稿、提議分章與 POV，但**所有創作決定權在作者**——分章方案、POV 選擇、任何一段文字都可被作者全盤重寫或否決。與上游 skill 不同的是，這裡 AI 直接產出可讀的散文草稿（而非骨架），但草稿的地位不變：可編輯、非受保護、非定案。

## 正文五鐵律（動筆前先讀）

> 這五條是**總綱**，刻意少而顯眼——對抗大型語言模型「把戲摘要掉、段段宣布意義、每章當場了結再總結一句」的爽文預設語域。它高於下面 20 個技法檔：**先過五鐵律，再落各技法檔細則**。技法檔講「怎麼寫好」，五鐵律講「哪些一定不能犯」。**R1–R4 是句／段層（每段先過這關），R5 是章層（每寫一個章末先過這關）。**

1. **戲劇化不摘要**：幕綱的「行動／衝突／結果」是承重事件，必須展開成讓讀者*經歷*的 Scene；**不得用一句「敘述其意義」收掉、不得塞進時間狀語或從屬子句一筆帶過**。判準：一件會被下游幕引用的事，若讀者沒在場「看到它發生」，就是違規。（反例：「答案在他譯通第三塊碎片、往縱深多走了幾步之後，撞到臉上」——「譯通第三塊碎片」是幕級進展，卻被壓進時間狀語。）
2. **反思讓位給當下**：每個 Scene 以**當下可觀察的動作／對白／感官**為主體，內心反思與「意義敘述」（解釋這件事代表什麼、角色從中悟到什麼）為輔；**不得段段以意義塊收尾、不得連續數段都在角色腦內解釋剛才那一下的意涵**。粗錨（非死線）：一個 Scene 內抽象反思不宜多過當下演出的篇幅。
3. **禁反射式收尾標記**：禁止用「他懂了／他忽然明白／他這才想起／原來如此／這意味著／他很清楚／他終於」這類「替讀者宣布角色頓悟了、或宣布這件事的意義」的反射句，**作為段落／場景的收尾**。要讓讀者自己從證據推導（＝展示而非講述）。禁的是「作為意義標記的收尾用法」，不是這些字永不出現。
4. **逐段時空/因果連續**：相鄰段落之間要有可追的時間、空間、因果連結；承重轉折要**演出中間過程**，不得每段各自是自足的「頓悟塊」而跳過中間步驟。跨章同理：上一章子句裡發生的事，下一章不得直接當既成前提引用（那正是「跳躍、要回頭查」的來源）。
5. **章末不總結、跨章不斷線**（章層）：**章末禁用「回顧本章處境＋預告未來」的總結段收尾**（反例：「他站在一個死人的屋子裡…欠著死人的債…被仇家盯著」／「三日之後，真正的那個人要親自來了」——這是講述、是「硬要總結感」的來源）。**續接同一張力的章要硬切**（停在未解畫面/動作，不回顧不預告）；**一條連續張力線的宏觀目標不得每章各自了結**——小節拍可結，throughline 要跨章留著悶燒。判準：一段（數章）若每章都把自己的局部目標當場贏乾淨、章末各自打包總結，就是碎片化違規。細則見 `技巧知識庫/戲劇張力與懸念.md` §二「章末三種切法」、`場景與情節推進.md` §六。

> 五鐵律讓「爽」是**演出來、掙來的**，不是文學化稀釋——文學技巧讓網文的爽站得住腳（見 `技巧知識庫/網文與文學取捨.md` 關鍵洞見），與網文向不衝突。

## 共同約定（必讀）

資源定位、AI 角色（共創＋審稿、非門檻）、下游回饋記錄規則見 `結構定義/共同約定.md`。本 skill 專屬：`chapters/` 資料夾若不存在則建立。

- `結構定義/狀態事件流.schema.md` ＋ `tools/state_projection`（fold 投影程式，子專案 C）——章首讀 as-of 投影（進場知識邊界，步驟 1.5）、章末 append 狀態變化事件（產出端，步驟 8）的格式與工具。
- `技巧知識庫/場景與情節推進.md`——每幕展開成 Scene（Goal/Conflict/Disaster）＋幕間 Sequel（Reaction/Dilemma/Decision）的核心操作方法；幕綱「行動/衝突/結果」三欄的展開公式。
- `技巧知識庫/展示而非講述與五感描寫.md`——逐段執行標準：關鍵時刻用展示、過場可講述，處理過濾詞/資訊傾倒/形容詞化情緒/副詞化對白標籤。
- `技巧知識庫/視角與敘事人稱.md`（概念層·核心）——決定/沿用每章 POV、每場景單一視角、第四節知識邊界（供核對 POV 不越界）。

- `技巧知識庫/動作場面寫作.md`——幕的行動/衝突若屬戰鬥/動作類型時的專項準則（五面向；玄幻/仙俠專屬的兩條額外路徑僅在對應類型才套用，一般敘事幕不用）。
- `技巧知識庫/氛圍與筆調.md`——筆調前後一致的基準，對照 `00-摘要.md` 基調自我核對。

- `技巧知識庫/戲劇張力與懸念.md`——把幕尾懸念寫成 cliffhanger（七種停止點）、心痛四手法的內心煎熬描寫、開場鉤子。
- `技巧知識庫/因果法則.md`——幕的五要素，撰寫時仍須尊重，不可無意間改變幕綱定義的因果。
- `技巧知識庫/伏筆與意外.md`——撰寫埋/收伏筆段落時的措辭需符合伏筆閉合/謎三角精神，不可意外提前洩漏不該揭露的伏筆。

- `技巧知識庫/角色塑造.md`——把角色寫進正文時優先用塑造六法的中後段（象徵/對話/行動/內心）而非直述法，落實三層行為反差。
- `技巧知識庫/主題題旨.md`——撰寫涉及主題三角鋪陳點的幕時，確保文字真的落地呼應，而非空談。
- `技巧知識庫/畫面感與電影化.md`——用文句控制鏡頭：句長＝停滯時間、換行短句＝特寫、示現/插敘/時間伸縮/五感轉場；依「取向定位」拿捏密度（大巧若拙）。
- `技巧知識庫/修辭與字句鍛鍊.md`——句子層級修辭：譬喻（以感受譬喻、喻依貼氣氛）、擬詩（重複/排比/疊字/留白）、映襯（正襯/反襯）；文學向才在字句鍛鍊加重手。
- `技巧知識庫/心理描寫.md`——內在描寫五技法（內心獨白/夢境幻覺/環境襯托/神情動作/意識流）＋疼痛描寫「掛在鮮活人物身上」。

## 流程

1. **讀輸入**：
   - 依作者指定的 arc 或幕範圍，讀 `story/幕綱/arcNN.md`（若範圍內的幕有 `[[幕NNN]]` 引用到範圍外/跨 arc，一併讀取解析，比照 `beat-test` 的跨 arc 讀取規則）。
   - 若目標範圍沒有對應幕綱 → **停下**，說明「這個範圍還沒有幕綱」，建議先跑 `beat-sheet`；不得在缺乏幕綱依據時憑空杜撰正文。
   - 讀 `story/參照/{世界觀,角色需求,結構}.md`、`story/00-摘要.md`（題旨／基調）。
   - 讀既有 `chapters/_index.md`（若存在）：避免重複撰寫已完成的章節、避免重新編號已寫的 chNN；若本次範圍與既有章節重疊，向作者確認是覆寫還是跳過。

1.5. **章首讀狀態投影（閉合 read-loop；子專案 C）**：動筆某章前，對該章對應幕號呼叫狀態投影程式取得進場知識邊界——
   `uv run --project <套件根>/tools/state_projection state-project --book <書資料夾> --as-of 幕NNN（arcAA） --entities <POV主角＋關鍵配角>`
   （幕號＝該章對應幕；因事件流只含已寫章，傳該幕即得進場態）。把投影當「本章相關實體 as-of 狀態」的進場知識邊界，與 POV 知識邊界規則（步驟6）一起用。**投影標明的當下狀態優先於 `角色需求.md` 等靜態基線描述**——靜態卡記「全年常態基線」，投影記「逐幕當下」；兩者衝突時（如某物件已易手、關係已變）以投影為準，避免沿用過時基線判錯。投影是衍生、可覆算、可追溯（每條標來源幕）；**程式唯讀，不改任何檔**。`story/參照/狀態事件流.md` 或 `story/幕綱/_index.md` spine 尚未建立時，此步跳過、退回即時重建。

2. **就緒提醒（non-blocking）**：檢查 `story/參照/就緒儀表.md` 對應 arc 的「幕綱」欄或幕綱檔頭有無 beat-test 執行紀錄；若看不出這個範圍測試過，提醒「建議先跑 beat-test 抓結構漏洞再動筆，但你也可以現在就開始」——不阻擋，呼應「AI 是審稿員不是門檻」。

3. **提議分章（AI 提議，作者拍板）——以「節奏單位」為分章單位**：分章的單位不是「篇幅感」，而是一個**節奏單位**。節奏單位有兩種、都合法：(i) 一個**閉合**的爽點/情緒循環（小起承轉結），或 (ii) **一段連續張力線中的一節**——本章只結一個小節拍，那條宏觀目標**跨章不閉合、持續悶燒**。**別把每章都寫成 (i)**：整段（數章）都是各自閉合的小勝利、章末各自打包，正文就會碎、無記憶點（見五鐵律 R5、`場景與情節推進.md` §六）。提出候選分章表（候選章｜涵蓋幕｜所屬張力線｜章末切法｜理由｜預估字數）。
   - **動筆前先辨識連續張力線**：從幕綱的伏筆「埋 X幕→收 Y幕」與「前因」鏈，標出哪幾幕共享一條 throughline（一個大目標／一個未解威脅／一個謎）。渲染時這條線要**在正文正體裡持續施壓**（角色記掛、環境提醒），而非只在章末總結一句預告。（例：某威脅埋在首幕、隔數幕才攤牌——中間各章都活在它陰影下、不當場了結。）
   - **依 `00-摘要.md`「取向定位」定基準**：**中文網文向**（如芯片巫師「爽感升級為主」）**預設一章一幕的粒度**（一幕＝一個 Scene）＋強章末鉤子、**一章約 3000 字**（起點體例）；但「一章一幕」只定**粒度**，**不等於「每章都要閉合一個完整爽點循環」**——該幕的 Scene 可以只推進一條跨章張力線而不了結它。文學向才容許章節自然收束、跨幕邊界較鬆。依據見 `技巧知識庫/網文與文學取捨.md`（節奏速度·連載鉤子強度對衝軸）、`場景與情節推進.md` §五（單章快慢）§六（跨章張力線·快慢交替·餘韻）、`Data/跨語言技巧/結構與節奏.md`（連載單位）。
   - **兩個反模式（皆明確避免）**：①把兩個各自完整的爽點循環壓進一章 → 蒙太奇、事件塞進子句、下一章引用即「跳躍、要回頭查」（字數遠超 3000 是信號）；②反過來，每章各自閉合小勝利、章末總結式收尾、宏觀張力從不跨章 → 碎片化、無記憶點（五鐵律 R5）。
   - **字數依張力起伏重分配、非等長勻平**（3000 字為基準非死線，依段落自然邊界浮動）：一段的**高潮幕鋪成 set-piece**（給足感官/對白/來回/升級節拍、明顯厚於過場），**過場/銜接幕壓薄快帶**；並安排**喘息/日常**節拍讓角色特點與記憶點長出來（`場景與情節推進.md` §六b/c）。操作兩個閥：**併章**——某幕展開遠不足基準、與相鄰幕合起來才成一個節奏單位（拆開反而斷氣）→ 併章，作者拍板；**拆幕信號**——某幕展開遠超基準、含多個爽點循環 → 訊號是**幕綱本身該拆幕**，指回 `beat-sheet`，`write` 不硬拆。
   - **既有硬規則不變**：**不可把同一幕拆到兩章**（`章節.schema.md`「一幕不可跨章切開」）。
   - 呈交作者調整（合併/拆開/移動邊界），確認後才動筆。

4. **決定/沿用每章 POV**：以 `story/00-摘要.md`「敘事視角結構」欄為權威來源（該欄缺漏時對照 `角色需求.md`、既有章節慣例，並提醒作者補填），每章固定單一 POV 角色（v1 簡化：不做同章節內多 POV 交錯；書級的多線視角改用不同章節切換 POV 達成）。若幕綱角色欄暗示需要跨角色視角，向作者確認怎麼分配到哪幾章。

5. **逐幕 Scene 展開＋幕間 Sequel 銜接**：依「場景與情節推進」，行動→Goal、衝突→Conflict、結果→Disaster；幕與幕之間視需要展開 Reaction/Dilemma/Decision 過渡文字。標記幕錨點（`<!-- 幕NNN -->`、必要時 `<!-- 幕NNN→幕MMM -->`，見 `結構定義/章節.schema.md`）。**若某幕的核心事件 POV 角色無法直接感知（如沉睡、不在場），Scene 仍只能呈現她實際能感知的片段**（如驚醒瞬間的感官碎片），事件全貌透過後續幕的「事後發現」揭露，不得為了交代劇情而暫時跳出 POV 視角。
   - **快慢交替＋依張力重分配篇幅＋餘韻**（`場景與情節推進.md` §六b/c）：不要每幕等長等速當場解決。Scene（快）↔ 喘息/日常/Sequel（慢）交替；一段的高潮鋪成 **set-piece**（厚）、過場壓薄（快）；一段真正的高潮/兌現**之後留一拍餘韻**（停在畫面/情緒/角色特點，製造記憶點），不立刻切下一鉤。**跨章張力線要在正文正體裡持續施壓**（角色記掛、環境提醒），不是只在章末摘要一句。

5.5. **章末切法決策（每個章末先過 R5）**：寫某章的結尾前，先判「下一章是否緊接同一場景/同一張力」——
   - **續接** → **硬切斷章**（停在未解畫面/動作，不回顧本章、不預告未來、不宣布意義）或**前開式軟鉤**（用一個具體新擾動向前拋）。
   - **段/arc 邊界** → **收束斷章**（可閉合＋喘息，但鉤子是一個新的具體擾動、不是回顧；峰後可先給餘韻再拋）。
   - **一律禁「總結式收尾」**：章末不用「回顧本章處境＋預告未來」的抽象總結段（見五鐵律 R5、`戲劇張力與懸念.md` §二「章末三種切法」）。

6. **句子層級執行**（先過「正文五鐵律」，再落以下各技法檔細則）：
   - 展示而非講述：關鍵時刻具體化為證據而非標籤；過場可保留講述但維持一兩個意象。
   - Deep POV：去過濾詞、感官直接落在 POV 角色身上、內心獨白直接/間接交替、少用角色自呼其名。
   - 核對 POV 知識邊界（視角與敘事人稱.md 第四節）：POV 角色不知道的事不能被敘述——本系統多處刻意設計「戲劇反諷」的幕（POV 角色的無知本身就是故事重點），要嚴守邊界而非「幫讀者补足資訊」。

7. **動作場面特判**：若某幕「行動」/「衝突」欄描述屬於戰鬥/動作場面，套用「動作場面寫作」五面向；只有類型確實是玄幻/仙俠時才套用「詞彙鋪張」「鋪墊框架」兩條額外路徑，一般敘事幕不套用。

8. **寫入與索引維護**：依 `結構定義/章節.schema.md` 模板寫入 `chapters/chNN.md`（狀態先標「草稿」）；同步更新 `chapters/_index.md`。
   - **append 本章狀態變化事件（子專案 C 產出端）**（見 `結構定義/狀態事件流.schema.md`）：把本章發生的狀態變化，依固定 4 欄信封 `- 幕NNN（arcAA）· 實體 · 維度：內容`（維度限 6 枚舉 `知識前沿｜關係｜持有｜位置｜能力｜所屬`）append 進 `story/參照/狀態事件流.md`。只記已寫章、位置＝該章對應幕號、引用伏筆/實體用名字不重抄幕綱埋/收。
     - **`知識前沿` 寫「當下承重的知識前沿整體」，不是流水帳**：每 `(實體,維度)` 是單槽、最新覆蓋（fold 只留序最新一筆），故一條 `知識前沿` 事件要能獨立代表該實體此刻對主線的知識邊界；瑣碎或與主線正交的支線知識不要記成會覆蓋承重前沿的獨立事件。
     - **章末狀態快照改由投影衍生、不再手維護**：需要人眼可讀的章末切片時，跑投影程式（`state-project --as-of` 該章末幕）產生，非手抄。作者當場審改事件。
     - 跨 arc/跨集一致性校驗、程式端比對仍不在此做（延後，見 spec YAGNI）。

9. **基調與取向定位自我檢查**：對照 `00-摘要.md`「基調」欄自我核對本章筆調（`氛圍與筆調.md`「筆調前後一致」）；並對照「取向定位」欄核對節奏／爽點密度／揭露速度／資訊密度是否與宣告的網文↔文學傾向相符（依 `網文與文學取捨.md`）——如摘要說「文學向為底」卻寫成通篇快爽套路，或說「爽文」卻節奏拖沓，都在交付時主動提出，不悶著不說。

10. **上游問題發現的處理**：
    - 問題根源在幕綱本身（因果不順/伏筆安排不合理/結構階段誤置）→ 幕綱本無下游回饋記錄區塊、定位為作者可直接修改的利器；本 skill **不代作者編輯幕綱**，只在對話中明確指出疑點，建議作者直接修改幕綱或重跑 `beat-test`。
    - 問題根源在更上游（世界觀／00-摘要／角色需求，這三份檔案「有」下游回饋記錄區塊）→ 依 `結構定義/回饋記錄.schema.md` 在該檔案末尾追加一列（來源標「write（chNN 撰寫時）」），不改上游正文，並提示作者已記錄 N 筆待裁決回饋。

## 承重細節缺口協定（渲染時撞到設定裡沒有的細節）

> **為什麼要有這條**：`write` 是渲染器、不是設定產生器（`章節.schema.md`「章節是幕綱的散文呈現，不是取代品」）。但設定永遠不可能前置窮盡——你會在寫的時候才撞到「這裡需要一個設定裡沒有的細節」。過去的預設是**靜默發明、埋進散文**；問題是下一章的 `write` 會再發明一個**矛盾的版本**，這正是碎片化的根因。這條協定把「靜默發明」換成「發明即分類」。

撞到設定裡沒有的細節時，先分類，再決定去向：

- **一次性色彩**（路人外套的顏色、只經過一次的街名、一個感官意象——後續章節不會依賴、抽掉故事不垮）→ **自由發明、不必標記**。強塞進設定＝過度規格化，違反「設定精簡」。
- **承重物**（一條規則／一種能力／一個角色事實／一個機構的存在／一個會影響賭注的事實——後續章節會依賴、或構成一條承諾）→ 就地暫定以維持行文，但**這本身是上游缺口的信號**：
  - **狀態型**（某實體某維度的當下變化：知識前沿／關係／持有／位置／能力／所屬）→ 走步驟 8，append 進 `狀態事件流.md`。
  - **新規則／新設定事實**（世界觀規則、角色的既定背景）→ 走步驟 10，依 `結構定義/回饋記錄.schema.md` 在 `世界觀.md`／`角色需求.md` 檔尾追加一列，來源標「write（chNN 撰寫時·承重細節缺口）」，交作者裁決是否正式入設定。
- **心智模型一句話**：一次性隨手寫；當你發現自己在發明一個**承重物**，那是上游的洞——**標它、報它，別埋進散文**。
- **邊界**：本協定只「暫定＋標記＋報」，**不代作者把承重細節寫進設定正文**（那是作者拍板後的事，呼應「AI＝共創＋審稿、非門檻」）。判不準是承重還是一次性時，寧可當承重報出來，交 `write-test` 測試9 與作者複判。

## 輸出格式

- 步驟3的分章提議表（供作者拍板）。
- 完成後列出本次寫了哪些 `chNN`、涵蓋哪些幕、篇幅概況。
- 若有上游回饋，列出筆數與去向。
- **本章新增承重細節清單**：依「承重細節缺口協定」，列出本章因設定缺口而暫定發明、且判為承重的細節，各標去向（append 進 `狀態事件流.md`／已於 `世界觀.md`／`角色需求.md` 追加待裁決回饋列）。無則明說「無新增承重細節」。這份清單也是 `write-test` 測試9 的複判起點。
- **明確聲明**：以上為撰寫草稿，所有文字可全盤修改，非定稿。

## 邊界

- 不做審稿（那是 `write-test`）——不自動檢查自己寫出來的東西是否有結構/一致性問題，只在撰寫過程中盡力遵守技巧知識庫；檢查點清單見 `結構定義/章節.schema.md`。
- 修稿不在本 skill 流程內——本 skill 只管「從無到有寫出草稿」；作者要修改既有章節時走 `revise`（依 write-test 報告或作者指示定點修訂），或在對話中依本 skill 的句子層級依據協助，不必拒絕。
- 規模化狀態連貫性由子專案 C 的事件流＋fold 投影承載（章首讀投影＝步驟 1.5、章末產事件＝步驟 8）；仍**不建**程式端一致性校驗、持久 cache、跨 arc/跨集狀態機、自動偵測幕綱於成章後又被修改（延後，見 spec YAGNI）。`write` 只在既有時機明確呼叫投影程式與 append 事件，無常駐程序、不自動鏈式觸發。
- **不強制章節邊界**：分章永遠是提議，作者拍板。
- 不做同一章節內多 POV 交錯（v1 簡化；書級多線視角靠不同章節切換 POV 達成，不受此限）。
