# 情境測試：skill 覆蓋 / 檔位

## 這是什麼、為什麼

把「作家在各創作階段常見的求助」逐一丟給系統，檢查每個情境**是否有對的 skill／檔位承接、且不誤觸發**。作用等同程式的回歸測試：**每次新增或改動 skill、技巧知識庫、結構定義後，回頭重跑本表，確認覆蓋沒有漂移**（既有的仍命中原路徑、新增的能力沒把別的情境搶走）。

- 本表只驗「**有沒有對的操作可接**」，不驗產出品質（品質由各 test 類 skill 與作者把關）。
- 承重原則不變：所有承重抉擇作者拍板；skill 只攤選項與後果。
- 只列常見情境，不窮舉罕例。

**判定圖例**：✅ 乾淨命中單一 skill／檔位　⚠️ 命中但流程對此需求偏重（應能降檔）　❌ 無對應操作（缺口）

## 三個檔位（判定基準）

系統對「非審稿」的求助有三個檔位（見 `CLAUDE.md`「沒有 skill 觸發、或需求太小…」段）：
1. **答問**——回答技巧問題，無產物；先查 `技巧知識庫/_index.md`。
2. **輕量輔助模式**——小型生成／定點編輯（只動單一產物局部、不產生新結構承諾、不需拍承重抉擇）；查 `_index` 拉技法檔＋權威資料，產單點提案，不寫檔或只提議一處，不跑整套 skill。
3. **完整 skill 流程**——大動作（產生新產物或結構承諾）。

小需求誤入完整 skill 時，靠**降檔閥**退回檔位 2；小需求牽出結構問題時，靠**升級閥**升到檔位 3。

## 觸發不亂（負向測試，改動 skill 後必跑）

新增／改 skill 後，確認新能力沒把既有情境搶走：

- 「把這幾幕寫成正文」→ 必須 `write`，**不得** `character`（雖含「角色」字樣）。
- 「幫我設計一個反派」→ `character`，**不得** `develop`（非冷啟動）或 `expand`（非劇情缺口）。
- 「這個 arc 只有一句話，長厚」→ `expand`，**不得** `organize`（非整理已收斂素材）或 `character`。
- 「整理成大綱」→ `organize`，**不得** `expand`。
- 小型單點需求（懸念一則／改一句）→ 檔位 2，**不得**被迫跑完整 skill；若 description 誤命中重 skill，應由降檔閥退回。
