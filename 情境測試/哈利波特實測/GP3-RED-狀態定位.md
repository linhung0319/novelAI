## 二、剝乾淨步驟與驗證

前五輪教訓：fixture 會被污染（前輪備忘、`就緒儀表` 裡 sync/worldbuild 的自發備忘、**git 工作區 diff**），剝要連這些一起剝。本輪 fixture 是**全新產出**，天生無前輪污染，但仍逐項驗證：

1. **無偷渡狀態**：`grep -ri '章末狀態快照|知識前沿|事件流|狀態事件|as-of|關係刻度'` 於 `GP3-fixture/` → **無匹配**（唯一早期命中在 README，已將 README 改寫成中性書介、移除「這是測試／答案在別處」等字樣，避免子代理誤入偏誤）。
2. **參照檔剝乾淨**：複製來的 世界觀／角色需求 已刪去全部「下游回饋記錄」列與「（2026-07-15 worldbuild 拍板）」「見情境測試…」等測試輪備忘；只留靜態規則與靜態弧線。無逐章時間線。
3. **git 工作區乾淨**：`git status --short` → 只有 `GP3-fixture/` 一個新資料夾；系統套件三資料夾零改動。**先把乾淨 fixture 提交到分支**（commit `5e0602c`），使子代理即使 `git diff`／`git log` 也撈不到任何未提交的提示（呼應第二輪「Arm 3b 從 git diff 撈回剝掉的備忘」教訓）。
4. **子代理冷跑框成真實寫作協助情境**：prompt 只指向 fixture 內特定檔、明示「這是獨立專案、只讀這個資料夾」、不告訴它這是測試、不透露正史答案、不下引導性提問（問「他此刻相信什麼」而非「他是不是還以為是史奈普」）。
5. **冷啟動隔離**：每一跑都是全新 `Agent` 子代理，天然隔離 context；同一查詢的多跑是互相獨立的冷啟動。

## 四、判定理由

### 為什麼是偽缺口（未複現），而非「fixture 太小沒逼出來」

1. **最該複現的兩類陷阱也沒複現。** 任務明說「學而後改／靜態弧線洩底最狠——讓小 fixture 也能複現失敗，因為失敗不是『讀不完』而是『往前讀把未來/終點洩回過去』」。這兩類**不依賴規模**、小 fixture 就能觸發洩題——結果 3/3 守住。冷跑穩定區分「哈利 as-of ch11 知道什麼」與「我（Claude）從正史/後章知道什麼」，未把 ch12 的翻案洩回 ch11、未把弧線終點洩回 ch03。這是對「小規模即可洩題」假設的**直接反證**。
2. **無結構化提示也守得住。** Q2/Q3/Q4 沒有任何 🧊 指標或狀態表在提示，冷跑純靠散文＋靜態弧線重建，仍全對。可見不是「答案被偷渡」造成的假陰性。
3. **現況本就冗餘承載。** 邊界探針證明連砍掉散文、只剩幕綱，都足以定位 as-of 信念。呼應前輪反覆出現的「系統講了六遍、實際什麼都沒漏」（第 1 群 RED 的飽和）。C 在此規模是重複建帳。
4. **與設計書一致、非矛盾。** 設計書 §1.2 明言「在 ch05 規模即時重建完全夠用、未複現失敗」「read-loop 只在規模化時才咬人」，§7 明言落地要等「資料量成長到即時重建不可靠時」再跑 RED。本輪把可讀規模拉到 12 章仍未複現，正是「尚未到咬人規模」的實證。

### 方法論張力（誠實揭露，供作者權衡）

設計書主張的失敗場景是「靠**重讀 30 章正文**既貴又不可靠」。要在 RED 裡複現「重讀不可靠」，就得把 fixture 撐到冷跑**讀不完**的規模——但那樣一來，「失敗」會是 context 上限的產物、而非系統設計缺口；而只要規模仍可讀，就不複現失敗。也就是說，這個失敗模式**用「冷跑讀散文」的 RED 方法在任一可讀規模都難以複現**。這把「缺口」的性質從**正確性失敗**（判錯／洩題）推向**效率／成本問題**（每次寫作都重讀 N 章正文很貴）——後者是真實的工程考量，但不是「剝乾淨冷跑會判錯」意義下的真缺口，也不該用「補契約防洩題」去解。

## 五、對子專案 C 的建議（判偽缺口 → 建議延後，不逕自實作）

1. **延後 C**，維持設計書「未落地」狀態。本 RED 提供了設計書 §7 要求的「現規模剝乾淨冷跑」證據：**未複現失敗**。改不改、何時改，一律等作者拍板。
2. **現況已足**：`chapters/_index.md` 章末狀態快照（write 收尾維護，別處沒有的角色視角切片）＋幕綱逐幕欄＋🧊 指標＋摘要 POV 鎖，四者冗餘承載 as-of 狀態，冷跑在此規模全對。零成本過渡已足以服務單集／連續章。
3. **重新界定 C 的觸發訊號**：C 若要落地，正當理由應是**效率／成本**（大規模下每次寫作重讀全部正文昂貴），而非「防判錯／防洩題」（本輪證明現況不判錯不洩題）。落地前應跑的 RED 不是「小 fixture 冷跑」（已證不複現），而是**成本量測**：在一本真正寫到數十章的書上，量「即時重建的 token/延遲成本」與「重建正確率是否隨規模衰減」——只有當**正確率隨規模實際下滑**（而非僅成本上升）時，才構成需要事件流投影的正確性理由。
4. **若日後真要落地，仍守設計書 §3 的最小形狀**：append-only 狀態變化事件流（每事實只寫一次、綁幕序、引用伏筆用名字）、快照降為可重算投影、沿既有 spine 做 as-of 回放、需求驅動不預先窮舉實體型別、不建帳本／不自動觸發／不另立登記簿。

## 六、方法論教訓（延續 `_index.md` 累積，第六條）

**「read-loop 只寫不讀」是體感真實、但「因此會判錯／洩題」在可複現的冷跑規模下未成立。** 前五輪的鐵律再次應驗：真缺口＝契約缺一條＋有機械消費者＋剝乾淨冷跑複現；GP3 三關全不過——狀態定位沒有機械消費者（是 LLM 讀上下文重建，非 parser 消費欄位），且冷跑 17/17 不複現。更關鍵的一條新教訓：**當一個假設的失敗模式，其複現條件（大到讀不完）與 RED 方法的前提（冷跑要能讀 fixture）互斥時，該「缺口」多半是效率/成本問題偽裝成正確性缺口**——要用成本量測而非剝乾淨冷跑去驗，且很可能該用工程手段（快取/投影）而非「補契約」去解。至此本系統所有經 RED 檢驗的「缺一個機制」型缺口，無一複現成正確性失敗。
