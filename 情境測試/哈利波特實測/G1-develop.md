# G1 · develop（粗階段冷啟動）

**判定：✅ 階段目標達成**

**① 作者目標**：「raw/ 一堆點子很亂、還沒收斂——幫我把主線、題旨、基調定出來。」（主角/視角/結局/題旨/取向/基調全未拍板。）

**② 系統路由觀察**：
- 路由縫測試：作者若照模板念「幫我把 raw/ 整理成大綱」，粗層未就緒時**正確終點是 develop 不是 organize**（organize description 明訂連局部都很亂要先 develop；develop description 命中「raw 未收斂出主線/題旨/結局」）。規則本身正確、organize 會退檔。
- **Skill 工具在子代理 context 內可用**：`Skill(develop)` 成功叫起，無須退回手動讀檔。（＝子代理協作模式可行，驗證通過。）

**③ 執行產物**（寫入 4 檔）：
- `story/00-摘要.md`：一句話＋50/100/200（一致）；題旨＝因為愛與選擇（母愛護盾＋「是選擇非血統決定你是誰」）→ 戰勝恐懼/偏見/死亡（標可檢查）；基調＝主暖副暗、開場暖漸沉（標主從＋轉調曲線）；視角＝單線第三人稱有限 Deep POV、鎖死貼哈利以成立「誰是真凶」懸疑；取向＝兒少商業向、五對衝軸逐條標＋系列首集擊退不消滅。
- `story/參照/世界觀.md`：6 條核心規則（隱藏世界／死亡不可逆／魔杖挑主人・兄弟杖／母愛護盾／未成年與電干擾／學院競賽）逐條標影響力（3 條「抽掉就垮」）；七維度盤點；4 條待確認矛盾。
- `story/參照/角色需求.md`：哈利四象限＋弧線＋馬斯洛＋兩難；黑巫師正當需求（拒死＋純血支配，鏡像對衝）；7 配角各一行 stub。
- `story/參照/就緒儀表.md`：世界觀/主線題旨基調＝就緒、角色＝草擬、結構＝空白；就緒清單 6 項全勾。
- 分揀四類有做並列回作者；三訣竅健檢有跑（別太多／別耍人各亮黃燈，皆為 HP 真實設定張力）；⚠️ 一致性限制聲明**逐字**出現在對話輸出。

**④ HP 正史對照評估**：**這階段作者目標達成。** 00-摘要 把一堆亂點子收斂成可下沉 organize 的方向，與 HP 正史一致；JK 的核心痛點（不知主線/題旨/基調怎麼定）被解掉。承重分叉涵蓋典型（主角視角/結局/題旨/取向）全命中且拍板與正史一致；分揀正確；健檢兩黃燈抓得準；角色打底達標。已用 Read 覆核 00-摘要.md 屬實、非紙上宣稱。

**⑤ 摩擦與缺口（→潛在新功能候選）**：
1. **模板路由縫（優先）**：`書本模板/README.md`（＝新書 README）第 3、11 行預設冷啟動指令是「整理成大綱和幕綱 → organize」，會把 raw 全未收斂的新書作者字面導向 organize，與檔位規則相衝，只能靠 organize 自己退檔。→ 模板預設語改分階段，或加「raw 還很亂就說『幫我把 raw 想清楚』（走 develop）」。
2. **基調未進 develop 承重分叉典型**：develop 核心分叉清單只有主角視角/結局/題旨/取向，**漏基調**；而基調是本例作者明列的頭號待決、且「無意轉調」是系統反覆強調的風險。目前靠 摘要 schema 必填欄兜住。→ 把「基調（主從＋轉調曲線）」補入強制分叉典型，或於 SKILL 明訂經 摘要 欄強制拍板。
3. **分揀結果無處落檔**：skill 要求把四類分揀列回對話，但沒規定寫進任何檔——對話結束即消失，下游 organize/worldbuild 看不到「哪些 raw 是 research 待查／哪些是既有結構片段」。→ 分揀表持久化（就緒儀表附區塊或 raw 索引檔）。
4. **互動假設 vs 單輪/批次執行**：develop 全程假設「攤選項→等作者拍板」多輪互動；自動化/子代理單輪跑時無天然「作者回覆」輪，逐字聲明/逐條健檢要靠執行者自覺塞進輸出。→ SKILL 補「非互動/批次模式最小落地清單」。
5. **粗階段可能過早坐實「揭露型機制」**：為讓題旨可檢查，「母愛護盾」被寫成世界觀公開核心規則，但它其實是第一集後段的 reveal。粗階段定為明規則，對後續 write 的資訊控制（Deep POV 不能比哈利早知道）有潛在張力。→ develop 對「屬 reveal 的機制」增設「標為隱藏規則/伏筆，不當公開設定」的區分。
