## 判定（先講結論）

**GREEN——控制臂失敗 5/5、處理臂成功 5/5。** context-cap 在**大規模 fixture ＋ 陳舊基線（stale-baseline）**設計下**乾淨咬到**：無子專案 C 時，AI 因讀不到「決定性早章事件」而**倒向靜態參照裡的陳舊基線、判錯當下狀態**；給它 fold 投影後，5/5 全部改判正確、並主動聲明「以逐幕事件流投影為準、蓋過全年基調式的靜態描述」。

這**不是**推翻兩輪 RED，而是補上它們結構上摸不到的那個 regime：兩輪 RED 證明「當視窗對某狀態**靜默**時，謹慎的模型不會亂猜、而是誠實標記『需回查早章』」——所以正確性失敗不複現。本輪找到會真正複現的失敗形狀：**當一份總是載入的靜態參照正面斷言了一個『當下已過時』的基線**（角色隨身斗篷），而推翻它的變化（沒收）落在視窗外時，模型會**確信地沿用陳舊基線而判錯**。這正是 event-sourcing／as-of 投影要解的「陳舊快照」問題，也正是規模化下真正咬人的失敗。

## 三、冷跑基線：兩臂結果

框法：「我要動筆寫 ch16 下活門，POV 貼哈利，盤點他章首此刻手邊有無隱形斗篷可用／認定誰要偷石／哪些不可寫。」中立框法、不說是測試、不給答案、每查詢全新冷啟動子代理。

## 四、誠實揭露：第一版 cap 沒咬到，為何、如何修正

**第一次嘗試（silent-window）失敗、未複現。** 初版把決定性狀態從「所有頻道」抽掉（早章正文＋逐幕幕綱），視窗對斗篷**完全靜默**。結果控制臂 5/5 **答對**（判「無斗篷」）——但不是靠 C：而是(1) 幕201 outline 當時寫「摸黑溜」洩漏了無斗篷線索；(2) 更根本地，框成「原創專案」時模型**拒絕套用 HP 正史先驗**、只從 source 推理，遇到視窗靜默就**誠實標記「ch01–12 在視窗外、我無法確認、請作者回查」**而非亂猜。這正是兩輪 RED 的結構天花板第三度應驗：**靜默視窗下，謹慎模型不會製造正確性失敗。**

**修正＝改用 stale-baseline。** 把斗篷寫成 `角色需求.md` 的靜態隨身基線（正面斷言「有」），沒收事件只留在視窗外的事件流／早章。這不是掏空、是**真實常見的場景**：靜態角色卡記常態、劇情事件改變它、後段寫作者不回讀那一章就會沿用過時基線。改版後控制臂 5/5 **失敗**（沿用陳舊基線判「有斗篷」），cap 乾淨咬到。此修正把失敗從「靜默→誠實標記」轉為「陳舊基線→確信判錯」，後者才是規模化真正的失敗形狀，也才是 C 該解的。

## 五、附帶發現（modeling 教訓，供 schema 與 write 整合參考）

- **`知識前沿` 是單槽、最新覆蓋**：schema 規定每 `(實體,維度)` 只保留序最新一筆。手搭事件流時曾誤加一條 arcG 支線的 `知識前沿`（沙漏惡作劇），**覆蓋掉**了承重的「疑胡奇夫人」。這是 fold 照 spec 正確行為，教訓是：`知識前沿` 事件應寫成**當下承重的知識前沿整體**，瑣碎/正交的支線知識不應記成會覆蓋的獨立事件（或需夾帶延續）。已移除該噪音事件。→ 建議 `write` 產 `知識前沿` 事件時，寫「本章結束時該實體對主線的知識前沿」，非流水帳。
- **跨 arc 定序真實運作**：查詢 as-of 幕104（arcG）能正確回溯 arcF 的持有/知識前沿（fold 依 `全書順序：arcF → arcG → arcH` 定序，非幕號大小）。

## 六、對子專案 C 與 Task 6 的意涵

- **C 的價值被乾淨驗證**：在「大到讀不完、只能餵最近 K 章＋靜態參照」的規模，決定性早章事件落在視窗外時，**無 C 會沿用陳舊基線判錯（5/5），有 C 則照投影答對（5/5）**。這是效率／可靠性基建的正當性證據——與兩輪 RED「小規模冗餘承載、偽缺口」並不矛盾：規模一大、基線一舊，缺口就真了。
- **接進 `write`（Task 6）仍待作者拍板**：本報告提供閘門所需的「控制臂失敗／處理臂成功」證據；是否、何時把「產事件＋章首讀投影」接進 `write`，一律等作者決定。
- **系統套件三資料夾本輪零改動**（`git status` 可覆核，改動只落在 `情境測試/`＋`tools/`）；fold 程式唯讀，未寫任何書內檔或事件。
