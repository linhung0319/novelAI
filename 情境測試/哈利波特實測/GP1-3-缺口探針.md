## GP1 · 章末懸念（測降檔閥）— 判定：✅ 檔位2＋降檔閥健康

**① 作者目標**：「幫我想一個章末懸念就好，不用排整個幕綱。」（ch05 收在幕104＝713 金庫。）

**② 系統路由判定**：正確落 **檔位2 輕量模式**。`beat-sheet` description（排幕/懸念/伏筆）會被「懸念/幕」字樣**部分誤命中**（＝skill覆蓋情境測試 #12 標的風險），但作者「只想要一個、不用排整個幕綱」滿足**降檔閥**條件（單點、不新增/移動幕、不動既定因果），退回檔位2並可回問「要不要跑完整 beat-sheet 做系統版」。無升級閥。

**③ 執行產物**：取自 `技巧知識庫/戲劇張力與懸念.md`§2 七種 cliffhanger 之**疑問型**＋§1「已知造成的期待」——把 ch05 暖收往前收一拍、停在 713 之謎，末句補海格警覺一瞥（「想惦記它的恐怕不只哈利一個」）把純疑問升級成「疑問＋已知期待」，正對正史「同日 713 遭竊」伏筆。只提案、不寫檔。

**④ HP 正史對照評估**：路由正確、輕量模式教科書案例；提案與正史伏筆結構同源、不逾越 Deep POV（只寫哈利看見的海格警覺）；**無大砲打小鳥**（只讀一技法檔兩小節＋摘要基調，產一則提案）。

**⑤ 去向**：唯一摩擦是 beat-sheet description 對「懸念」吸附力偏強，靠降檔閥兜住即可、**暫不改 description**（改了反傷正命中）。檔位2＋降檔閥＝健康。

## GP2 · 單一場景/set-piece 設計（測已知缺口）— 判定：✅ 偽缺口，輕量模式可承接、不加映 skill

**① 作者目標**：「設計『哈利第一次進大廳、被會變夜空的天花板震撼』這一個 set-piece 的細節。」（arc03，純探針。）

**② 系統路由判定**：skill覆蓋情境測試「已知缺口」表列為未補（character 的對稱 sibling，待 character 驗完再定）。character 已於 G12 驗完。**現況權宜落點＝檔位2 輕量模式可部分承接**：查 `_index` 定位 `場景與情節推進.md`（Scene）＋`展示而非講述與五感描寫.md`（五感）＋`畫面感與電影化.md`（運鏡/句長）＋`氛圍與筆調.md`（基調），拉摘要/世界觀當 context，產「單景感官+運鏡提案」，不落結構。無 skill 誤觸發。

**④ HP 正史對照評估**：**缺口是真的，但比 character 當初更輕。** character 值得加映是因為它產出**跨全書複用的持久產物**（角色需求.md 一筆，帶四象限方法骨架＋與既有角色衝突檢查）；set-piece 的震撼細節是**一次性消化品**（寫進那一幕正文即用罄），不需持久檔、不需承重方法骨架——它的「設計」本質就是場景 craft，與輕量模式四個技法檔一一對應，沒有「方法論缺一塊」的硬缺。

**⑤ 去向（該不該加映）**：**判斷——現在不加映完整 set-piece skill，把輕量模式正式認定為它的家。** (a) 短期：情境套件補一列「設計單一 set-piece → 檔位2 輕量模式」，把已知缺口表此列由「未補/暫緩」改判為「輕量模式承接（✅ 非缺口）」，路由顯性化；(b) 升級條件：只有當實測出現「反覆要求把 set-piece 存成可跨章/跨集複用的產物」時，才做一個**薄** set-piece skill（輸出場景設計卡、不進正文）。在該訊號前，加映屬過度工程。

## GP3 · 逐章角色狀態演變追蹤（測已知缺口）— 判定：⚠️ 真缺口且最痛，C 去向正確＋建議零成本過渡

**① 作者目標**：「逐章追蹤哈利/榮恩/妙麗的狀態——誰知道了什麼、關係到哪一步、身上帶著什麼，一章一章推進。」

**② 系統路由判定**：skill覆蓋情境測試「已知缺口」表的「角色隨章節/時間演變狀態」，現況**無機制／去向子專案 C**。逐一排除：角色需求.md＝靜態弧線（軸向錯，每角一份非逐章時間線）；chapters/_index.md＝逐章但無「章末誰知道什麼」欄；sync 反向＝「已承諾事實清單」（供粗層收斂，非逐章狀態）。**確認無對應 skill/檔位，且連輕量模式都接不住**（要的是跨章持續維護的狀態機＝持久基建，非單點提案）。

**④ HP 正史對照評估**：**對長篇/系列極痛，三探針中最該優先。** 用 HP 為證同時卡三命脈——(1) **Deep POV 資訊控制**：摘要硬鎖「讀者不超過哈利所知、誰是真凶靠此成立」，須逐章知道哈利此刻知道什麼（何時知魔法石、何時仍誤信史奈普），否則 POV 洩漏防不勝防；(2) **伏筆連貫**：713→同日遭竊→活門→魔法石回收鏈的「哪章誰碰過/誰知道」若無狀態表易斷鏈，chapters/_index 現只能在「備註」硬記「埋伏筆:713」＝需求已敲門；(3) **關係演變**：鐵三角「初識→山怪結盟」刻度跨章跨集不追蹤會崩。**子專案 C 去向正確**（規模化連貫性基建，非單一 skill 能了）。

**⑤ 去向（C 之前的輕量過渡）**：**建議 C 之前先做近乎零成本過渡——`chapters/_index.md` 加「章末狀態快照」欄（三主角三格：知道了什麼/關係刻度/持有物），由 `write` 寫完每章順手維護。** 理由：(a) 已存在、已逐章、已由 write 維護的檔，加一欄邊際成本極低，且「備註」欄已在硬塞伏筆狀態＝需求早敲門；(b) 直取最承重的「知識前沿」一格，直接服務摘要鎖死的 POV 規則與伏筆回收，不必等 C 完整基建；(c) 定位清楚：只是單集/連續章內輕量快照，跨 arc/跨集時間線與一致性校驗仍歸 C，可平滑併入。落地：章節.schema.md 為 _index 增訂該欄、write SKILL 收尾加「更新章末狀態快照」。
