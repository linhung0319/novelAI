## 要驗的假設（H）

規模化時（例：第 30 章要用第 8 章確立的狀態），AI 因為「章末狀態快照只寫不讀、沒有事件流」而無法精準定位某角色在第 XX 章當下的**知識前沿／關係／持有**，於是**判錯或洩題**（把未來狀態或角色弧線終點洩漏回過去、或洩漏仍在水下的伏筆）。

## Fixture 設計（核心；不要偷懶也不要自導自演）

現有 `harry_potter/` 只寫成 ch05（一章），**單章逼不出規模化缺口**。請打造一份**專用 fixture**，遵守四道紀律：

1. **散文，不是整理好的狀態表**：失敗機制是「從一長串散文回放、折疊出某章當下狀態會漏或會串」。若給 AI 結構化狀態清單/事件流，等於把答案偷渡進去（假陰性/假陽性）。所以 fixture 必須是**正文散文章節**，逼 AI 真做重建。品質可粗（測的是狀態定位，不是文筆），但**狀態變化要忠於 HP 正史**。
2. **HP 正史當 ground truth**：沿用整套哈利波特實測的設計——AI 知道正史，你才能判它的 as-of 答案對錯。可接 `harry_potter/` 既有世界觀/角色需求/幕綱。
3. **刻意埋「狀態陷阱」**（重點；量不必大，8–12 短章、陷阱夠密即可）：

   | 陷阱類型 | HP 例 | as-of 查詢 | 測 GP3 哪條命脈 |
   |---|---|---|---|
   | 得而復失（持有） | 某物 chA 得到、chB 用掉/失去 | 「chA 與 chB 之間、之後各持有什麼？」 | 持有刻度 |
   | 學而後改（知識/信念） | chA 相信 X、chC 才知真相 | **「chB（介於中間）時他相信什麼？」** | Deep POV 資訊控制、防未來洩題 |
   | 關係翻轉 | 敵→友（山怪一役） | 「翻轉前/後那章的關係刻度？」 | 關係演變 |
   | 靜態弧線洩底 | `角色需求.md` 寫的是**終局**需求 | 「早期某章他到那個體悟了嗎？」 | 防把弧線終點當現況 |
   | 🧊 仍在水下 | 史奈普誤導線、魔法石身分 | 「某章的 POV 知道了嗎？」 | 伏筆連貫、知識邊界 |

   「學而後改」「靜態弧線洩底」最狠——讓小 fixture 也能複現失敗（失敗不是「讀不完」，而是「往前讀時把未來/終點洩漏回過去」）。
4. **剝乾淨**：冷跑控制臂前，把 `write` 自動產生的章末狀態快照、以及所有污染（含 git 工作區/worktree 殘留、就緒儀表自發備忘）剝光並驗證。**建議整個 RED 在 git worktree 隔離下做**，避免污染主工作區與 fixture 互相咬。

## 怎麼跑（兩臂對照）

- **控制臂（現況，無 C）**：給冷啟動子代理散文 fixture（快照已剝光、無事件流）＋幕綱＋靜態 `角色需求.md`。問它上表的 as-of 查詢：「我要在第 XX 章協助撰寫，角色 Y 此刻知道什麼／關係到哪／身上帶什麼？」對照正史判對錯。
  - 若控制臂**自己就答對** → 失敗未複現 → 該規模下缺口**不成立**（記為偽缺口或需更大規模）。
  - 若控制臂**判錯/洩題** → 失敗複現 → 進處理臂。
- **處理臂（模擬 C）**：同 fixture，但額外提供一份**手搭的狀態變化事件流**（依設計書第三節的最小形狀），再問同樣查詢。看是否修好。
- 每個查詢建議跑**多次冷跑**（如 3–5 次）看複現率——單次判對/判錯都不足以定案（呼應「跨跑變異」判準）。

- 結論要能回答一個問題：這缺口是「**真缺口**（契約/機制缺一塊、有機械消費者、冷跑複現）」，還是「**體感/已被既有設計覆蓋**」的偽缺口？
