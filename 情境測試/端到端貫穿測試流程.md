# 端到端貫穿測試流程（可重用）

> **這是什麼**：一套跨多 session 的測試流程——用一本現成好書當素材，由自動化的「模擬作者」驅動本系統，從 `raw/` 一路把整卷小說內文建構出來，藉此測「AI 小說寫作輔助系統」在**真實互動 + 大量內容**下的效果與問題。
> **設計依據**：`docs/superpowers/specs/2026-07-20-端到端貫穿測試流程-design.md`（唯一真實來源，本 doc 是其操作版）。
> **首跑標的**：`examples/一世之尊/卷一`。流程零硬編某本書，換書見 §8。
>
> **新 session 從這裡開始 → 直接跳 §5「續跑協定」。**

---

## 1. 定位（跟其他測試套件的分工）

| 套件 | 測什麼 |
|------|--------|
| `skill覆蓋情境測試.md` | 路由回歸：作者求助→有沒有對的 skill／檔位承接、不誤觸 |
| `哈利波特實測/` | 逐 skill 產物品質（HP canon 當 ground truth） |
| `芯片巫師/` | 管線貫穿（原創書＋**靜態** fixture），重壓正文層 |
| **本流程** | **端到端互動整合 + elicitation 易用性 + 規模/長程效應** |

一句話：**芯片巫師式管線貫穿測試 + 動態神諭作者 + 多 session 續跑**。相對 `芯片巫師/` 的唯一但關鍵升級——作者從靜態 fixture 變成「被問時能即時翻查原著」的**動態神諭**。

## 2. 兩個地基決定（不可動搖）

1. **量的是「系統輔助力」，不是「重建保真度」。** 原著 canon 只當「真實作者素材來源 + 作者臨場改編的一致性參照」，**不是拿來比對的正解**。系統是創作輔助、作者有自由，產物本就不必貼合原著。獎勵「貼合 canon」＝獎勵背誦，正是要避開的失敗。
2. **編排＝檔案共享 + 冷啟單步 + 作者當神諭。** 連續性靠**磁碟上的檔案**，不靠 agent 記憶。

## 3. 三個角色與硬防火牆

| 角色 | 誰來扮 | 能讀 | **絕不能讀** |
|------|--------|------|--------------|
| **導演（中介）** | 主 agent（跑本 session 的你） | 全部 | —（但對系統↔作者間**只逐字轉述**，不改寫、不順手補） |
| **系統 agent（受測體）** | 每個管線步驟開一個**冷** subagent | `<書>/` 書資料夾＋系統套件（`結構定義/`／`技巧知識庫/`／`.claude/skills/`） | `examples/`、`情境測試/`（**硬防火牆**） |
| **作者 agent（神諭）** | 每次被問時開一個冷 subagent | `<書>/raw/`＋`情境測試/<書>/`＋（fallback）`examples/<書>/` | — |

**硬防火牆是測試有效性的地基**：系統 agent 完全不讀原著，否則無法區分「系統輔助產出」與「AI 背誦原著」。

**操作要點**：
- 系統 agent 用 `Agent` 冷啟，prompt 明列「**你不得讀取 `examples/` 或 `情境測試/`；你的知識只有書資料夾與系統套件；不確定的設定要回問作者，不要自行假設**」。用「作者口吻」下該階段任務。
- 作者 agent 用 `Agent` 冷啟，prompt 給它「你是《X》的作者，你的靈感寫在 `<書>/raw/`、你腦中未寫下的想法在 `情境測試/<書>/`；先從這兩處回答；**若都沒有**才去 `examples/<書>/` 找，並照你自己想出來的口吻回答」＋**要求它回報：本次答案是來自腦（raw/情境測試）還是 fallback（examples）**。
- 導演轉述時**逐字**，不加料。導演自己認得原著，注入是最容易破壞測試的動作。

## 4. 資料佈局與知識邊界

```
<書>/                        ← 書資料夾（版控、累積共享狀態）
  raw/                       ← 系統『讀得到』的素材（凌亂、不完整）
  story/ chapters/           ← 系統產出（下游逐步累積）
情境測試/<書>/                ← 作者『腦裡』（系統看不到，只能靠問對問題撈出）
                             ＋測試 meta：session台帳.md、fallback-log.md、每 session 記錄 SNN-*.md
examples/<書>/…              ← 原著：作者臨場改編的神諭＋人工通讀的合理性參照（系統絕不可讀）
```

**raw vs 情境測試 的切分原則**（P session 產生時遵守）：
- `raw/` = 真作者會**寫下**的凌亂靈感：幾個場景、基調、主角種子、也許結局念頭——**片段、無序、不完整**，逼系統去結構化／追問／擴充而非抄寫。
- `情境測試/<書>/` = 作者**腦裡有、但不主動寫**的連接組織：次要角色細節、世界規則、特定 reveal——只在系統**問到**時才吐。
- **刻意留缺口**：答案不能都塞進 `raw/`，否則 elicitation 測不到。理想狀態是「raw 足以起步，但系統必須問出一堆東西才能往下走」。

## 5. 續跑協定 ⭐（新 session 從這裡開始）

**開場三讀**（照順序）：
1. 本 doc（`情境測試/端到端貫穿測試流程.md`）。
2. `情境測試/<書>/session台帳.md` —— 上個 session 做到哪、下一步該做什麼、有沒有待決 elicitation。
3. `<書>/story/參照/就緒儀表.md`（若已存在）—— 系統自維的 in-world 進度。

**做事**：照 §6 的 session 序列，跑本 session 該跑的階段。過程照 §7 執行迴圈。

**收場二寫**：
1. 更新 `情境測試/<書>/session台帳.md`（見下方格式）。
2. commit 書資料夾（版控＝天然快照，可診斷可回滾）。

**鐵則**：
- **絕不切斷一次 elicitation 迴圈**——寧可少做一階段，不要問到一半就收 session、留懸空線。
- **兩本台帳分離**：`就緒儀表` 是系統維護的 in-world 產物（在書資料夾內、系統看得到）；**session台帳／fallback-log／每 session 記錄是測試 meta，只放 `情境測試/<書>/`，永不進書資料夾**（進去＝把測試提示餵給系統 agent＝污染，HP 教訓）。
- **狀態永不 reset、累積前進**——這正是規模測試本體。

### session台帳.md 格式

```markdown
# <書> · 端到端貫穿測試 · session 台帳

## 進度總覽
| Session | 階段 | 範圍 | 狀態 | commit |
|---------|------|------|------|--------|
| P  | 前置：產 raw＋情境測試＋init 書架 | — | ✅ | <hash> |
| S1 | develop | 全書 | ✅ | <hash> |
| S2 | organize→outline-test | 全書大綱 | 🚧 進行中 | — |
| …  |

## 下一步（新 session 讀這段）
- 要做：<具體階段＋範圍>
- 前置狀態：<上游已就緒到哪、就緒儀表指到哪>
- 待決 elicitation：<有無上個 session 沒問完的線；理想是「無」>

## 累積留意（長程效應追蹤）
- 跨 arc 伏筆：<埋在哪、該在哪收、收了沒>
- 設定自洽風險：<累積到現在有無新舊矛盾>
- 角色狀態漂移：<主要角色狀態演變有無斷裂>
```

### 每 session 記錄 `SNN-<階段>.md` 格式（可追溯留痕）

```markdown
# SNN <階段> — <範圍>

## 系統路由觀察
作者這樣說 → 系統走了哪個 skill／檔位（對不對）

## Elicitation 紀錄
系統問了什麼 → 作者答什麼 → 來源（腦／fallback:examples）
（每筆 fallback 另記進 fallback-log.md）

## 產物
產了/改了哪些檔；inline 審稿 skill（outline-test/beat-test/write-test）結果摘要

## 摩擦與缺口
真作者會想要、系統接不住的地方（未來新功能候選）

## purity 檢查
系統有無吐出「沒被告知的原著專屬細節」（命中＝污染警訊，記下）
```

## 6. Session 切分（管線序列）

一個 session = 在有界範圍上跑**完整的幾個管線階段**。大小以「塞得下雙 agent 轉述開銷後仍有餘裕」為準（比單 agent 跑得少）。

| Session 型 | 做什麼 | 產出 |
|-----------|--------|------|
| **P（每本一次）** | 讀原著 → 按 §4 切分原則產 `raw/`＋`情境測試/<書>/`；複製 `書本模板/` init 書資料夾；建 session台帳 | 素材＋神諭腦＋空書架 |
| **S1** | develop（raw→摘要＋設定冷啟動） | `story/00-摘要`、`story/設定/` |
| **S2** | organize→outline-test（→大綱） | `story/01-大綱` |
| **S3…** | 逐 arc：beat-sheet→beat-test | `story/幕綱/arcNN` |
| **…** | 逐 arc：write→write-test→revise | `chapters/chNNNN` |

範圍是流程的**參數**：首跑目標＝一世之尊卷一全部內文，明確接受需跨多 session 逐步建成。一次一個乾淨的階段邊界收 session。

## 7. 執行迴圈（每個管線步驟）

1. 導演選下一階段＋範圍。
2. 冷啟**系統 agent**（帶防火牆），用作者口吻下任務。
3. 系統做事；需要作者輸入時**回傳問題**（不硬闖假設）。
4. 導演**逐字**轉問 → 冷啟**作者 agent** → 得答案（＋來源標記）→ **逐字**轉回。fallback 記進 `fallback-log.md`。
5. 系統完成、寫入書資料夾；系統照常 inline 跑自己的審稿 skill。
6. 導演 commit 快照、寫 `SNN-*.md`、更新台帳。→ 迴圈。

## 8. 評估模型

**自動化只負責『忠實產出 + 留痕』，不自評打分。真正的評估＝作者本人在內文全部產出後的人工通讀，憑讀感找問題。** 卷一之長是刻意的**規模/長程效應**壓力測試。

自動化迴圈只做三件事：
1. **Inline 跑系統自己的審稿 skill**（outline-test／beat-test／write-test）——真作者用這系統本來就會跑，是管線一部分，非另加評分層。
2. **honesty guards**：硬防火牆、fallback log、purity 檢查（§5 記錄格式）。
3. **可追溯留痕**：`SNN-*.md`＋每階段 git 快照。用途——作者日後讀到某章讀感不對時能**回溯**：壞正文 → 哪個 write session → 哪個幕綱 → 哪個設定決定 → 哪次 elicitation。

**規模/長程效應（本流程特有觀察對象）**：狀態累積不 reset，盯只在大量內容下才破的東西——跨 arc 伏筆償還、多章後設定自洽、角色狀態漂移、基調一致性。台帳「累積留意」段逐 session 更新。

**purity side-lane**：系統 agent 即便不讀 examples/ 仍可能**認出**一世之尊而憑記憶重建。持續盯「吐出沒被告知的原著專屬細節」；命中＝有效性警訊、記一筆，非扣分。

**post-hoc 診斷 lens（人工通讀時用，非自動門檻）**：需要時沿五維檢視某段——① 路由　② elicitation 品質（fallback log 當量尺）　③ 上下游承接　④ 產物品質　⑤ 摩擦與缺口。

## 9. 陷阱與有效性威脅

- **污染**（HP 教訓）：端到端本就累積上游屬正常；但**測試 meta 別漏進書資料夾**；日後做 ablation 對照組須先剝乾淨、連 git 工作區狀態一起剝。
- **導演注入**：導演也認得原著，故只逐字轉述、不補話。
- **作者過度翻原著→重建偏誤**：最小發明、按問題具體度回答、腦裡真沒有才撈、每次撈都記 log。
- **辨識性洩漏**：選一世之尊（比 HP 難背）已降風險，purity 續盯。
- **鷹架有沒有用（循環性）**：綠燈只證明「管線不礙事且產出連貫」；要把品質歸功於 skill／schema，需偶爾 ablation（剝掉某 skill 指引冷跑、看品質是否掉）——**選配深挖**，借 HP RED 紀律，非每跑必做。

## 10. 換書 onboarding（跑一次 P session）

流程零硬編一世之尊。換書：
1. 選一本現成好書（放 `examples/<書>/`）——挑「寫得好、但 AI 較難整段背誦」的，降辨識性洩漏。
2. 冷啟一個讀過該書的 subagent，按 §4 切分原則產 `<書>/raw/`＋`情境測試/<書>/`（刻意留 elicitation 缺口）。
3. 複製 `書本模板/` 成 `<書>/`，init 書架。
4. 建 `情境測試/<書>/session台帳.md`（用 §5 格式，P 列標 ✅）。
5. 之後照 §5 續跑協定、§6 序列逐 session 跑。
