# novelAI — AI 輔助小說寫作系統套件

這個 repo **本身就是「寫作系統套件」**，不是某一本書。它承載「每次寫作都需要、且跨所有書不變」的方法與工具；小說內容（每本都不同）則放在各自的書資料夾裡，兩者嚴格分離。

> 設計依據：`docs/superpowers/specs/2026-07-10-novel-structure-pipeline-design.md`（子專案 A，唯一真實來源）、`docs/superpowers/specs/2026-07-11-subproject-b-write-skill-design.md`（子專案 B）。
> 子專案 A：把 `raw/` 整理成結構化 `story/`，並對「幕綱」跑結構測試，範圍終點＝**幕綱通過測試**。
> 子專案 B：依幕綱動筆寫章節——`write`（幕綱展開成散文草稿）、`write-test`（正文審稿）、`revise`（依報告定點修訂）已齊。

---

## 三個組成部分

| 部分 | 資料夾 | 角色 |
|------|--------|------|
| **結構定義** | `結構定義/` | 「符合結構的產出」到底長怎樣（schema）：摘要／世界觀／大綱／幕綱／章節／角色需求／結構／就緒儀表／下游回饋記錄 |
| **技巧知識庫** | `技巧知識庫/` | 從 `Data/`（洛克老師教學）提煉的「方法」：創作流程／因果法則／結構公式／需求四象限／伏筆與意外／主題題旨／世界觀與背景／氛圍與筆調／選題與收斂；`外部延伸/` 補 Data 所缺的方法論（POV／Scene-Sequel／Show-Don't-Tell／動作場面等） |
| **Skills** | `.claude/skills/` | 「操作」：`develop`（粗階段冷啟動）、`organize`（整理大綱，支援全書或局部 arc 兩種模式）、`outline-test`（大綱測試，可指定 arc）、`beat-sheet`（幕綱共創）、`beat-test`（幕綱測試，可指定 arc）、`write`（依幕綱撰寫正文）、`write-test`（正文審稿）、`revise`（依報告定點修訂正文）、`sync`（上下游同步診斷：上游改動找過期下游、回頭收斂前盤點已承諾事實） |

分工：**技巧知識庫承載「方法」，skill 承載「操作」**——skill 執行時載入知識庫與結構定義來工作。

### 為什麼 skill 放在「專案層」`.claude/skills/`

只有在 `novelAI/` 這個目錄工作時，這三個寫作 skill 才會被 Claude Code 載入，因此**天生與你平常寫程式用的其他 skill／plugin 隔離**，不會互相污染。日後若要跨目錄使用，可再升級成獨立 plugin。

## 開新書怎麼做

1. 複製 `書本模板/` 到 repo 根目錄，改名為你的書名（例：`奇幻冒險/`）。
   > 書資料夾與系統套件一起在本 repo 版控（見下方第 7 點）。
2. 把你手上的瑣碎筆記、靈感、半成品（任意格式）丟進 `你的書/raw/`。
3. 若 raw 還很亂、主線/題旨/結局未定，先對 Claude 說「幫我把 raw/ 想清楚」（觸發 `develop`）收斂大方向；若大方向已大致就緒，直接說：**「幫我把 raw/ 整理成符合結構的大綱和幕綱」**（觸發 `organize`，產出全書 `story/01-大綱.md`）。若只有**某一段**已想清楚、全書結局還沒定，可以說「先幫我把『這一段』整理成大綱」——`organize` 會改產出局部的 `story/大綱/arcNN.md`，不必替尚未收斂的全書結局搶先拍板；待全書收斂後再回來走一次全書模式。
4. 大綱/摘要大致定稿後，可用 `outline-test` 審大綱；再反覆用 `beat-sheet` 共創幕綱、用 `beat-test` 審稿，直到幕綱通過測試。
5. 幕綱就緒後，說「幫我把這幾幕/這一章寫成正文」（觸發 `write`）——AI 會先提議分章方案供你拍板，再依幕綱展開成 `chapters/` 下的散文草稿。寫完用 `write-test` 審稿，要把可疑點修掉用 `revise`。
6. **迭代**：回頭改了上游（結局拍板、題旨修改、世界觀規則變更、幕綱在成章後又改）之後，說「哪些下游過期了」（觸發 `sync`）列出待重驗清單；要回頭收斂粗層（結局、配角、中後段）之前，也可先用 `sync` 反向盤點已下沉段落「已承諾的事實」。
7. **書的版本歷史**：書資料夾與系統套件**一起在本 repo 版控**（刻意設計——迭代型創作經常回頭改，小說內容需要版本歷史）。開新書後記得把書資料夾 commit 進來即可，不需另外設定。

每本書資料夾結構：

```
你的書/
├── raw/        # 你丟進來的任意格式素材
├── story/      # 整理後符合結構的產出（摘要／大綱／幕綱／參照）
└── chapters/   # 依幕綱寫的章節草稿（子專案 B，由 write 產生）
```

## 核心約束（設計原則）

- **嚴格分離**：系統/技巧（不變、跨書）與小說內容（每本不同）分開。skill 不依賴任何一本書的絕對路徑。
- **幕綱是結構化 Markdown**：每幕一 section＋固定欄位；用 `[[幕NNN]]` 指前因、`埋[[伏筆:x]]`／`收[[伏筆:x]]` 標伏筆。**伏筆不另立登記簿，幕綱裡的標記就是唯一真實來源。**
- **局部下沉是一等公民，不是變通**：大方向不平均沒關係——已清楚的一小段（arc）可以先在大綱層（`story/大綱/arcNN.md`）與/或幕綱層（`story/幕綱/arcNN.md`）下沉做細，其餘留粗；下沉品一律標「暫定，待粗層鎖定」。
- **beat-test 是「審稿員」不是門檻**：輸出可疑點報告交作者判斷，不做自動 pass/fail。
- **AI＝共創＋審稿**，所有創作決定權在作者。
